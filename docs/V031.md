# V031 — edinet / jpxbrl パッケージ分割計画

## 目的

`edinet` ライブラリの **汎用 XBRL パーサー部分** を `jpxbrl` パッケージとして分離し、**EDINET 固有ロジック** のみを `edinet` パッケージに残す。

- TDnet ライブラリが `jpxbrl` を共有基盤として利用できるようにする
- EDINET / TDnet 両方で使える汎用 XBRL 機能を1箇所で管理する
- `edinet` パッケージは `jpxbrl` に依存する形になる
- 既存ユーザーの `from edinet.xbrl import ...` は引き続き動作する（後方互換）

---

## 前提: V030 完了後に実施

V030（ConceptResolver DI）を先に完了することで:

- 名寄せロジックが Protocol で抽象化済み → 分割時に依存の方向が明確
- `extract_values()` の名寄せ部分が `edinet` パッケージ内に閉じている
- `jpxbrl` は名寄せロジックを一切知らない純粋な XBRL パーサーになる

## モジュール分類

### jpxbrl に移動するモジュール（汎用 XBRL）

| モジュール | 理由 |
|---|---|
| `xbrl/parser.py` | `RawFact`, `RawContext`, `RawUnit`, `ParsedXBRL` — 純粋な XML パーサー。EDINET 固有ロジックなし |
| `xbrl/ixbrl_parser.py` | iXBRL パーサー + `merge_ixbrl_results()` — HTML → XBRL 変換。EDINET 固有ロジックなし |
| `xbrl/contexts.py` | `StructuredContext`, `DurationPeriod`, `InstantPeriod` — 期間・次元の構造化。**要パラメータ化**（後述） |
| `xbrl/units.py` | `SimpleMeasure`, `DivideMeasure`, `StructuredUnit` — 単位構造化。EDINET 固有ロジックなし |
| `xbrl/_linkbase_utils.py` | リンクベース共通定数（`ROLE_LABEL` 等）。XBRL 標準仕様の定数のみ |
| `xbrl/linkbase/calculation.py` | Calculation Linkbase パーサー。汎用 |
| `xbrl/linkbase/definition.py` | Definition Linkbase パーサー。汎用 |
| `xbrl/linkbase/presentation.py` | Presentation Linkbase パーサー。汎用 |
| `xbrl/text/__init__.py` | TextBlock 抽出。汎用 HTML 処理 |
| `xbrl/validation/calc_check.py` | 計算バリデーション。XBRL 標準仕様に基づく |

### edinet に残すモジュール（EDINET 固有）

| モジュール | 理由 |
|---|---|
| `xbrl/_namespaces.py` | `EDINET_BASE = "http://disclosure.edinet-fsa.go.jp/"` — EDINET URI パターンへの完全依存 |
| `xbrl/dei.py` | `jpdei_cor` 名前空間の DEI 要素抽出。EDINET 固有の `AccountingStandard` enum |
| `xbrl/taxonomy/__init__.py` | `TaxonomyResolver` — EDINET 標準タクソノミのラベル解決。ファイルパス・キャッシュ戦略が EDINET 固有 |
| `xbrl/taxonomy/concept_sets.py` | `ConceptSet` — EDINET Presentation Linkbase に基づく科目セット導出 |
| `xbrl/taxonomy/custom.py` | 拡張科目検出 — `statements.py` / `dei.py` に依存 |
| `api/` | EDINET API クライアント |
| `models/` | Filing, DocType 等の EDINET データモデル |
| `financial/` | 名寄せ、財務諸表組み立て、ConceptResolver 等 |
| `display/` | 表示レイヤー |
| `dataframe/` | DataFrame 変換 |

### 境界ケース（要設計判断）

#### 1. `xbrl/facts.py` — RawFact → LineItem 変換

**現状**: `RawFact` + `StructuredContext` + `TaxonomyResolver` → `LineItem` を生成。

**問題**: `LineItem` は `edinet.models.financial` に定義された EDINET 固有モデル。`TaxonomyResolver` も EDINET 固有。

**対処案**:
- `jpxbrl` には `RawFact` → 基本的な構造化 Fact の変換関数のみを置く
- `edinet` の `facts.py` は `jpxbrl` の構造化 Fact + `TaxonomyResolver` → `LineItem` の変換を担う
- あるいは、`facts.py` は丸ごと `edinet` に残し、`jpxbrl` は `RawFact` までの責務とする

**推奨**: `facts.py` は `edinet` に残す。`jpxbrl` の責務は `ParsedXBRL`（RawFact 群）の生成まで。ラベル解決や LineItem への変換はサービス固有。

#### 2. `xbrl/data/` — 埋め込みデータ

**現状**: concept_sets の JSON データは EDINET Presentation Linkbase から生成。

**対処案**: `data/` ディレクトリは `edinet` に残す。`jpxbrl` にはデータファイルを持たない。

## 主要な技術的課題と対処

### 課題 1: `contexts.py` の連結軸名ハードコード

**現状**: `contexts.py:36` に以下がハードコードされている:

```python
_CONSOLIDATED_AXIS_LOCAL = "ConsolidatedOrNonConsolidatedAxis"
_NON_CONSOLIDATED_MEMBER_LOCAL = "NonConsolidatedMember"
```

- EDINET: `ConsolidatedOrNonConsolidatedAxis`
- TDnet: `ConsolidatedNonconsolidatedAxis`（"Or" がない）

**同時に**: `statements.py:59-61` にも同じ定数がある:

```python
_CONSOLIDATED_AXIS_SUFFIX = "ConsolidatedOrNonConsolidatedAxis"
_CONSOLIDATED_MEMBER_SUFFIX = "ConsolidatedMember"
_NONCONSOLIDATED_MEMBER_SUFFIX = "NonConsolidatedMember"
```

`statements.py` は `endswith()` で判定しているため、`ConsolidatedNonconsolidatedAxis` も `ConsolidatedOrNonConsolidatedAxis` もマッチする。

**対処案**:
- `jpxbrl/contexts.py` では連結軸の判定を **行わない**（純粋な構造化のみ）
- 連結/個別の判定は `edinet` 側（`statements.py`）の責務として残す
- `jpxbrl` の `StructuredContext` は `dimensions` を構造化するが、どの dimension が連結軸かは知らない
- TDnet ライブラリも同様に自前で連結判定を行う

### 課題 2: `_namespaces.py` のサービス固有 URI

**現状**: `_namespaces.py` に EDINET 固有の定数と正規表現:

```python
EDINET_BASE = "http://disclosure.edinet-fsa.go.jp/"
EDINET_TAXONOMY_BASE = "http://disclosure.edinet-fsa.go.jp/taxonomy/"
_STANDARD_TAXONOMY_PATTERN = re.compile(r"^http://disclosure\.edinet-fsa\.go\.jp/taxonomy/...")
_FILER_TAXONOMY_PATTERN = re.compile(r"^http://disclosure\.edinet-fsa\.go\.jp/...")
```

**対処案**:
- `jpxbrl/_namespaces.py` には **XBRL 標準名前空間のみ** を置く:
  - `NS_XBRLI`, `NS_XBRLDI`, `NS_LINK`, `NS_XLINK`, `NS_XSI`, `NS_XML`
  - `NS_ISO4217`, `NS_XBRLDT`, `NS_NUM`, `NS_NONNUM`, `NS_UTR`
- `classify_namespace()` と EDINET 固有パターンは `edinet` に残す
- TDnet は `tdnet/_namespaces.py` に `TDNET_BASE = "http://www.xbrl.tdnet.info/taxonomy/"` を持つ

### 課題 3: `facts.py` の `LineItem` 依存

**現状**: `build_line_items()` は以下に依存:

```python
from edinet.models.financial import LineItem
from edinet.xbrl.contexts import StructuredContext
from edinet.xbrl.taxonomy import TaxonomyResolver
```

**対処案（推奨）**: `facts.py` は `edinet` に残す。

- `jpxbrl` の責務: bytes → `ParsedXBRL`（`RawFact` 群）
- `edinet` の責務: `RawFact` + `StructuredContext` + `TaxonomyResolver` → `LineItem`
- TDnet ライブラリも独自の `build_line_items()` を持てる

### 課題 4: 後方互換（import パス）

**影響を受ける import パターン**:

```python
# 全て引き続き動作させる
from edinet.xbrl import parse_xbrl_facts
from edinet.xbrl.contexts import DurationPeriod, InstantPeriod
from edinet.xbrl.units import SimpleMeasure
from edinet.xbrl.linkbase.calculation import parse_calculation_linkbase
```

**対処**: `edinet.xbrl.__init__.py` で `jpxbrl` のシンボルを re-export。

`__all__` を維持し、`import *` も引き続き動作する。

### 課題 5: テストの移動

**方針**:
- `jpxbrl` に移動するモジュールのテストは `packages/jpxbrl/tests/` に移動
- `edinet` 固有のテストは `packages/edinet/tests/` に残す
- 統合テスト（`tools/` 配下の E2E）は `edinet` 側で実行

## EDINET / TDnet / jpxbrl の依存関係

```
         ┌──────────┐
         │  jpxbrl   │  汎用 XBRL パーサー (lxml のみ)
         └────┬──────┘
              │
     ┌────────┴────────┐
     │                 │

│           │    │            │
│ API/名寄せ│    │ API/DEI等  │
└───────────┘    └────────────┘
```

- `jpxbrl`: 依存ゼロに近い（`lxml` のみ）。XBRL/iXBRL のパース、Context/Unit 構造化、リンクベース解析
- `edinet`: `jpxbrl` + `httpx` + `pydantic` + `platformdirs`。EDINET API、名寄せ、財務諸表組み立て
- `tdnet` (将来): `jpxbrl` + `httpx`。TDnet API、TDnet 固有 DEI/メタデータ

## リスクと軽減策

| リスク | 軽減策 |
|---|---|
| 後方互換の破壊 | `edinet.xbrl.__init__.py` での re-export で全既存 import パスを維持 |
| 循環依存 | `jpxbrl` は `edinet` を一切 import しない。依存は一方向のみ |
| PyPI 名前の衝突 | `jpxbrl` は事前に PyPI で名前を確保 |
| モノレポの CI 複雑化 | uv workspace で `uv run pytest` が全パッケージを一括テスト可能 |
| パフォーマンス | re-export のオーバーヘッドは import 時のみ。実行時コストなし |
| 段階的リリース | Phase 1-4 を `edinet` 内で完了してから PyPI リリース。中間段階では `jpxbrl` は editable install |
