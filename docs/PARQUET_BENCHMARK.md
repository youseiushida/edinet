### サイズ推定

| 期間 | 書類数 | Parquet サイズ（推定） |

> 推定は実測値 108 KB/件ベース。圧縮設定（zstd 等）で 30-50% 削減可能。

### 処理時間推定

| 期間 | XBRL 件数 | パース時間 | 書き出し時間 | 合計 |
|:---|---:|---:|---:|---:|
| 1年 | 64,853 | ~6.4h | ~55min | ~7.5h |
| 5年 | ~320,000 | ~32h | ~4.5h | ~37h |

> パース時間がボトルネック（ネットワーク I/O + CPU）。書き出し・インポートは比較的高速。

### メモリ要件

現在の `export_parquet()` は全行をメモリに蓄積してから一括書き出しを行う。

| 処理単位 | ピークメモリ（推定） | 16GB PC |
|:---|---:|:---|
| 3日間（ストレステスト） | ~3 GB | OK |
| 1ヶ月分（~5,400件） | ~4 GB | OK |
| 1年分（76,745件） | ~35 GB | NG |

> Python dict のメモリオーバーヘッドは Parquet ファイルサイズの 3-5 倍。
> 16 GB 環境では **月次など適切なチャンクに分割して処理** する必要がある。

チャンク処理の例:

```python
from edinet.extension import export_parquet

# 月ごとに分割してエクスポート
for month_start in month_starts:
    filings = edinet.documents(start=month_start, end=month_end)
    data = [(f, f.xbrl()) for f in filings if f.has_xbrl]
    export_parquet(data, "./out", prefix=f"{month_start:%Y-%m}_")
```

## 発見されたバグと修正

本ベンチマークの過程で以下のバグを発見・修正した。

### 1. xbrl()/axbrl() パース失敗時の ZIP メモリリーク

`_build_statements()` が例外を投げた場合、`_zip_cache`（数 MB/件）が解放されなかった。
大量の Filing をループで処理すると、失敗分の ZIP がメモリに蓄積し OOM を引き起こす。

**修正**: `try/except` で失敗時も `_zip_cache` と `_xbrl_cache` を解放。

### 2. install_taxonomy() の Shift-JIS バックスラッシュ問題

タクソノミ ZIP 内のフォルダ名「タクソノミ」が Shift-JIS でエンコードされており、
「ソ」のバイト `0x5C` が Windows 上でパス区切り `\` として解釈される。
結果として展開後のディレクトリ構造がプラットフォームによって異なり、
`taxonomy/` ディレクトリの検出に失敗していた。

**修正**: `_detect_zip_prefix()` を任意の深さに対応させ、`taxonomy` / `samples` を
探索する際にネスト階層を固定しないよう変更。

### 3. v0.1.0 時代の IFRS 未対応警告の残存

IFRS 対応済みにもかかわらず「IFRS / US-GAAP の Filing は v0.1.0 では未対応です」
という警告が残っていた。

**修正**: 該当コードブロックを削除。


## フィードバック
PyArrowの ParquetWriter を使ったストリーミング書き出し
メモリに全件ため込まず、1000件パースするごとに既存のParquetファイルに「追記（Append）」していく処理にすれば、16GBのPCでも1年分（あるいは5年分）のデータをメモリ溢れなしで1回のスクリプト実行で作れるようになります。

圧縮アルゴリズムの指定（zstd）
PandasやPyArrowのエクスポート時に compression='zstd' を明示するだけで、デフォルトのSnappyよりも圧倒的な圧縮率（かつ展開速度も速い）を実現できます。
