# Q-2.4.a: EDINET API 提供期間

## 質問

EDINET API の `documents()` で取得可能な最古日はいつか（5年? 10年?）。`file_type="1"` でダウンロード可能な最古の書類はいつか。これにより対応すべきタクソノミバージョン範囲が決まる。

## 品質

確度: **A（実証済み）** — スクリプトで API を実際に呼び出し、最古日とダウンロード可否を確認。

## 回答

### [F1] API 最古取得可能日: 2016年3月

- `documents()` API は **2016年3月1日以降** のデータを返す。
- 2016年1月・2月は HTTP 404 が返り、データが存在しない。
- 最古の Filing: **2016-03-03**（株式会社ダイサン、四半期報告書、doc_id=S100733W）
- **出典**: `Q-2.4.api_oldest.py` 実行結果

### [F2] XBRL ダウンロード可否: 成功

- 最古 Filing（S100733W, 2016-03-03）の `file_type="1"` ダウンロードは **成功**。
- ZIP サイズ: 77,332 bytes
- **出典**: `Q-2.4.api_oldest.py` 実行結果

### [F3] 閲覧期間の仕様: 10年

- 仕様書の閲覧期間テーブル（`EDINETAPI仕様書.md` L.175-179）:
  - 有価証券報告書: 縦覧5年 + 延長5年 = 閲覧 **10年**
  - 半期報告書: 縦覧5年 + 延長5年 = 閲覧 **10年**
  - 四半期報告書: 縦覧3年 + 延長7年 = 閲覧 **10年**
- v1.1（2023年5月）で5年→10年に延長（`EDINETAPI仕様書.md` L.2398）
- 10年を経過したファイル日付のファイルは日次更新処理で削除される（`EDINETAPI仕様書.md` L.1361）

### [F4] 2016-03 Filing のフォーマット上の注意

- 2016年3月のデータは存在するが、多数の「不正行」警告が出る（1日あたり100-500件）
- これは古いフォーマットの Filing メタデータが現行パーサーと互換性がないことを示す
- Filing オブジェクト自体は正常に構築され、ダウンロードも成功する

## 推論

### [R1] 対応すべきタクソノミバージョン数

- [F1] 最古日 2016-03 + [F3] 10年閲覧期間 → **約10バージョンのタクソノミに対応が必要**
- EDINET タクソノミは年次更新（`YYYY-11-01` 形式）のため:
  - 2015-11-01 版（2016年3月提出の書類が参照する可能性あり）
  - 2016-11-01 版
  - ...
  - 2025-11-01 版（最新）
- 対応すべき範囲: **2015-11-01 〜 2025-11-01 の最大11バージョン**

### [R2] 最古日は今後ローリングで変動

- [F3] 10年経過で日次削除 → 2026年3月以降は 2016年3月のデータが順次消失
- パーサーとしては「最新10年分」の対応で十分であり、古いバージョンのサポートは自然に不要になる
