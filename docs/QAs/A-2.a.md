## A-2. Context 要素の構造

### 質問への対応

| サブ質問 | 回答 |
|----------|------|
| A-2.1 id 属性の命名規則 | EDINET 仕様書で厳密に規定されている。`{相対期間又は時点}{期間又は時点}(_{メンバー要素名})*(_連番3桁)?` のパターン。企業が自由に命名することはできない |
| A-2.2 entity/identifier | scheme は `http://disclosure.edinet-fsa.go.jp` の1種類のみ。identifier の値は `{EDINETコード}-{追番3桁}` (例: `X99001-000`)。他の scheme はない |
| A-2.3 segment vs scenario | EDINET では **scenario** を使用する。segment は使用しない（仕様書 図表 5-4-6 No.3「セグメント: 指定しない」） |
| A-2.4 segment の内部構造 | segment は使用されないため該当なし。dimension 情報は **scenario** 内に `xbrldi:explicitMember` で設定 |
| A-2.5 Typed vs Explicit Dimension | EDINET では **Explicit Dimension のみ**使用。Typed Dimension は仕様書に記載なく、サンプルにも出現しない |
| A-2.6 デフォルト dimension | 連結/個別軸 (`ConsolidatedOrNonConsolidatedAxis`) のデフォルトは **連結** (`ConsolidatedMember`)。連結の場合は scenario にメンバーを設定しない。個別の場合のみ `NonConsolidatedMember` を設定 |
| A-2.7 Context の意味的等価性 | 仕様上、同一内容の Context が重複定義されるケースは想定されていない。Context ID が命名規則で一意に決まるため |
| A-2.8 Context の総数の目安 | P-2 のトヨタ実データで 278 個。サンプル J-GAAP でも多数（セグメント軸×期間の組合せ）。辞書 (`dict[str, Context]`) で十分に管理可能 |
| A-2.9 period の形式 | instant: `YYYY-MM-DD` 固定。duration: `startDate` + `endDate` いずれも `YYYY-MM-DD`。タイムゾーンなし。`forever` は不使用。四半期は `Interim` プレフィックスで累計期間 |
| A-2.10 segment 内の dimension 要素 | scenario 内に `xbrldi:explicitMember` (`http://xbrl.org/2006/xbrldi`)。`dimension` 属性は QName 形式。xbrldi スキーマは `ALL_20251101` 同梱 |
| A-2.11 エンティティ識別子の変動 | EDINET コードは法人に紐付くため、合併・社名変更でもコード自体は変わりうる。ただし「同一法人 = 同一 EDINET コード」が基本原則 |

### Context ID 命名規則の詳細

**基本パターン** (図表 5-4-2):

```
{相対期間又は時点}{期間又は時点}(_{メンバー要素名})*(_連番3桁)?
```

**{相対期間又は時点}** の取りうる値 (図表 5-4-4):

| 値 | 意味 |
|----|------|
| `CurrentYear` | 当年度 |
| `Interim` | 中間期 |
| `Prior1Year` | 前年度 |
| `Prior1Interim` | 前中間期 |
| `Prior2Year` | 前々年度 |
| `Prior{n}Year` | n年度前 |
| `Prior{n}Interim` | n年度前中間期 |
| `FilingDate` | 提出日 |
| `RecordDate` | 議決権行使の基準日 |
| `RecentDate` | 最近日 |
| `FutureDate` | 予定日 |

**{期間又は時点}**:

| 値 | 意味 |
|----|------|
| `Instant` | 時点（BS 等） |
| `Duration` | 期間（PL 等） |

**連結/個別の表現** (図表 5-4-3):
- 連結: scenario なし（デフォルト）→ Context ID に `_NonConsolidatedMember` なし
- 個別: scenario に `NonConsolidatedMember` を設定 → Context ID に `_NonConsolidatedMember` 付与

### Context 設定例（サンプル .xbrl からの実例）

**当期連結時点** (`CurrentYearInstant`):
```xml
<xbrli:context id="CurrentYearInstant">
  <xbrli:entity>
    <xbrli:identifier scheme="http://disclosure.edinet-fsa.go.jp">X99001-000</xbrli:identifier>
  </xbrli:entity>
  <xbrli:period>
    <xbrli:instant>2026-03-31</xbrli:instant>
  </xbrli:period>
</xbrli:context>
```

**当期個別期間** (`CurrentYearDuration_NonConsolidatedMember`):
```xml
<xbrli:context id="CurrentYearDuration_NonConsolidatedMember">
  <xbrli:entity>
    <xbrli:identifier scheme="http://disclosure.edinet-fsa.go.jp">X99001-000</xbrli:identifier>
  </xbrli:entity>
  <xbrli:period>
    <xbrli:startDate>2025-04-01</xbrli:startDate>
    <xbrli:endDate>2026-03-31</xbrli:endDate>
  </xbrli:period>
  <xbrli:scenario>
    <xbrldi:explicitMember dimension="jppfs_cor:ConsolidatedOrNonConsolidatedAxis">jppfs_cor:NonConsolidatedMember</xbrldi:explicitMember>
  </xbrli:scenario>
</xbrli:context>
```

**セグメント軸付き** (`CurrentYearDuration_jpcrp030000-asr_X99001-000CommunicationsEquipmentReportableSegmentMember`):
```xml
<xbrli:context id="CurrentYearDuration_jpcrp030000-asr_X99001-000CommunicationsEquipmentReportableSegmentMember">
  <xbrli:entity>
    <xbrli:identifier scheme="http://disclosure.edinet-fsa.go.jp">X99001-000</xbrli:identifier>
  </xbrli:entity>
  <xbrli:period>
    <xbrli:startDate>2025-04-01</xbrli:startDate>
    <xbrli:endDate>2026-03-31</xbrli:endDate>
  </xbrli:period>
  <xbrli:scenario>
    <xbrldi:explicitMember dimension="jpcrp_cor:OperatingSegmentsAxis">jpcrp030000-asr_X99001-000:CommunicationsEquipmentReportableSegmentMember</xbrldi:explicitMember>
  </xbrli:scenario>
</xbrli:context>
```

**大株主軸付き** (`CurrentYearInstant_No1MajorShareholdersMember`):
```xml
<xbrli:context id="CurrentYearInstant_No1MajorShareholdersMember">
  <xbrli:entity>
    <xbrli:identifier scheme="http://disclosure.edinet-fsa.go.jp">X99001-000</xbrli:identifier>
  </xbrli:entity>
  <xbrli:period>
    <xbrli:instant>2026-03-31</xbrli:instant>
  </xbrli:period>
  <xbrli:scenario>
    <xbrldi:explicitMember dimension="jpcrp_cor:MajorShareholdersAxis">jpcrp_cor:No1MajorShareholdersMember</xbrldi:explicitMember>
  </xbrli:scenario>
</xbrli:context>
```

### Dimension 軸の実例（サンプルから確認）

| 軸 (dimension) | 名前空間 | 用途 | メンバー例 |
|----------------|---------|------|-----------|
| `ConsolidatedOrNonConsolidatedAxis` | `jppfs_cor` | 連結/個別 | `NonConsolidatedMember`（デフォルト = 連結のため省略） |
| `OperatingSegmentsAxis` | `jpcrp_cor` | 事業セグメント | `ReportableSegmentsMember`, `UnallocatedAmoundsAndEliminationMember`, 提出者固有メンバー |
| `MajorShareholdersAxis` | `jpcrp_cor` | 大株主 | `No1MajorShareholdersMember` 〜 `No10MajorShareholdersMember` |

### 推論（Reasoning）

1. [F1]+[F5] より、EDINET の Context 構造は固定的: entity の scheme は `http://disclosure.edinet-fsa.go.jp` の1種類のみ、identifier は `EDINETコード-追番`、segment は使用しない。これは全提出者で共通。

2. [F2]+[F3]+[F4] より、Context ID は命名規則で決定論的に構築される。パーサーでは Context ID 文字列を解析して「相対期間」「時点/期間」「メンバー一覧」を抽出できるが、実装上は Context の XML 構造（period, scenario）から直接値を取得する方が確実。

3. [F7]+[F8] より、EDINET ではディメンション情報を **scenario** に設定し、**segment は不使用**。これは XBRL Dimensions 仕様の2つのオプション（segment / scenario）のうち scenario を選択したことを意味する。パーサーでは segment の存在チェックは不要（存在しないことが保証）。

4. [F7]+[F8]+[F13] より、デフォルトメンバーの場合は scenario 自体が省略される。連結/個別軸のデフォルトは連結（`ConsolidatedMember`）であり、連結の Fact には `ConsolidatedOrNonConsolidatedAxis` が scenario に設定されない。パーサーでは scenario がない Context は全ディメンションがデフォルト値であることを意味する。

5. [F14]+[F15] より、ディメンション軸のメンバーには標準タクソノミ定義のもの（`jpcrp_cor:No1MajorShareholdersMember`）と提出者別タクソノミ定義のもの（`jpcrp030000-asr_X99001-000:CommunicationsEquipmentReportableSegmentMember`）がある。後者は企業固有のセグメント名等。

6. [F6]+[F9]+[F11] より、period は `instant` か `duration` (startDate + endDate) の2択。日付形式は `YYYY-MM-DD` 固定でタイムゾーン情報はない。パーサーでは `datetime.date` で十分（`datetime.datetime` は不要）。`forever` 期間は EDINET で使用されない。

7. [F12] より、Context 総数は有報1件で 278 個程度。辞書 (`dict[str, Context]`) で十分に管理可能で、メモリ上の問題は生じない。

8. Typed Dimension について: 仕様書 §5-4-5 では `xbrldi:explicitMember` のみ記載されており、`xbrldi:typedMember` への言及がない。サンプル .xbrl にも Typed Dimension は出現しない。EDINET では Explicit Dimension のみ使用と結論できる。

### 確信度

- **Context ID 命名規則**: 高（仕様書の詳細な規定 + サンプル全 Context の一致で確認済み）
- **entity/identifier**: 高（scheme が1種類であることを仕様書 + サンプルで確認）
- **scenario 使用（segment 不使用）**: 高（仕様書 図表 5-4-6 で「segment: 指定しない」と明記）
- **Typed Dimension 不使用**: 中-高（仕様書に記載なし + サンプルに出現なし。否定の証明は完全ではないが、実務上は Explicit のみと判断して問題ない）
- **period 形式**: 高（仕様書 + サンプルで YYYY-MM-DD 固定を確認。タイムゾーン付加の記載なし）
- **forever 不使用**: 高（仕様書に forever の記載なし + サンプルに出現なし）
