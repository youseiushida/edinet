## §1. 訂正報告書は全体を再提出する

**結論**: 訂正報告書（doc_type 130, 150, 170 等）は**訂正箇所のみではなく、全体を再提出**する。訂正後の XBRL 書類一式（提出者別タクソノミ、報告書インスタンス、マニフェストファイル）を完全に添付する。原本（訂正前）の `doc_id` でも引き続きダウンロード可能であり、API 上 `withdrawalStatus` は取下げがない限り `"0"` のまま変化しない。

### 情報源 (Fact)

- **F1**: P-1.a.md [F5] — 識学 (S100VUX5, doc_type=130) の ZIP 構造: ファイル数 50、.xbrl 3件、.htm 26件、.xsd 3件、.xml 11件。通常の有報（P-1.a.md [F1]-[F4]）と同一構造の完全な ZIP。ファイル名に `_02_`（2回目の提出）が含まれる。
- **F2**: B-3.a.md — doc_type=130/150/170 の実データ分析: いずれも AuditDoc あり、BS/PL/CF を含む完全な財務諸表が存在。
- **F3**: EDINETタクソノミの概要説明.md L.653-659 — 「訂正後のインライン XBRL 書類を添付して提出する」と明記。
- **F4**: EDINET概要書.md L.285-296 — 「訂正後の XBRL 書類一式（提出者別タクソノミ、報告書インスタンス及びマニフェストファイル）を添付して、訂正報告書を提出します。」

### 推論 (Reasoning)

F3・F4 の仕様書により、訂正報告時には「訂正後のインライン XBRL 書類」を添付する仕様であり、F5（フレームワーク設計書）でも「訂正対象書類の訂正後インラインXBRLドキュメント一式」と述べられていることから、差分ではなく全置換であることが明確。F1・F2 の実データでも、訂正報告書 ZIP の構造が原本と同一（PublicDoc + AuditDoc の完全なファイルセット）であることを確認済み。訂正報告書の提出は原本の取下げを意味しないため、原本は `withdrawalStatus="0"` のまま API でダウンロード可能（F4）。

## §2. 差分メカニズムは存在しない（全置換方式）

**結論**: EDINET の XBRL には訂正前との差分を表現する仕組みは**存在しない**。訂正は**全置換**方式であり、訂正後の XBRL ファイル一式が丸ごと再提出される。作成方法も初回提出と変わらない。

### 情報源 (Fact)

- **F5**: EDINETフレームワーク設計書.md L.1206-1210 — 「訂正対象書類の訂正後インラインXBRLドキュメント一式を、添付書類として提出する。」差分ファイルの概念は言及なし。
- **F6**: EDINETタクソノミの概要説明.md L.706 — 「提出者別タクソノミの作成において初回提出の場合又は訂正再提出の場合で、作成方法が変わることはありません。」

### 推論 (Reasoning)

F5 の「ドキュメント一式」および F6 の「作成方法が変わることはありません」から、XBRL のデータモデル上に差分を表現するメカニズムは存在しない。訂正後のファイルは原本と同一フォーマットの完全なインスタンスであり、パーサーの観点では原本と訂正報告書を区別なく処理できる。差分の特定が必要な場合はアプリケーション側で原本と訂正後の Fact を比較する必要がある。

## §3. XBRL 内に訂正元 doc_id への参照がある

**結論**: DEI 要素 `jpdei_cor:IdentificationOfDocumentSubjectToAmendmentDEI` により、XBRL ヘッダー内に訂正元の書類管理番号（doc_id）への参照が**存在する**。原本ではこの要素は `xsi:nil="true"` となる。

### 情報源 (Fact)

- **F7**: 2026サンプルインスタンス dir 20（S021XXXX、訂正有価証券報告書）header L.246-250:
  ```xml
  <ix:nonFraction name="jpdei_cor:NumberOfSubmissionDEI" ...>2</ix:nonFraction>
  <ix:nonNumeric name="jpdei_cor:AmendmentFlagDEI" ...>true</ix:nonNumeric>
  <ix:nonNumeric name="jpdei_cor:IdentificationOfDocumentSubjectToAmendmentDEI" ...>S002XXXX</ix:nonNumeric>
  <ix:nonNumeric name="jpdei_cor:ReportAmendmentFlagDEI" ...>true</ix:nonNumeric>
  <ix:nonNumeric name="jpdei_cor:XBRLAmendmentFlagDEI" ...>false</ix:nonNumeric>
  ```
- **F8**: 2026サンプルインスタンス dir 02（S002XXXX、原本の有価証券報告書）header L.246-250:
  ```xml
  <ix:nonFraction name="jpdei_cor:NumberOfSubmissionDEI" ...>1</ix:nonFraction>
  <ix:nonNumeric name="jpdei_cor:AmendmentFlagDEI" ...>false</ix:nonNumeric>
  <ix:nonNumeric name="jpdei_cor:IdentificationOfDocumentSubjectToAmendmentDEI" ... xsi:nil="true"/>
  ```
- **F9**: EDINETタクソノミの概要説明.md L.680 — 「DEI の内容については、『提出回数』及び『訂正の有無』を更新し、『訂正対象書類の書類管理番号』を設定します。」
- **F10**: EDINETタクソノミの概要説明.md L.694-698 — 訂正の種類フラグ: 記載事項訂正の場合は `ReportAmendmentFlagDEI=true`、XBRL のみ訂正の場合は `XBRLAmendmentFlagDEI=true`。両方が同時に true になることはない。

### 推論 (Reasoning)

F7 の `IdentificationOfDocumentSubjectToAmendmentDEI=S002XXXX` により、訂正報告書 S021XXXX は原本 S002XXXX を訂正対象として明示的に参照している。F8 で原本では `xsi:nil="true"` であることから、この要素の有無で訂正報告書かどうかを判定できる。F9 の仕様書でもこの設定が義務付けられている。F10 により、訂正の種類（記載事項訂正 vs XBRL のみ訂正）も DEI フラグで識別可能。

## §4. 複数回訂正時の関係

**結論**: 原本→第1回訂正→第2回訂正の場合:
- ファイル名の提出回数が `_01_` → `_02_` → `_03_` とインクリメントされる
- `NumberOfSubmissionDEI` も 1 → 2 → 3 とインクリメントされる
- `IdentificationOfDocumentSubjectToAmendmentDEI` は**常に原本の doc_id** を参照する（2026サンプルの実データによる）

### 情報源 (Fact)

- **F11**: 2026サンプルインスタンスの3段チェーン（特定有価証券府令、Y99009）:

  | dir | doc_id | ファイル名提出回数 | NumberOfSubmissionDEI | AmendmentFlagDEI | IdentificationOfDocument... |
  |-----|--------|-------------------|----------------------|-----------------|---------------------------|
  | 11 | S022XXXX | `_01_` | 1 | false | `xsi:nil="true"` |
  | 21 | S023XXXX | `_02_` | 2 | true | S022XXXX |
  | 22 | S024XXXX | `_03_` | 3 | true | S022XXXX |

- **F12**: EDINETタクソノミの概要説明.md L.684-686 — 「ファイル一式の各ファイル名の{提出回数}をインクリメント（1ずつ増加）します。」「ファイル一式の各ファイル名の{提出回数}が一致していることを確認します。」
- **F13**: dir 22 のファイル名: `jpsps060700-drs-001_Y99009-000_2027-01-31_03_2027-06-26_ixbrl.htm` — `_03_` が3回目の提出を示す。

### 推論 (Reasoning)

F11 の実データにより、S023XXXX（第1回訂正, `_02_`）も S024XXXX（第2回訂正, `_03_`）も、`IdentificationOfDocumentSubjectToAmendmentDEI` は**いずれも原本 S022XXXX を参照**している。これは「訂正対象書類の書類管理番号」が「直前の訂正」ではなく「原本」を指すことを意味する。F12 の仕様で「訂正対象書類の書類管理番号を設定する」とあり、訂正対象は原本（最初に提出された書類）である。ファイル名と NumberOfSubmissionDEI の提出回数は 1 → 2 → 3 と順次インクリメントされ、何回目の提出かを追跡できる（F12, F13）。

## §5. API `parentDocID` と XBRL 内参照の関係

**結論**: API レスポンスの `parentDocID` と XBRL 内の `IdentificationOfDocumentSubjectToAmendmentDEI` は**同じ訂正元（原本）の doc_id** を指す。API の `parentDocID` のみで訂正チェーンを辿ることが可能であり、XBRL 内の DEI 要素は追加の確認手段として機能する。取下げの確認には `withdrawalStatus` を使用する。

### 情報源 (Fact)

- **F14**: EDINETAPI仕様書.md L.1060-1061 — `parentDocID` (No.30): 「親書類管理番号が出力されます。」半角8文字。
- **F15**: EDINETAPI仕様書.md L.1066-1067 — `withdrawalStatus` (No.32): 取下書は `"1"`、取り下げられた書類は `"2"`、それ以外は `"0"`。
- **F16**: EDINETAPI仕様書.md L.1423-1614 — 取下げの例: 親書類が取下げられると子書類も `withdrawalStatus="2"` に更新される。`parentDocID` により親子関係が保持される。

### 推論 (Reasoning)

F14 により API の `parentDocID` は訂正報告書から原本への参照を提供し、§4 の F11 で確認した XBRL 内の `IdentificationOfDocumentSubjectToAmendmentDEI` と同じ原本を指す。したがって:

1. **API のみで訂正チェーンを辿ることは可能**: `parentDocID` でチェーンを遡れる
2. **XBRL 内 DEI は冗長だが有用**: API にアクセスせず XBRL ファイルだけで訂正関係を把握したい場合に使える
3. **取下げの確認**: F15-F16 により `withdrawalStatus` で取下書（`"1"`）と取下げられた書類（`"2"`）を識別でき、取下げ済みの書類をフィルタできる

パーサー実装の観点では、API の `parentDocID` + `withdrawalStatus` で訂正チェーンの管理と取下げフィルタが完結し、XBRL 内の DEI を個別にパースする必要はない。
