# Z-2. 科目定義 JSON の自動生成可能性 — Presentation Linkbase からの導出

## 背景

本ライブラリの `statements.py` は手動作成の JSON ファイル（`pl_jgaap.json`, `bs_jgaap.json`, `cf_jgaap.json`）を科目リスト・表示順序の定義に使用している。これらの JSON はタクソノミの Presentation Linkbase（`_pre_*.xml`）に含まれる情報の手動コピーである可能性がある。

タクソノミの年次更新時にこれらの JSON を手動メンテナンスする保守コストがどの程度か、また Presentation Linkbase をランタイムまたはビルド時に解析することで自動生成可能かを調査する。

FEATURES.md の `pres_tree`（Presentation Linkbase の木構造解析）実装の優先度判断にも直結する。

## 質問

### Z-2.1 現在の JSON とタクソノミの対応関係

`pl_jgaap.json` / `bs_jgaap.json` / `cf_jgaap.json` の各 concept は、タクソノミの Presentation Linkbase（`_pre_*.xml`）にどの程度含まれているか？差分がある場合、その原因は何か？

### Z-2.2 Presentation Linkbase のファイル構造

一般事業会社（`cns`）の _pre ファイルはどのように分割されているか？PL / BS / CF それぞれについて、1つの _pre ファイルで完結するか、複数ファイルのマージが必要か？

### Z-2.3 業種別タクソノミの構造

23 業種ディレクトリのそれぞれで _pre ファイルはいくつ存在するか？業種固有の概念セット（銀行業の CF 等）も同じ構造で自動抽出可能か？

### Z-2.4 deprecated concept の自動検出

タクソノミの `deprecated/` ディレクトリから、廃止された concept の一覧と代替先を自動的に抽出可能か？

### Z-2.5 自動化の限界

タクソノミから自動生成できない辞書（マッピング）は何か？手動メンテが不可避な領域を特定する。
