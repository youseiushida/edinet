## §1. jptoi / jptoo-* タクソノミの概要

**結論**: 公開買付関連書類は **2つの府令** にまたがり、合計 **6つのタクソノミ分割単位** で管理される。自社株買付（jptoi）と他社株買付（jptoo-*）に大別され、jptoo は書類の様式ごとに5つのサブモジュールに分割されている。

### 情報源 (Fact)

- **F1**: 仕様書「EDINETタクソノミの概要説明」L.905-917: 図表 2-3-3（他社株買付府令）:

| No | 分割単位 | 書類種別 | 様式番号 |
|----|----------|---------|---------|
| 1 | jptoo-ton | 公開買付届出書 | 第二号様式 |
| 2 | jptoo-pst | 意見表明報告書 | 第四号様式 |
| 3 | jptoo-wto | 公開買付撤回届出書 | 第五号様式 |
| 4 | jptoo-tor | 公開買付報告書 | 第六号様式 |
| 5 | jptoo-toa | 対質問回答報告書 | 第八号様式 |

- **F2**: 同 L.919-929: 図表 2-3-4（自社株買付府令）:

| No | 分割単位 | 書類種別 | 様式番号 |
|----|----------|---------|---------|
| 1 | jptoi | 公開買付届出書 | 第二号様式 |
| 2 | jptoi | 公開買付撤回届出書 | 第三号様式 |
| 3 | jptoi | 公開買付報告書 | 第四号様式 |

- **F3**: 各 XSD の規模:
  - `jptoo-ton_cor`: 239行（最大。届出書の詳細項目）
  - `jptoo-pst_cor`: 41行（意見表明報告書）
  - `jptoo-wto_cor`: 55行（撤回届出書）
  - `jptoo-tor_cor`: 65行（公開買付報告書）
  - `jptoo-toa_cor`: 31行（対質問回答報告書）
  - `jptoi_cor`: 147行（自社株買付3様式共有）

- **F4**: バージョン日付: jptoo-ton, jptoo-pst = 2025-11-01（最新更新）。jptoo-wto, jptoo-tor, jptoo-toa = 2013-08-31（長期未更新）。jptoi = 2025-11-01。

### 推論 (Reasoning)

F1-F2 により、他社株買付は様式ごとに独立したタクソノミ（5モジュール）、自社株買付は3様式を1つのタクソノミ（jptoi）で管理する設計であることがわかる。F4 のバージョン日付から、jptoo-wto/tor/toa は2013年以降更新がなく安定した書類であることが推察される。一方、jptoo-ton/pst と jptoi は2025年に更新されており、制度変更への対応がなされている。

## §2. 公開買付価格・買付期間の構造化

**結論**: 公開買付価格・買付期間等の情報は主に **テキストブロック** として記載される。一部の数値要素（金額・株式数・割合）が構造化されているが、買付価格や買付期間そのものを直接表す専用の数値要素は限定的。

### 情報源 (Fact)

- **F5**: `G-6.verify_jptoi_jptoo.py` による jptoi_cor / jptoo-ton_cor XSD の要素型分析:

  **jptoi_cor（147行、136要素）— 数値型ゼロ:**
  - `stringItemType`: 82要素（Heading + 表紙情報）
  - `textBlockItemType`: 53要素（買付目的、買付条件、決済方法等）
  - `dateItemType`: 1要素（提出日）
  - `monetaryItemType` / `sharesItemType` / `percentItemType`: **全て0**

  **jptoo-ton_cor（239行、227要素）— 数値型あり:**
  - `stringItemType`: 122要素
  - `textBlockItemType`: 89要素
  - `nonNegativeIntegerItemType`: 10要素（議決権数関連）
  - `dateItemType`: 4要素
  - `percentItemType`: 2要素（保有割合関連）

- **F6**: jptoo-ton_cor XSD が公開買付タクソノミ中最大（239行）。テキストブロック中心だが、議決権数（`nonNegativeIntegerItemType`）と保有割合（`percentItemType`）は構造化されている。

- **F7**: サンプルインスタンス 16_自社株買付: ファイル `jptoi020000-ton-001_X99013-000_2026-09-28_01_2026-09-28.xbrl`。Context は `FilingDateInstant` のみ。テキストブロック要素にHTML内容が含まれる。

- **F8**: 提出者別タクソノミ作成ガイドライン L.97-99: 公開買付届出書、撤回届出書、公開買付報告書、対質問回答報告書が XBRL 作成ツール対象。

### 推論 (Reasoning)

F5 の XSD 分析（`G-6.verify_jptoi_jptoo.py` による機械的確認）から、**jptoi_cor は完全にテキスト専用タクソノミ**（数値型ゼロ）である。一方、**jptoo-ton_cor には議決権数（nonNegativeIntegerItemType: 10要素）と保有割合（percentItemType: 2要素）の構造化数値要素が存在**する。公開買付タクソノミ全体としてはテキストブロック中心だが、jptoo-ton に限り部分的な数値構造化がなされている。Dimension 構造はいずれも確認されない。

F7 のサンプルでは Context が `FilingDateInstant` のみであり、期間ベースの財務データは含まれていない。公開買付の具体的な条件（価格、期間、株式数等）はテキストブロック内のHTML表として表現されることが多く、構造化された数値 Fact としての抽出は限定的。

## §3. エンティティ・関係表現

**結論**: 1つの ZIP 内には原則として単一エンティティ（提出者）の Fact のみが含まれる。対象者・買付者等の関係は Context の `entity` ではなく、テキストブロック内の記述で表現される。Dimension による複数エンティティの区別は公開買付タクソノミでは使用されない。

### 情報源 (Fact)

- **F9**: サンプルインスタンス 16_自社株買付: entity の identifier は提出者の EDINET コード1つのみ。対象会社等の情報はテキストブロック要素内に記載。
- **F10**: jptoo-ton_cor XSD / jptoi_cor XSD: Dimension 関連要素（hypercubeItem, dimensionItem）は定義されていない。
- **F11**: 公開買付の対象者（意見表明報告書の提出者）は別の ZIP（別の提出書類）として EDINET に提出する。つまり、買付者の届出書と対象者の意見表明報告書は別々の Filing として扱われる。

### 推論 (Reasoning)

F10 により、公開買付タクソノミには Dimension 構造がなく、大量保有報告書（jplvh）のような複数主体の構造化は行われていない。F11 により、買付者と対象者はそれぞれ別の Filing として EDINET に提出するため、1つの ZIP 内に複数エンティティの構造化データが混在することはない。F9 のサンプルでも entity は1つのみ。

パーサーの観点では、公開買付関連書類は通常の有報等と同様に単一エンティティとして処理すればよい。ただし、テキストブロック内容から対象会社名等を抽出したい場合は HTML パースが必要になる。
