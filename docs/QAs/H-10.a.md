## H-10. パーサーの入力インターフェース制約

### 質問への対応

| サブ質問 | 回答 |
|----------|------|
| H-10.1 入力は `bytes`（ZIP 全体）か展開済みファイルパスか | ZIP 全体の `bytes` を入力とし、内部で必要ファイルを展開する方式を推奨。展開は `BytesIO` + `zipfile` でメモリ上で完結可能 |
| H-10.2 iXBRL IXDS の複数ファイル同時参照 | 必要。ヘッダ `.htm`（Context/Unit 定義）+ 本文 `.htm`（Fact）を同時に参照する。一時ディレクトリ展開は不要で、`BytesIO` から個別に読み出し可能 |
| H-10.3 提出者 `.xsd` と標準タクソノミの同時参照 | 必要。提出者 `.xsd` は ZIP 内、標準タクソノミは `ALL_20251101` のファイルシステム上。lxml カスタムリゾルバで統合解決可能 |
| H-10.4 ZIP 内ファイルとローカルタクソノミの統合解決 | lxml の `XMLParser` + カスタム `Resolver` で対応可能。一時ディレクトリ展開は不要 |
| H-10.5 `xml:base` 属性 | 仕様書に言及なし。サンプルデータにも確認されず。無視して問題ない |
| H-10.6 `Filing.fetch()` の返り値と必要ファイル群 | 新 API（例: `filing.xbrl_bundle()`）を追加し、ZIP 内の全 XBRL 関連ファイルへのアクセスを提供すべき |

### 情報源（Fact）

- [F1] パス: `stubs/edinet/models/filing.pyi` — 現在の `Filing.fetch()` は `tuple[str, bytes]` を返す（primary XBRL のパスとバイト列のタプル。XBRL が無い場合は `EdinetAPIError` を raise）
- [F2] パス: `src/edinet/api/download.py` — `extract_zip_member()` が ZIP 内の個別ファイルを `bytes` として取得可能。`find_primary_xbrl_path()` が primary XBRL を特定。安全制限: MAX_ZIP_BYTES=50MB, MAX_MEMBER_BYTES=20MB
- [F3] パス: `docs/QAs/K-2.a.md` — iXBRL の IXDS 構造: ヘッダファイル（`0000000_header_*.htm`）に `ix:header` / `ix:resources`（Context/Unit 定義）、本文ファイル（`0101010_honbun_*.htm` 等）に Fact がタグ付けされる
- [F4] パス: `docs/QAs/H-1.a.md` — `schemaRef` は提出者 XSD への相対パス。リンクベースは XSD の `appinfo` セクションの `link:linkbaseRef` で参照
- [F5] パス: `docs/QAs/H-1b.a.md` — 提出者 XSD の `xs:import` は絶対 URL（`http://disclosure.edinet-fsa.go.jp/taxonomy/...`）。変換規則: `http://disclosure.edinet-fsa.go.jp/taxonomy/{path}` → `ALL_20251101/taxonomy/{path}`
- [F6] パス: `docs/QAs/B-1.a.md` — ZIP 構造: `PublicDoc/` に `.xbrl`, `.xsd`, `_lab.xml`, `_lab-en.xml`, `_pre.xml`, `_cal.xml`, `_def.xml` が配置。`AuditDoc/` に監査報告書の `.xbrl` と `_pre.xml`, `_def.xml`
- [F7] パス: `docs/QAs/H-5.a.md` — ZIP サイズは最大でも数 MB。`BytesIO` でのメモリ上展開は現実的

### 推論（Reasoning）

1. **入力インターフェース設計** ([F1] + [F2] + [F7]): 現在の `Filing.fetch()` は primary XBRL 1 ファイルのパスとバイト列のタプルのみ返すが、パースには ZIP 内の複数ファイルが必要。ZIP 全体の `bytes` を保持し、必要に応じて `extract_zip_member()` で個別ファイルを取り出す方式が最も効率的。H-5 より ZIP は最大でも数 MB であり、メモリ上で完結可能。

2. **iXBRL の複数ファイル参照** ([F3]): IXDS ではヘッダ `.htm` と複数の本文 `.htm` を同時に参照する必要がある。ただし各ファイルは独立した XML 文書であり、lxml で個別にパースしてから Python オブジェクト上でマージすればよい。一時ディレクトリへの展開は不要。

3. **ZIP 内ファイルとローカルタクソノミの統合解決** ([F4] + [F5]): 提出者 `.xsd` は ZIP 内（`bytes`）、標準タクソノミは `ALL_20251101`（ファイルシステム）にある。統合解決の方式は以下の 2 案:
   - **案 A（推奨）: Python レベルの自前解決**: lxml による XML Schema のフル検証は行わず、提出者 `.xsd` を XML として読み取り、`xs:import` の `schemaLocation` URL を [F5] の変換規則でローカルパスに変換して個別に読む。リンクベースも同様に `link:linkbaseRef` の `xlink:href` を相対パス解決して ZIP 内から取り出す。
   - **案 B: lxml カスタムリゾルバ**: `lxml.etree.Resolver` を継承し、`resolve()` メソッドで URL パターンに応じて ZIP 内 `bytes` またはローカルファイルを返す。lxml の `XMLSchema` / `parse()` がこのリゾルバを使って外部参照を解決する。
   - 案 A は実装が単純で依存が少ない。案 B は lxml の schema validation を活用したい場合に有用。v0.1.0 では案 A を推奨。

4. **`xml:base` 属性**: 仕様書に `xml:base` への言及がなく、サンプルデータにも確認されない。EDINET では全ファイルが同一ディレクトリ（`PublicDoc/`）内に配置され、相対パスは単純なファイル名のみ（ディレクトリ階層なし）のため、`xml:base` によるベース URI 変更は発生しない。

5. **`Filing.fetch()` の拡張提案** ([F1] + [F6]):
   - 現在の `fetch()` は primary XBRL 1 ファイルのパスとバイト列（`tuple[str, bytes]`）のみを返す
   - パースに必要なファイル群: `.xbrl`（または `.htm` × N）、`.xsd`、`_lab.xml`、`_lab-en.xml`、`_pre.xml`、`_cal.xml`、`_def.xml`
   - **推奨**: ZIP 全体の `bytes` を保持する `XbrlBundle` クラスを導入し、`filing.xbrl_bundle()` で取得。内部で `zipfile.ZipFile(BytesIO(zip_bytes))` を保持し、`bundle.primary_xbrl()`, `bundle.schema()`, `bundle.linkbase("pre")` 等のアクセサを提供
   - ファイル間参照関係: `.xbrl` → `.xsd`（schemaRef）→ `_lab.xml`, `_pre.xml` 等（linkbaseRef）→ 標準タクソノミ XSD（xs:import）。全て同一ディレクトリ内の相対パスまたは絶対 URL
