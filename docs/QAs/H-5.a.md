## H-5. ファイルサイズの目安

### 質問への対応

| サブ質問 | 回答 |
|----------|------|
| H-5.1 一般的な有報 `.xbrl` のサイズ範囲 | 12KB〜13MB。中央値 26KB（大半は小規模 filing）、大企業の有報は数 MB 規模 |
| H-5.2 巨大な `.xbrl` のケース | 最大約 13MB（スクリプト結果）。P-1.a のトヨタは約 9MB。`etree.parse()` で問題ないサイズ |
| H-5.3 パース方法の選択基準 | `etree.parse()` 一括読み込みで十分。`iterparse` は不要 |
| H-5.4 ZIP 全体のサイズ | 有報: 574KB〜1.5MB（平均 956KB）。最大でも 3.2MB 程度。`BytesIO` での処理は現実的 |

### 推論（Reasoning）

1. [F1] + [F2] より、一般的な有報の `.xbrl` ファイルは数十 KB〜13MB の範囲にある。大半は 1MB 以下だが、大企業（トヨタ等）では数 MB に達する。13MB でも `etree.parse()` の一括読み込みで数秒以内にパース可能であり、`iterparse` は不要。

2. [F1] より、ZIP 全体のサイズは有報で平均 956KB、最大でも 3.2MB 程度。[F3] の仕様上限 100MB と比較して非常に小さい。`BytesIO` でのメモリ上展開は完全に現実的（100MB の上限ケースでも展開後 200MB 程度でメモリ上処理可能）。

3. [F4] より、iXBRL `.htm` ファイルは 1 ファイルあたり 2.5MB 以下。[F1] で確認した最大 5.7MB は複数 `.htm` ファイルの合計であり、個別ファイルは上限内に収まっているはず。

4. [F1] の最大 filing（イシン株式会社、訂正有報）でも ZIP 3.2MB、展開後 `.xbrl` 合計 3.3MB + `.htm` 合計 6MB 程度。バッチ処理で 1,000 社を処理する場合でも、1 社ずつ処理すれば最大でも数十 MB のメモリで十分。

5. **パース方法の結論**: `etree.parse()` / `etree.fromstring()` の一括読み込みで全ケースに対応可能。ストリーミングパーサーやファイルシステムへの一時展開は不要。既存の `BytesIO` ベースの設計（`extract_zip_member()`）をそのまま維持してよい。

### 確信度

- 高（59 filing の実データ統計 + 既存回答の実測データ + 仕様書の上限値で三重に確認）
- 注意: サンプルに含まれていない特殊な filing（巨大な投資信託有報等）では ZIP が数十 MB に達する可能性がある。ただし仕様上限 100MB を超えることはない
