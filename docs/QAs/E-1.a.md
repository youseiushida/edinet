### 識別ルール

#### 原則: 同一 concept が複数の財務諸表に出現しうる

EDINET タクソノミでは、全ての財務4表の concept が `jppfs_cor` 名前空間に定義されている。同一の concept（例: `NetAssets` = 純資産）が BS と SS の両方に出現するため、concept 名だけでは「この Fact は BS のもの」と判断できない。

**決定的な判別基準は Role URI（ELR）である**。Presentation linkbase の `<link:presentationLink xlink:role="{roleURI}">` に設定された roleURI がどの財務諸表のロールかを判定することで、concept を正しい財務諸表に分類できる。

#### Role URI による分類方法

I-1.a.md で整理した 3 つの方法のうち、いずれかで分類可能:

**方法 A: roleID のキーワードパターンマッチング**（推奨）

最も直接的かつ言語非依存な方法。role URI の末尾（roleID）に含まれる英語キーワードで判定する。

| 財務諸表 | roleID キーワード | 例 |
|----------|------------------|-----|
| BS | `BalanceSheet` | `rol_std_ConsolidatedBalanceSheet` |
| PL | `StatementOfIncome` | `rol_std_ConsolidatedStatementOfIncome` |
| CI | `StatementOfComprehensiveIncome` | `rol_std_ConsolidatedStatementOfComprehensiveIncome` |
| SS | `StatementOfChangesInEquity` or `StatementOfChangesInNetAssets` | `rol_std_ConsolidatedStatementOfChangesInEquity` |
| CF | `StatementOfCashFlows` | `rol_std_ConsolidatedStatementOfCashFlows-indirect` |

注意: 連結 PL のロール `ConsolidatedStatementOfIncome` の definition は「連結損益（及び包括利益）計算書」であり、「包括利益計算書」を含む。`StatementOfComprehensiveIncome` より先に `StatementOfIncome` をマッチングし、CI と誤判定しないよう注意。

**方法 B: ELR 番号による判定**

`3____` 番台の 2〜4 桁目で本表種類を判定する（I-1.a.md より）:

| 2〜4 桁目 | 財務諸表 |
|-----------|---------|
| 100 | BS（貸借対照表） |
| 210 | PL（損益計算書） |
| 220 | CI（包括利益計算書） |
| 300 | SS（株主資本等変動計算書） |
| 410 | CF 直接法 |
| 420 | CF 間接法 |

ELR 番号は definition テキストの先頭に `310010 連結貸借対照表` のように記載されている。

**方法 C: definition の日本語キーワードマッチング**

`link:definition` テキスト内のキーワードで判定する。ただし PL と CI の曖昧性あり（方法 A 推奨）。

### jpcrp_cor は財務諸表本表に含まれない

`jpcrp_cor`（開示府令語彙）の concept は経営指標等（EPS、BPS、従業員数等）や有価証券報告書の各セクション（事業の状況、設備の状況等）に使われるが、財務4表（BS/PL/CF/SS）の role URI 配下には含まれない。財務4表の role URI 配下の concept は全て `jppfs_cor` に属する。

これは E-1b.a.md で確認した通り、jpcrp_cor は「経理の状況」以外のセクションのデータを定義する名前空間であり、財務諸表本表は jppfs_cor が担当する分業構造になっている。

### 推論（Reasoning）

1. **分類基準が名前空間プレフィックスではない理由**: [F7] より、全財務4表の concept は同一の `jppfs_cor` 名前空間に属する。同一 concept（例: `NetAssets`）が BS と SS の両方の role URI 配下に出現しうるため、concept 名や名前空間だけでは財務諸表の種類を判定できない。[F1] の Role URI が、concept をどの財務諸表に帰属させるかの唯一の確実な手段である。

2. **Role URI による判定の信頼性**: [F1]+[F2]+[F4]+[F5] より、EDINET の roleID 命名規約は厳密に定義されている。英語キーワード（`BalanceSheet` 等）と ELR 番号体系の両方が利用可能であり、いずれも一意な判定を提供する。方法 A（roleID キーワード）が最も直接的で言語非依存、方法 B（ELR 番号）が最も曖昧さが少なく堅牢。

3. **jpcrp_cor の除外**: [F6] より、jpcrp_cor は有報の非財務セクションのデータを定義する名前空間である。EPS/BPS 等の 1 株当たり情報は経営指標等セクション（`jpcrp_cor`）に属し、財務4表の role URI（`3____` 番台）配下には含まれない。パーサーが財務4表を処理する際は `jppfs_cor` の concept のみを対象とすればよい。

4. **CI と PL の区別**: [F3] より、日本基準では 2 つの方式がある。1 計算書方式（PL と CI を一体として作成）の場合は `ConsolidatedStatementOfIncome` ロールに包括利益の科目も含まれる。2 計算書方式の場合は `ConsolidatedStatementOfIncome` と `ConsolidatedStatementOfComprehensiveIncome` が別ロールとして存在する。roleID のキーワードマッチで先に `StatementOfIncome` をチェックすることで正しく判定できる。
