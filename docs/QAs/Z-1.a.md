## EDINET DB の名寄せ構造

### Layer 1: 標準 element_id マッピング（141 パターン）

24 指標 × 3 会計基準 = 最大 72 の基本パターン。一部指標は特定の会計基準にのみ存在する（経常利益は JP GAAP のみ、税引前利益は IFRS/US GAAP のみ）。加えて、営業利益の JP GAAP フォールバック（PL 本体からの取得）等の優先順位付きマッピングが加わり合計 141 パターン。

**重要**: element_id は `jpcrp_cor:*SummaryOfBusinessResults` 系（有報の「経営指標等の推移」セクション）であり、`jppfs_cor:*`（財務諸表本体）ではない。

### Layer 2: 正規表現フォールバック（~5% カバー）

IFRS 企業の独自タクソノミ拡張（`jpcrp030000-asr_EXXXXX` 形式）を正規表現で捕捉する。除外パターンを併用:

- Revenue regex: "Intersegment", "Cost", "Gain", "Loss" を除外
- Net income regex: "Ordinary", "Operating", "BeforeTax", "Gross", "Equity", "Earnings", "Business" を除外
- Operating income regex: 同様のフィルタ

### 連結/個別の処理

2 パス方式:
- Pass 1: 連結データを取得（`NonConsolidatedMember` を除外）
- Pass 2: ~525 社の個別のみ企業にフォールバック（ただし EPS・設備投資・従業員数は除外）

### 品質パイプライン（30 ルール）

- STEP 1（16 ルール）: Layer 1/2 のマッピング適用
- STEP 2（1 ルール）: 行→列ピボット（1 社 1 年 = 24 指標の 1 行）
- STEP 3（11 ルール）: 閾値ベースの異常値 NULL 化（EPS > 100 万円、BPS > 1000 万円、配当性向 > 200% 等）
- STEP 4（2 ルール）: コア指標（売上高・純利益・総資産・自己資本）の NULL 率モニタリング

## 推論（Reasoning）

### 1. Layer 1 の設計思想

[F1] より、Layer 1 は「会計基準ごとに異なる element_id を直接マッピングで統一する」アプローチ。141 パターンは 24 指標 × 3 基準の基本 72 パターンに加え、JP GAAP の営業利益のようにサマリーテーブルに専用 element がない場合の PL 本体へのフォールバック等が加わった数。

### 2. Layer 2 の設計思想

[F1] より、Layer 2 は IFRS 企業の独自タクソノミ拡張を正規表現で捕捉するフォールバック。~5% のカバレッジを担う。除外パターン（"Intersegment", "Cost" 等）により誤捕捉を防ぐ。本ライブラリの `_collect_extra_items()` + `_looks_like_cf()` [F7] と概念的に対応するが、アプローチが異なる（サフィックスヒューリスティック vs 正規表現除外パターン）。

### 3. 品質パイプラインの特徴

[F1] より、EDINET DB は 30 ルールの品質パイプラインで異常値を NULL 化する。これはスクリーニング目的のサービスとして合理的だが、本ライブラリの「元データを忠実に構造化する」方針（CODESCOPE.md §5「ルールに合致しない Fact は捨てない」）とは異なるアプローチ。

### 4. データソースの根本的な違い

[F2] + [F4][F5][F6] より、EDINET DB と本ライブラリは **異なる XBRL タグからデータを取得している**:

- EDINET DB: `jpcrp_cor:NetSalesSummaryOfBusinessResults`（有報の「経営指標等の推移」テーブル）
- 本ライブラリ: `jppfs_cor:NetSales`（損益計算書本体の XBRL Fact）

これは同じ「売上高」でも情報源が異なる。サマリーテーブル（`SummaryOfBusinessResults`）は 5 期分の主要指標を 1 filing 内に含むが、勘定科目の内訳（販管費の明細、営業外収益の内訳等）は含まない。財務諸表本体（`jppfs_cor`）は当期の全勘定科目を含むが、5 期分の比較は複数 filing が必要。

### 5. 本ライブラリの現在の実装との対応関係

| 観点 | EDINET DB | 本ライブラリ（v0.1.0） |
|------|-----------|----------------------|
| データソース | `jpcrp_cor:*SummaryOfBusinessResults` | `jppfs_cor:*` |
| 粒度 | 24 指標（サマリー） | 44 concept（勘定科目レベル） |
| 会計基準 | JP GAAP / IFRS / US GAAP | JP GAAP のみ |
| 分類方法 | element_id 直接マッピング | JSON concept 集合 + periodType 振り分け |
| 非標準科目 | 正規表現フォールバック | サフィックスヒューリスティック + namespace 判定 |
| 異常値処理 | 閾値ベースの NULL 化 | なし（元データ忠実） |
| 目的 | 企業横断スクリーニング | 1 社の財務諸表構造化表示 |
| 時系列 | 1 filing で 5 期分 | 1 filing で 1 期分 |
| 内訳の深さ | 合計のみ | 勘定科目の明細まで |

### 6. 採用判断

[F8] より、IFRS / US GAAP 対応は v0.1.0 スコープ外。Layer 1 の element_id マッピングは全会計基準対応が前提のため、v0.1.0 の `statements.py` に直接導入するのは不適切。

**v0.2.0+ での推奨配置**:

1. **`SummaryOfBusinessResults` 層を独立モジュールとして追加** — FEATURES.md の `dei` / `eda/summary` に該当する位置づけ。`statements.py` の PL/BS/CF 組み立てとは独立
2. **Layer 1 のマッピングテーブルを参考に `standards/normalize` を設計** — 24 指標の正規化キーは edgartools の `standardization/` に相当
3. **現在の `statements.py`（財務諸表本体の構造化）は維持** — `SummaryOfBusinessResults` では取れない勘定科目の内訳が本ライブラリの差別化要因

両者は **代替ではなく補完** の関係にある:

```
ユースケース A（スクリーニング）:
  SummaryOfBusinessResults → 24 指標 → 3848 社横断比較
  → EDINET DB 相当。jquants の代替

ユースケース B（深掘り）:
  jppfs_cor → PL/BS/CF 全勘定科目 → 1 社の財務分析
  → 本ライブラリ v0.1.0 の現在の提供価値

理想形: 両方を提供し、A で絞り込んだ企業を B で深掘りするワークフロー
```
