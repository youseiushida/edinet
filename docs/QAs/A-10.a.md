- [F6] A-9.a.md [F1]-[F4] — `textBlockItemType` は `escape="true"` によりXHTML タグをインスタンス値に含める。エスケープされた HTML 構造の一部として空白・改行が意味を持つ

### 推論（Reasoning）

1. [F1]+[F2] より、EDINET は `stringItemType`（レイアウトが意味を持たない）と `textBlockItemType`（レイアウトが意味を持つ）を明確に区別している。この区別が空白の扱いの基本方針を規定する。

2. [F2]+[F3]+[F4] より、`textBlockItemType` のインスタンス値はエスケープされた HTML 文字列であり、改行・空白・インデントは HTML 構造の一部として保持されている。`<p> </p>` のような空白のみの要素も意味を持つ可能性がある（段落間の余白表現等）。したがって TextBlock のテキスト値を strip してはならない。

3. [F5] より、XML パーサーは要素コンテンツの空白を正規化しない（`xs:string` 型の標準動作）。lxml 等の標準的な XML パーサーで `.xbrl` をパースすれば、TextBlock の値は改行・空白を含めてそのまま取得される。追加の空白処理は不要。

4. [F1] より、`stringItemType` は「レイアウトの有無が意味の理解に影響しない」とされている。したがって空白・改行に意味的な重要性はない。前後の空白を strip しても実害は少ないが、仕様書に「strip すべき」という明示的な指示はない。安全策としてそのまま保持するのが望ましい。

5. [F5] より、XML の要素コンテンツは `xs:string` 型として扱われるため、`xs:normalizedString`（改行→空白変換）や `xs:token`（前後空白除去+連続空白圧縮）の正規化は**適用されない**。パーサーで独自の正規化処理を実装する必要はない。

### パーサー実装への示唆

| データ型 | 空白の扱い | 推奨実装 |
|---|---|---|
| `textBlockItemType` | 保持（レイアウトが意味を持つ） | strip しない。HTML 構造として処理する |
| `stringItemType` | 意味を持たない | strip しなくても動作するが、表示時に前後 strip は許容 |
| DEI 要素等（`stringItemType`） | 意味を持たない | 前後 strip しても実害なし |

### 確信度

- 高（仕様書の明示的な型区分 + サンプル `.xbrl` の実例で空白保持を確認済み）
- `stringItemType` の前後 strip の可否: 中（仕様書に明示的な指示なし。実害はないが仕様書の裏付けがない）
- XML パーサーの空白正規化動作: 高（XBRL 2.1 / XML 1.0 仕様の標準動作に基づく）
