# H-3.a: タクソノミバージョニング

## 質問

バージョン間互換性、deprecated 構造、名前空間 URI のバージョン依存性を明らかにする。（9問）

## 品質

確度: **A（実証済み）** — ローカルタクソノミ解析 + 実 Filing ダウンロードで検証。

## 回答

### H-3.1: ALL_20251101（2025年版）で2024年以前の書類をパースできるか

**結論: できない（そのままでは）。**

- [F1] タクソノミの名前空間 URI にはバージョン日付が含まれる（H-3.6 参照）
- [F2] 2024年提出書類は `2022-11-01` 版や `2023-12-01` 版を参照する（H-3.4 参照）
- [F3] `ALL_20251101` には `2025-11-01` 版のモジュールのみ含まれ、古いバージョンのファイルは存在しない
- [R1] [F1]+[F2]+[F3] → 古い書類の `xs:import` URL は `ALL_20251101` 内で解決できない。ただし、概念の `local_name` は大部分が共通のため、ラベル引きは local_name フォールバックで対応可能（Q-2 で詳述）

### H-3.2: deprecated ディレクトリの内容

- [F4] `deprecated/` ディレクトリは各モジュールに存在し、以下の deprecated XSD を含む:
  - `jppfs_dep_2025-11-01.xsd`: **229要素**
  - `jpcrp_dep_2025-11-01.xsd`: **531要素**
  - `jpigp_dep_2025-11-01.xsd`: **538要素**
- [F5] deprecated XSD の `targetNamespace` は正規 XSD と **同一** の名前空間（例: `http://disclosure.edinet-fsa.go.jp/taxonomy/jppfs/2025-11-01/jppfs_cor`）
  - 仕様書確認: 設定規約書の図 2-2-3 ※2「deprecatedボキャブラリスキーマの名前空間プレフィクスと名前空間 URI は、非推奨化前の名前空間プレフィクスと名前空間 URI と同一」
  - **出典**: `EDINETタクソノミの設定規約書.md` L.873-879
- [F6] deprecated XSD には `_lab.xml` と `_lab-en.xml` のラベルファイルが付属（日英両方）
- [F7] deprecated 要素は旧→新の明示的マッピング情報を**持たない**（XSD の element 定義のみ）
  - 例: `MinorityInterests` → `NonControllingInterests` の対応は推論が必要
- **出典**: `H-3.versioning.py` + ローカルファイル直接読取

### H-3.3: ラベル辞書の分離 vs 統合

- [R2] [F1] 名前空間 URI にバージョン日付が含まれるため、`{namespace_uri}:{local_name}` をキーにするとバージョンごとに別エントリになる
- [R3] **推奨: local_name でラベル辞書を統合し、バージョン非依存にする**。deprecated 要素のみ旧ラベルを保持する別テーブルで対応。
  - 根拠: H-3.9 で確認した通り、deprecated 要素数は全体の5-10%程度であり、大部分の概念は `local_name` が安定している

### H-3.4: schemaRef のバージョンバインディング

- [F8] 各年の提出書類が参照するタクソノミバージョン（スクリプト実測）:

| 提出時期 | 企業名 | 参照バージョン |
|---|---|---|
| 2025年6月 | 三井住友DSアセットマネジメント | `2013-08-31`, `2023-12-01` |
| 2024年6月 | 大和アセットマネジメント | `2013-08-31`, `2022-11-01` |
| 2023年6月 | 三井住友DSアセットマネジメント | `2013-08-31`, `2021-11-01` |
| 2022年6月 | 三井住友DSアセットマネジメント | `2013-08-31`, `2020-11-01` |
| 2021年6月 | アセットマネジメントOne | `2013-08-31`, `2019-11-01` |
| 2020年6月 | エービーシー・マート | `2013-08-31`, `2018-03-31` |

- [F9] P-3.a.md [F3] より: トヨタ（2025年6月提出、2025-03-31 期）は `2024-11-01` 版を参照
- [R4] [F8]+[F9] → **タクソノミバージョンは提出日ではなく、適用されるタクソノミ更新時期に基づく**。通常、提出日の前年11月1日版が使われるが、投信等は異なるバージョン体系を持つ場合がある。
- [R5] `2013-08-31` は DEI (jpdei) と common (iod) のバージョンであり、これらは2013年以降変更されていない固定モジュール。
- **出典**: `H-3.versioning.py` 実行結果

### H-3.5: ALL_20251101 単体利用の可否

- [R6] [F3]+[R1] → **ALL_20251101 単体では過去全書類のパースは不可**
- 対策として以下が必要:
  1. 各年の ALL_YYYYMMDD を保持する（最大10バージョン）、または
  2. local_name ベースのフォールバックでラベルを引く（精度はやや落ちるが実用的）

### H-3.6: 名前空間 URI のバージョン依存性

- [F10] **全ての EDINET タクソノミモジュールの名前空間 URI にバージョン日付が含まれる**:
  - `http://disclosure.edinet-fsa.go.jp/taxonomy/jppfs/2025-11-01/jppfs_cor`
  - `http://disclosure.edinet-fsa.go.jp/taxonomy/jpcrp/2025-11-01/jpcrp_cor`
  - `http://disclosure.edinet-fsa.go.jp/taxonomy/jpigp/2025-11-01/jpigp_cor`
  - `http://disclosure.edinet-fsa.go.jp/taxonomy/jpdei/2013-08-31/jpdei_cor`（固定）
  - `http://disclosure.edinet-fsa.go.jp/taxonomy/common/2013-08-31/iod`（固定）
- [R7] [F10] → 同一概念（例: `Assets`）でもバージョンが異なれば名前空間 URI が異なるため、`{namespace_uri}:{local_name}` キーではバージョン間でマッチしない。ラベル辞書のキーは `local_name` のみ、またはモジュール名 + local_name（例: `jppfs_cor:Assets`）とすべき。
- **出典**: `H-3.versioning.py` + ローカルタクソノミ直接読取

### H-3.7: 年内パッチの有無

- [F11] 2026年タクソノミ更新概要（`2026年EDINETタクソノミ更新概要.ocr.md` sec.4-5）には年内パッチの記述はない
- [F12] ALL_YYYYMMDD の YYYYMMDD はリリース日であり、通常は年1回の `YYYY-11-01` 更新
- [R8] [F11]+[F12] → **年内パッチは原則なし**。ただし、緊急対応として臨時更新がありうる可能性は否定できないが、実運用上は年次更新のみを考慮すれば十分。

### H-3.8: バージョン間差分規模

- [F13] deprecated 要素数（累積、2025-11-01 版時点）:
  - jppfs: **229要素**（財務諸表）
  - jpcrp: **531要素**（府令）
  - jpigp: **538要素**（IFRS）
- [F14] これは過去全バージョンからの累積 deprecated であり、単年の差分ではない
- [F17] タクソノミ差分情報 HTML（`docs/仕様書/2026/タクソノミ差分情報/`）の統計ページ（page5.html）から、2024-11-01 → 2025-11-01 の単年差分（スキーマ要素 = concept レベル）を実測:

| モジュール | 追加 | 削除 | 変更 |
|---|---|---|---|
| jppfs | 8 | 6 | 325 |
| jpcrp | 27 | 6 | 325 |
| jpigp | 27 | 7 | 395 |
| jplvh | 48 | 33 | 3 |
| jpsps | 160 | 120 | 6505 |
| jptoi | 3 | 42 | 3 |
| jptoo-pst | 4 | 7 | 1 |
| jptoo-ton | 15 | 48 | 1 |

  - ※ jppfs は全23業種で同一（追加8、削除6）。jpcrp は報告書タイプにより若干の差異あり（上表は有報 jpcrp030000-asr の値）
  - ※ 「変更」は属性変更（バージョン番号の書換え等）を含むため数値が大きいが、concept の実質的な追加/削除は少ない
  - **出典**: `H-3.diff_stats.py` 実行結果

- [R9] [F17] → **concept の単年追加は最大27件、削除は最大7件（主要3モジュール: jppfs/jpcrp/jpigp）**。旧回答の「年約20-50要素」推定は概ね妥当だった。実装負荷は小さく、local_name フォールバック戦略で十分に吸収可能。

### H-3.9: deprecated 要素の使用状況

- [F15] deprecated 要素の名前パターン分析（jppfs 229要素）:
  - `MinorityInterest` 関連: 7要素（→ `NonControllingInterests` への改名）
  - `DeferredTax` 関連: 8要素（流動/固定の区分変更）
  - `Provision` 関連: 14要素
  - `Amortization` 関連: 12要素
  - `Prior` 関連: 5要素（前期損益修正の廃止）
- [F16] deprecated XSD の `targetNamespace` は正規版と同一 → deprecated 要素は正規要素と**同一名前空間に属する**
- [R10] [F15]+[F16] → 古い Filing は deprecated 要素を使用している可能性が高い（例: 2015年以前の Filing で `MinorityInterests` を使用）。パーサーは deprecated 要素のラベルも保持すべき。警告は不要（正常な過去データ）。
