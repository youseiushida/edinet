## 推論（Reasoning）

### 1. PL の Presentation Linkbase 構造（分割バリアント方式）

[F1] + [F2] より、一般事業会社(cns)の PL は単一の `_pre_pl.xml` では完結せず、以下のバリアントファイルに分割されている:

```
_pre_pl.xml                                     → 営業利益以降（OperatingIncome〜ProfitLoss）
_pre_pl-2-PL-01-Sales-1-Net.xml                 → 売上高区分（Net方式: NetSales, CostOfSales, GrossProfit）
_pre_pl-2-PL-01-Sales-3-ByType.xml              → 売上高区分（種類別方式）
_pre_pl-2-PL-04-SGA-1-ByAccount.xml             → 販管費区分（勘定科目別明細）
_pre_pl-2-PL-04-SGA-2-OneLine.xml               → 販管費区分（一括表示）
_pre_pl-2-PL-09-CI-1-SingleStatementNetOfTax.xml → 包括利益（税効果後一計算書方式）
_pre_pl-2-PL-09-CI-2-SingleStatementBeforeTax.xml→ 包括利益（税効果前一計算書方式）
```

提出者の entrypoint スキーマが上記のどのバリアントを import するかで、使用する concept セットが決まる。**完全な PL は `_pre_pl.xml` + 選択されたバリアントのマージ**で構成される。

JSON にのみ存在していた 4 concept のうち 3（NetSales, CostOfSales, GrossProfit）は Sales バリアントに、1（SellingGeneralAndAdministrativeExpenses）は SGA バリアントに存在する。つまり **全 16 JSON concepts はタクソノミ内に存在**し、ファイル分割が差分の原因。

### 2. BS の Presentation Linkbase 構造（スーパーセット方式）

[F1] より、BS は JSON の 20 concepts が Presentation の 68 concepts の完全な部分集合。JSON は「主要科目」のみを選択しており、Presentation は詳細科目（個別の引当金、繰延税金資産等）まで全て含む。

pres_tree を実装すれば **bs_jgaap.json の 20 concepts + 追加の 48 concepts が自動取得可能**。JSON の手動メンテは不要になる。

### 3. CF の Presentation Linkbase 不在問題

[F5] + [F6] + [F7] より、一般事業会社(cns)には CF の _pre ファイルが存在しない。これは：

- 銀行業(bk1)には `_pre_cf-in-*`（間接法）と `_pre_cf-di-*`（直接法）が存在 [F6]
- `jpigp` には IFRS 用 CF が存在 [F7]
- cns の CF concept は `jppfs_cor` の XSD で定義されているが、cns 固有の Presentation バリアントがない

つまり **J-GAAP 一般事業会社の CF は Presentation Linkbase からの自動生成が直接にはできない**。代替手段として：
- Calculation Linkbase（`_cal_pl.xml` 等と同様に `_cal_cf*` がないか要確認）
- 実データから CF role URI（`rol_*StatementOfCashFlows*`）を持つ concept を収集
- 銀行業(bk1)の CF Presentation から `jppfs_cor` concept のみを抽出し、業種固有（`*BNK` サフィックス）を除外

### 4. バリアントファイルのマージ戦略

[F1] + [F2] より、pres_tree の実装では以下の戦略が必要：

1. **単一財務諸表 = 複数 _pre ファイルのマージ**: PL は `_pre_pl.xml` + バリアントの合成
2. **バリアント選択**: 提出者の entrypoint スキーマの import 先から、どのバリアントが使用されているかを判定
3. **role URI による統合**: 全バリアントが同一の role URI（例: `rol_std_ConsolidatedStatementOfIncome`）を共有しており、role 単位でマージ可能

### 5. 業種別の自動拡張

[F3] より、23 業種の全てが同一 XML 構造の _pre ファイルを持つ。業種コードの違い（`cns`, `bk1`, `sec` 等）はファイル名に反映されるが、XML 構造は同一。

FEATURES.md の `sector/*`（銀行業・保険業・証券業等）は、pres_tree パーサー 1 つで全業種をカバーできる。業種ごとの手動 JSON は不要。

### 6. 自動化の限界 — 会計基準間マッピング

以下は自動化可能:

| 辞書 | ソース | 自動化可否 |
|------|--------|-----------|
| PL/BS concept リスト + order | `_pre_*.xml`（複数ファイルのマージ） | **自動** |
| CF concept リスト（J-GAAP 一般事業会社） | 業種別 _pre から共通 concept 抽出、または実データ | **半自動** |
| 業種別 concept セット | `r/{業種コード}/` 配下の _pre_*.xml | **自動** |
| ラベル（日本語/英語） | `_lab.xml` / `_lab-en.xml` | **自動**（実装済み） |
| 計算関係（weight, 親子） | `_cal_*.xml` | **自動** |
| deprecated concept 一覧 | `deprecated/*.xsd` + `*_lab.xml` | **自動** |
| deprecated → 代替先のマッピング | deprecated XSD 内に機械可読な参照なし | **半自動**（ラベル名の類似度で推定可能） |
| **会計基準間マッピング（J-GAAP↔IFRS）** | **公式データが存在しない** | **手動** |

会計基準間マッピングが手動不可避な理由:
- J-GAAP の `jppfs_cor:NetSales`（売上高）と IFRS の `ifrs-full:Revenue`（収益）が「同じ意味」であるという公式データは金融庁も IFRS 財団も提供していない
- Z-1 の EDINET DB も Layer 1 で手動マッピングを使用（141 パターン）
- ただし主要指標は 50-100 程度であり、一度作成すれば年次メンテは軽微
