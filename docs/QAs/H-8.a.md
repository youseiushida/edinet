## H-8. 実データの品質・エラー率

### 質問への対応

| サブ質問 | 回答 |
|----------|------|
| H-8.1 仕様違反の提出書類の存在程度 | EDINET は提出時にバリデーションを実施するため、XBRL 2.1 レベルの構文エラーは原則排除されている。ただしベストプラクティス（FRTA/FRIS/GFM）違反は例外規定があり存在しうる |
| H-8.2 よくあるエラーパターン | 構文エラーは稀。品質上の問題としてはラベル欠損（拡張要素の英語ラベル未設定）、計算リンクの不整合（丸め誤差）が発生しうる |
| H-8.3 strict/lenient mode の方針 | 計算リンクの不整合は警告にとどめる lenient mode を推奨。構文レベルは strict で問題ない |
| H-8.4 「該当なし」「N/A」の表現方法 | `xsi:nil="true"` が主要な表現手段。テキスト文字列の「該当なし」要素も存在。空文字列や `-` 文字列は使用されない |

### 情報源（Fact）

- [F1] パス: `docs/仕様書/EDINETタクソノミの設定規約書.md` L.2221-2306 — FRTA/FRIS/GFM の例外事項一覧。計 20 項目以上の例外が定義されており、これらの項目は仕様書レベルで「仕様に準拠しなくてよい」と明示されている
- [F2] パス: `docs/仕様書/XBRL Instance Guideline 報告書インスタンス作成ガイドライン.md` L.1750-1763 — `xsi:nil="true"` の使用方法: 数値項目に値がない場合（ダッシュ「−」表記）、`ix:nonfraction` 要素に `xsi:nil="true"` を設定する。`unitRef` は残るが `decimals` は省略
- [F3] パス: `docs/仕様書/XBRL Instance Guideline 報告書インスタンス作成ガイドライン.md` L.1960-1971 — 「該当なし」の処理: 目次項目に該当する内容がない場合、EDINET タクソノミに用意された「該当なし」要素（例: `NotApplicableXxx`）を使用する
- [F4] パス: `docs/QAs/A-3.a.md` — 実データ（トヨタ）での確認: `xsi:nil="true"` の使用実例あり。`decimals` 分布は -6（百万円）, 0（円）, 2, 3, 4。値のないセルでは空要素 + `xsi:nil="true"` で表現
- [F5] パス: `docs/仕様書/XBRL Instance Guideline 報告書インスタンス作成ガイドライン.md` L.1766-1772 — 「△0（マイナスゼロ）」は `"-0"` として値を記載する。`xsi:nil` ではなく実値として扱う
- [F6] パス: `docs/仕様書/EDINETタクソノミの設定規約書.md` L.2286-2287 — GFM 例外: インラインXBRLにおける重複 Fact の許容、Footnote linkbase の role 属性の扱い等
- [F7] スクリプト: `docs/QAs/scripts/H-8.quality.py` 実行結果 — 15 filing（有報・訂正有報）の品質統計:
  - 総 Fact 数: 4,847
  - `xsi:nil="true"`: 406（全体の 8.4%）
  - 空要素（nil でない）: 0（0.0%）— nil 以外で値が空の Fact は存在しない
  - NotApplicable 系: 1 件（`PlaceForPublicInspectionCoverPageTextBlock` にテキスト「該当なし」）
  - マイナスゼロ: 0 件
  - decimals 分布: 0（1,057 件）, -3（960 件）, -6（499 件）, 4（58 件）, 3（39 件）, 2（26 件）
  - nil が多い concept: DEI 要素（`SecurityCodeDEI`, `FundNameInEnglishDEI` 等）が大半

### 推論（Reasoning）

1. [F1] より、EDINET は FRTA/FRIS/GFM に対して多数の例外規定を設けている。これは提出者のシステム制約に合わせた現実的な妥協であり、ベストプラクティス違反が公式に許容されていることを意味する。ただしこれらは「バグ」ではなく「仕様上の例外」であり、パーサーが特別に対処すべきものではない。

2. [F2] + [F4] + [F5] より、「データなし」の表現は以下の3パターンに整理される:
   - **数値 nil**: `xsi:nil="true"` + `unitRef` あり + `decimals` なし → パーサーは `None` として扱う
   - **マイナスゼロ**: 値 `"-0"` → パーサーは数値 `0`（または `Decimal("-0")`）として扱う
   - **テキスト「該当なし」**: EDINET タクソノミの `NotApplicable` 系 concept にテキスト値「該当事項はありません」等 → パーサーはテキストとして扱う

3. [F1] の例外事項一覧に「計算リンクの不整合」に関する明示的な例外はないが、丸め（`decimals` 属性による精度の差異）により計算合計が一致しないケースは XBRL 仕様上許容される（XBRL 2.1 §5.2.5.2 の丸め許容規定）。パーサーは計算検証を行う場合、丸め許容幅を設けた lenient mode が適切。

4. EDINET の提出フローにはバリデーションが組み込まれているため（提出書類はシステムでチェック後に受理される）、Well-formed でない XML や名前空間エラーのような構文レベルの問題は原則発生しない。パーサーは構文レベルでは strict で問題ない。

5. [F7] より、実データ 15 filing の品質は高い:
   - `xsi:nil="true"` が全 Fact の 8.4% で使用されており、パーサーでの nil 処理は必須。nil の大半は DEI 要素（投資信託の場合に証券コードが nil 等）
   - 空要素（nil でない）は 0 件であり、「nil 以外で値が空」というエッジケースは発生しない
   - NotApplicable 系はテキスト Fact（TextBlock）に「該当なし」という文字列が入る形が 1 件のみ。concept 名に `NotApplicable` を含むものは確認されなかった
   - マイナスゼロ（`-0`）は 0 件であり、仕様上は存在しうるが実際には稀
   - decimals=0（円単位）と decimals=-3（千円単位）が最多で、投資信託系の filing に多い

### 確信度

- 中〜高（仕様書の記述 + 15 filing の実データ検証で裏付け。ただしサンプル数が限定的であり、特に大企業の有報が少ない）
- 不確実な点: EDINET の提出時バリデーションの詳細仕様（何をチェックし何を通すか）は仕様書に明記されておらず、大企業の有報での品質統計は追加サンプリングが望ましい
