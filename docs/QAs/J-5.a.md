### 推論（Reasoning）

#### 1. 前期 Fact は同一の concept 名を使用する（J-5.1）

[F1] + [F3] + [F6] より、EDINET の XBRL において前期データと当期データは**完全に同一の concept 名・名前空間**を使用する。

- `jppfs_cor:NetSales`（売上高）は当期の `CurrentYearDuration` でも前期の `Prior1YearDuration` でも同じ要素
- `jppfs_cor:Assets`（資産合計）は `CurrentYearInstant` でも `Prior1YearInstant` でも同じ要素
- 名前空間 URI も同一（例: `http://disclosure.edinet-fsa.go.jp/taxonomy/jppfs/2025-11-01/jppfs_cor`）

前期/当期の区別は**Context ID の相対期間部分**（`CurrentYear` vs `Prior1Year`）と、Context 要素の `period` に設定される実際の日付で行う。concept 定義自体は時間非依存である。

パーサーにとっての実用的な含意: 前期 Fact を取得する際に concept 名のマッピング変換は不要。Context の period 日付で期間を識別し、concept 名でデータ項目を識別すればよい。

#### 2. Context ID の命名規則と前期データ（J-5.2）

[F1] + [F3] + [F5] より、前期データの Context ID は以下の体系的な命名規則に従う。

| 相対期間 | Duration Context ID | Instant Context ID | 意味 |
|---------|--------------------|--------------------|------|
| 当年度 | `CurrentYearDuration` | `CurrentYearInstant` | 当期のデータ |
| 前年度 | `Prior1YearDuration` | `Prior1YearInstant` | 1年前のデータ |
| 前々年度 | `Prior2YearDuration` | `Prior2YearInstant` | 2年前のデータ |
| n年度前 | `Prior{n}YearDuration` | `Prior{n}YearInstant` | n年前のデータ |

各 Context の `period` 要素には実際の日付が設定される。例えば:
- `CurrentYearDuration`: `2024-04-01` 〜 `2025-03-31`
- `Prior1YearDuration`: `2023-04-01` 〜 `2024-03-31`
- `Prior1YearInstant`: `2024-03-31`

Dimension メンバー（連結/個別、セグメント等）は Context ID の `_{メンバー要素名}` 部分で付加される。例: `Prior1YearDuration_NonConsolidatedMember`（前年度・個別）。

#### 3. 3期以上の比較データ（J-5.3）

[F4] + [F5] より、有報には最大5期分（`Prior4Year` まで）のデータが含まれる。

- **経営指標等の推移**: 有報の「主要な経営指標等の推移」は5期分の時系列データを含むため、`Prior1Year` 〜 `Prior4Year` の Context が使用される。内容は売上高、利益、総資産、自己資本比率、1株当たり情報等。
- **財務諸表本表**: BS/PL/CF は通常2期比較（当期 + 前期）のため、`CurrentYear` と `Prior1Year` のみ。ただし SS（株主資本等変動計算書）は遡及処理時に `Prior2Year` まで使用しうる。
- **3期比較の BS**: 一部の IFRS 適用企業では会計方針変更時に3期分の BS を開示することがある。この場合 `Prior2YearInstant` が使用される。

#### 4. 修正再表示・遡及適用（J-5.4）

[F2] + [F7] より、EDINET タクソノミには遡及処理に関する以下の仕組みがある。

**Dimension 軸**: `jppfs_cor:RetrospectiveApplicationAndRetrospectiveRestatementAxis`
- Axis 要素は jppfs_cor XSD に定義されている（`substitutionGroup=xbrldt:dimensionItem`）
- **ただし、対応する Member 要素（`RetrospectiveApplicationMember`, `RetrospectiveRestatementMember`）は jppfs_cor XSD には定義されていない**。これらの Member は提出者別タクソノミで定義される可能性がある。【検証時修正: 元の回答では「Member が XSD に存在する」としていたが、XSD 直接検証で不在を確認】

**金額型要素（3件）**: `CumulativeEffectOfRetrospectiveApplicationAndRetrospectiveRestatementFND`（遡及処理の累積的影響額）、`SurplusDeficitAtBeginningOfPeriodBeforeRetrospectiveApplicationAndRetrospectiveRestatementFND`（遡及処理前期首剰余金）、`CumulativeEffectsOfChangesInAccountingPolicies`（会計方針の変更による累積的影響額）。いずれも `monetaryItemType`, `instant`, `credit`。

**SS での遡及処理**: [F2] より、SS の「会計方針の変更による累積的影響額」行（concept: `CumulativeEffectsOfChangesInAccountingPolicies`）は前期末日のコンテキスト（前連結 SS なら `Prior2Year`、当連結 SS なら `Prior1Year`）でタグ付けされる。これにより、遡及処理の影響が期首残高の調整として表現される。

**PL/BS の前期比較データ**: 遡及適用が行われた場合、前期の PL/BS 比較データは**修正後の値が `Prior1Year` コンテキストに直接入る**。修正前の値は通常含まれない。つまり、当期有報に含まれる前期データは常に「最新の正しい値」（遡及適用後）である。

**修正前後の比較**: `RetrospectiveApplicationAndRetrospectiveRestatementAxis` は jppfs_cor XSD に定義されているが、対応する Member 要素（`RetrospectiveApplicationMember` 等）は jppfs_cor XSD には存在しない。提出者別タクソノミで Member を定義して使用する可能性はあるが、実務上は SS の累積的影響額行で修正の影響を把握し、PL/BS は修正後の値のみが提供されることが一般的である。【検証時修正: Member 不在を反映】
