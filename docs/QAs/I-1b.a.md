## I-1b. 提出者独自の roleType

### 質問への対応

| サブ質問 | 回答 |
|----------|------|
| I-1b.1 提出者 XSD で独自 roleType が定義されるケースはあるか | **サンプルインスタンスでは存在しない。** 全 56 件の XSD を走査し、提出者定義の `link:roleType` は 0 件であった。提出者は EDINET 標準タクソノミにあらかじめ定義された提出者用ロール（`rol_*`, `_std_` なし）を使用する。ただし仕様上は独自 roleType の定義が可能（設定規約書 L.1914-1916「開示書類等提出者が独自に追加する場合の詳細ツリーは、EDINET タクソノミで定義する同提出者用の拡張リンクロールの命名規約に従って、定義する」）。 |
| I-1b.2 独自 role URI のパターン | 仕様上の命名規約は `http://disclosure.edinet-fsa.go.jp/role/jp[府令略号](-[報告書略号])/rol_{ルート要素名}(-{修飾語}/-{連番})`（設定規約書 L.1466-1489）。`_std_` や業種略号を含まない。 |
| I-1b.3 独自 role と _pre.xml の関係 | はい。提出者が独自 roleType を定義した場合、対応する `_pre.xml` 内の `presentationLink` の `xlink:role` にその role URI を使用し、`roleRef` で定義場所を参照する。 |
| I-1b.4 独自 roleType の用途 | 仕様上は「複数の詳細ツリーにひも付く場合」や「提出者用の拡張リンクロールのみを用意」するケースで使用される（設定規約書 L.1485）。具体的には独自の注記セクションの詳細ツリーが想定される。ただしサンプルインスタンスでは全て EDINET 標準の提出者用ロールで賄えており、独自定義の実例は未確認。 |

### 情報源（Fact）

観察事実のみを記載する。推論や解釈は含めない。

- [F1] スクリプト: `docs/QAs/scripts/I-1b.custom_roles.py` 実行結果 -- 全サンプルインスタンスの XSD 走査結果: 56 ファイル走査、roleType 発見 0 件。「提出者の XSD に独自 roleType は定義されていない」
- [F2] スクリプト: `docs/QAs/scripts/I-1b.custom_roles.py` 実行結果 -- サンプル 02（有価証券報告書）の roleRef 照合: _pre.xml の roleRef 64 件全てが EDINET 標準タクソノミ（jppfs_rt + jpcrp_rt、計 1454 件）に定義済み。標準にない roleRef は 0 件。
- [F3] スクリプト: `docs/QAs/scripts/I-1b.custom_roles.py` 実行結果 -- jppfs ロールの内訳: 提出者用ロール（`rol_*`, std なし）11 件、EDINET タクソノミ用ロール（`rol_std_*`）0 件。提出者は `rol_ConsolidatedBalanceSheet`, `rol_BalanceSheet`, `rol_ConsolidatedStatementOfIncome` 等の提出者用ロールのみを使用。
- [F4] パス: `docs/仕様書/EDINETタクソノミの設定規約書.md` L.1466-1489 -- 「図表 3-2-36 開示書類等提出者用の拡張リンクロール」: `roleID` = `rol_{ルート要素名}(-{修飾語}/-{連番})`。`roleURI` は `http://disclosure.edinet-fsa.go.jp/role/jp[府令略号|dei](-[報告書略号])/` に続く名称。
- [F5] パス: `docs/仕様書/EDINETタクソノミの設定規約書.md` L.1914-1916 -- 「3-4-3 開示書類等提出者が新規追加する詳細ツリー」:「提出者別タクソノミで開示書類等提出者が独自に追加する場合の詳細ツリーは、EDINET タクソノミで定義する同提出者用の拡張リンクロールの命名規約に従って、定義する。また、その場合はジェネリックラベルリンクも追加する。」
- [F6] パス: `docs/仕様書/EDINETタクソノミの設定規約書.md` L.1485 -- ※1:「提出者用の拡張リンクロールのみを用意し、対応する EDINET タクソノミの関係リンクのルート要素が特定できない場合も、ルート要素として通常利用される EDINET タクソノミの要素を基に命名する。」
- [F7] 回答依存: I-1.a.md [F14] -- `docs/仕様書/EDINETフレームワーク設計書.md` L.910-930「EDINET タクソノミの拡張リンクロールと提出者用拡張リンクロール」: 対応関係は必ずしも 1 対 1 とならない。0 対 1（提出者のみ）のケースがある。
- [F8] スクリプト: `docs/QAs/scripts/I-1b.custom_roles.py` 実行結果 -- _def.xml の roleRef: 科目一覧ツリー用ロール（`ListOfAccounts` 付き）が 4 件含まれる。全 roleRef が標準タクソノミ内。

### 推論（Reasoning）

1. **I-1b.1 独自 roleType の存在**: [F1] の走査結果により、サンプルインスタンスの全 56 XSD ファイルに `link:roleType` 要素は 0 件であった。[F2] + [F3] により、提出者は EDINET 標準タクソノミにあらかじめ定義された提出者用ロール（`rol_*`, `_std_` なし）を使用している。しかし [F5] + [F6] + [F7] により、仕様上は提出者が独自に roleType を定義することが明確に許容されている。[F7] の「0 対 1（提出者のみ）」ケースがこれに該当する。サンプルインスタンスに独自 roleType がないのは、サンプルが標準的な開示のみを含むためであり、実際の提出書類では独自注記セクション等で独自 roleType が定義される可能性がある。

2. **I-1b.2 role URI パターン**: [F4] より、提出者用ロールの命名規約は EDINET タクソノミ用ロールから `_std_` や業種略号を除いたパターンである。基底 URI は `http://disclosure.edinet-fsa.go.jp/role/jp[府令略号](-[報告書略号])/` に続けて `rol_{ルート要素名}` を付与する。独自に追加する場合も同じ命名規約に従う（[F5]）。

3. **I-1b.3 独自 role と _pre.xml**: 仕様上、提出者が独自 roleType を定義した場合、XSD の `<xsd:annotation><xsd:appinfo>` 内に `<link:roleType>` を定義し、対応するリンクベースファイルの `roleRef` からその XSD を参照する。`presentationLink` の `xlink:role` 属性にその role URI を使用する。この構造は EDINET 標準のロールと同一パターンである。

4. **I-1b.4 用途**: [F5] により「独自に追加する場合の詳細ツリー」として使用される。[F6] によれば「提出者用の拡張リンクロールのみを用意し、対応する EDINET タクソノミの関係リンクのルート要素が特定できない場合」にも独自ロールが必要になる。具体的には、企業固有の注記事項のセクションが EDINET 標準のロールでカバーされていない場合に、提出者が独自ロールを定義して詳細ツリーを構築する。

### 確信度

- I-1b.1 独自 roleType の存在: **高** -- 56 ファイルの網羅的走査で 0 件を確認。仕様上の定義可能性も仕様書で確認。
- I-1b.2 role URI パターン: **高** -- 設定規約書に命名規約が明記。
- I-1b.3 独自 role と _pre.xml: **中** -- 仕様上の推論。実例がサンプルにないため直接検証は不可。ただし XBRL の標準的な仕組みと整合。
- I-1b.4 用途: **中** -- 仕様書の記述から推論。実例未確認。

### v0.1.0 への影響

パーサーは提出者の XSD に `link:roleType` が存在する可能性を考慮すべきだが、v0.1.0 では標準タクソノミのロールのみを対象とすれば実用上十分である。独自 roleType に対応するのは将来の拡張事項とする。
