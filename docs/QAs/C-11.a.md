## C-11. リンクベースのマージアルゴリズム（提出者 + 標準）

### 質問への対応

| サブ質問 | 回答 |
|----------|------|
| C-11.1 同一 role URI の extendedLink が両方に存在する場合 | XBRL 2.1 仕様ではアークは「加算（マージ）」される。EDINET では名称リンクのみがリユーズ方式のためマージ対象。標準と提出者の labelLink は共に `http://www.xbrl.org/2003/role/link` を使用するため、同一 role URI の extendedLink が両方に存在し、XBRL 2.1 マージルールが適用される。ただし提出者の labelArc は新規拡張要素向けが大半で、標準要素のラベルと競合するケースは英語ラベルの上書き時に限られる。 |
| C-11.2 同一 from/to の arc が複数ある場合の priority 解決 | XBRL 2.1 仕様: 等価な arc のうち priority が最も高いものが有効。同一 priority かつ同一 use の等価 arc が複数あると XBRL 2.1 では処理エラー。EDINET ではこれを回避するため「提出者別タクソノミ内で priority 属性の値が重複しないように」と規定。標準タクソノミの labelArc は全て priority=0（デフォルト）のため、提出者が priority=1 以上を使えば常に提出者側が優先される。 |
| C-11.3 use="prohibited" の効果 | 同一 from/to/arcrole の等価 arc のうち、priority が同等以下のものを無効化する。prohibited arc 自体も最終結果からは除外される。EDINET では名称リンクにおいてのみ prohibited が出現し得る。設定規約書が GFM Rule 1.4.3 を対象外とした理由が「ラベルの上書きを許可することで、名称リンクに use 属性が prohibited のアークが存在し得るため」と明記。 |
| C-11.4 適用範囲 | EDINET では**名称リンク（ラベルリンク）のみ**。表示リンク・定義リンク・計算リンクはリキャスト方式であり、prohibited は明示的に禁止されている。マージルール自体は XBRL 2.1 仕様として全リンクベース共通だが、EDINET の設計上マージが発生するのは名称リンクのみ。 |
| C-11.5 具体例 | q.md の例は表示リンクの例だが、EDINET では表示リンクはリキャスト方式のため、この例は発生しない。名称リンクでの具体例として、英語ラベル上書きのシナリオを推論セクションで示す。 |
| C-11.6 arc の等価性ルール | XBRL 2.1 仕様: `from` + `to` + `arcrole` が同一の arc が「等価」。`order` 属性は等価性判定に含まれない。名称リンクでは arcrole は常に `concept-label` であるため、等価性 = 同一の from（concept ロケーター）+ 同一の to（label リソース）。 |

### 推論（Reasoning）

#### C-11.1: 同一 role URI の extendedLink — 加算かどうか

1. [F10] より、標準タクソノミと提出者タクソノミの labelLink は共に拡張リンクロール `http://www.xbrl.org/2003/role/link` を使用している。同一の role URI を持つ extendedLink が両方に存在する。
2. XBRL 2.1 仕様（Section 3.5.3.9）では、同一 role URI の extendedLink 内のアークはマージ（加算）される。つまり「置換」ではなく「全てのアークが合わせて1つの extended link として扱われる」。
3. [F1] + [F14] より、名称リンクはリユーズ方式であり、提出者のラベルファイルは EDINET タクソノミのラベルを参照する形で動作する。DTS（Discoverable Taxonomy Set）に標準タクソノミのラベルファイルと提出者のラベルファイルが両方含まれるため、XBRL 2.1 のマージルールに従って labelArc が統合される。
4. ただし [F13] + [F9] より、実態としてサンプル提出者の labelArc は全て新規拡張要素向けであり、標準タクソノミの要素と同じ from/to を持つ arc は存在しない。よって競合は発生せず、結果的には単純な「加算（追加）」となっている。

**結論**: XBRL 2.1 仕様上はマージ（加算）。EDINET の実態でも加算。ただし英語ラベル上書き時のみ等価 arc が発生し、priority による解決が必要になる。

#### C-11.2: priority の解決ルール

5. [F8] より、標準タクソノミの全 47,352 labelArc は priority 属性を指定していない（XBRL 2.1 デフォルト = 0）。
6. [F2] より、提出者がラベルを上書きする場合「priority 属性の値が重複しないように」との注意がある。これは、提出者が複数のラベル上書きを行う場合に、提出者内で priority 値を一意にする必要があることを意味する。
7. XBRL 2.1 仕様（Section 3.5.3.9.7.4）: 等価な arc（同一 from/to/arcrole）が複数存在する場合、最も高い priority を持つ arc が有効。同一 priority かつ同一 use の等価 arc が複数あると処理エラー。
8. [F8] + 推論5-6 より、標準タクソノミの arc は priority=0 であるため、提出者が priority=1 以上の arc を定義すれば、常に提出者側が優先される。提出者内で複数の上書きを行う場合は、各上書きで異なる priority 値を使用する必要がある（[F2]の注意事項）。

**結論**: 最も高い priority が有効。標準=0、提出者>=1 により提出者が常に優先。提出者内で priority の重複は不可。

#### C-11.3: use="prohibited" の効果

9. [F4] より、EDINET は GFM Rule 1.4.3（ineffectual arcs の禁止）を対象外としている。その理由は「ラベルの上書きを許可することで、名称リンクに use 属性が prohibited のアークが存在し得るため」。これは名称リンクで prohibited が使用される可能性を EDINET が想定していることを示す。
10. [F9] より、サンプル提出者別タクソノミには prohibited を使用する labelArc は含まれていない。サンプルでは英語ラベルの上書き例が存在しないため。
11. XBRL 2.1 仕様（Section 3.5.3.9.7.4）: use="prohibited" の arc は、等価な arc のうち priority が同等以下のものを無効化する。prohibited arc 自体も最終的な effective arc の集合からは除外される。
12. [F3] より、英語ラベルの上書きが可能。上書きの手順として、(a) prohibited arc で標準の英語ラベルへの arc を無効化し、(b) 新しい label リソースへの arc を定義する、という2段階になると推測される。ただし、名称リンクでは from=concept ロケーター、to=label リソースであり、異なる label リソースへの arc は from/to が異なるため等価ではない。したがって、prohibited が必要になるのは「同一 concept から同一ラベルロールの label への arc を除去する」場合である。

**結論**: prohibited は等価な arc（同一 from/to/arcrole）のうち priority が同等以下のものを無効化する。名称リンクにおいてのみ出現し得る。

#### C-11.4: 適用範囲

13. [F5] + [F6] + [F7] より、表示リンク・定義リンク・計算リンクでは prohibited が明示的に禁止されている。
14. [F1] より、表示リンク・計算リンク・定義リンク（ディメンション）はリキャスト方式のため、標準タクソノミの同名 role URI の arc と共存する設計ではない（提出者が独立したファイルを新規作成する）。
15. 推論13-14 より、XBRL 2.1 のアークマージルールが EDINET で実際に適用される場面は名称リンクのみ。

**結論**: XBRL 2.1 仕様としてはリンクベース共通のルールだが、EDINET の設計上マージが発生するのは名称リンクのみ。

#### C-11.5: 具体例

16. q.md の例（表示リンクでの prohibited）は EDINET では発生しない（推論13-14）。名称リンクでの具体例を構成する:

**シナリオ: 英語ラベルの上書き**

標準タクソノミ（`jppfs_2025-11-01_lab-en.xml`）:
```xml
<link:labelLink xlink:role="http://www.xbrl.org/2003/role/link">
  <link:loc xlink:href="...#jppfs_cor_Assets" xlink:label="Assets"/>
  <link:label xlink:label="label_Assets" xlink:role=".../role/label"
              xml:lang="en">Assets</link:label>
  <link:labelArc xlink:from="Assets" xlink:to="label_Assets"
                 xlink:arcrole=".../concept-label"/>
  <!-- priority=0 (default), use=optional (default) -->
</link:labelLink>
```

提出者別タクソノミ（英語ラベル上書き）:
```xml
<link:labelLink xlink:role="http://www.xbrl.org/2003/role/link">
  <link:loc xlink:href="...#jppfs_cor_Assets" xlink:label="Assets"/>
  <!-- Step 1: 標準の英語ラベルを prohibited で除去 -->
  <link:label xlink:label="label_Assets_orig" xlink:role=".../role/label"
              xml:lang="en">Assets</link:label>
  <link:labelArc xlink:from="Assets" xlink:to="label_Assets_orig"
                 xlink:arcrole=".../concept-label"
                 use="prohibited" priority="1"/>
  <!-- Step 2: 新しい英語ラベルを追加 -->
  <link:label xlink:label="label_Assets_new" xlink:role=".../role/label"
              xml:lang="en">Total Assets</link:label>
  <link:labelArc xlink:from="Assets" xlink:to="label_Assets_new"
                 xlink:arcrole=".../concept-label"
                 priority="2"/>
</link:labelLink>
```

マージ結果:
- 標準の `Assets → label_Assets` (priority=0, use=optional) と提出者の `Assets → label_Assets_orig` (priority=1, use=prohibited) は、XBRL プロセッサの locator 解決により同一概念を指す。ただし to が異なる label リソースを指すため、厳密には等価ではない。
- 実際の上書きメカニズムは、同一概念 + 同一ラベルロール + 同一言語の組合せで、XBRL プロセッサが effective label を決定する形で動作する可能性が高い。

**注**: サンプルインスタンスには英語ラベル上書きの実例が含まれていないため（[F13]）、上記はフレームワーク設計書の記述（[F3]）と XBRL 2.1 仕様からの推測に基づく構成例である。

#### C-11.6: arc の等価性ルール

17. XBRL 2.1 仕様（Section 3.5.3.9.7.4）: 2つの arc が等価とは、`from`（ロケーターが解決する要素）、`to`（リソースまたはロケーターが解決する要素）、および `arcrole` が同一であることを意味する。
18. `order` 属性は等価性判定に含まれない。`order` は同一親の下での表示順序を制御するのみ。
19. 名称リンクでは、arcrole は常に `http://www.xbrl.org/2003/arcrole/concept-label` であるため、等価性は `from`（concept ロケーター） + `to`（label リソース）のみで決まる。

**結論**: arc の等価性 = from + to + arcrole。order は含まれない。名称リンクでは arcrole が固定のため、実質 from + to で判定。

- C-11.2 priority 解決: **高** -- XBRL 2.1 仕様 + EDINET ガイドラインの明示的な注意事項（[F2]）+ 実データで標準=priority 0 を確認（[F8]）

- C-11.4 適用範囲: **高** -- 表示/定義/計算で prohibited 禁止が3箇所に明記（[F5][F6][F7]）、リキャスト/リユーズの区別も明確（[F1]）
- C-11.5 具体例: **中** -- フレームワーク設計書の記述と XBRL 2.1 仕様からの推測に基づく構成例。サンプルインスタンスに英語ラベル上書きの実例がなく、実際の prohibited 使用パターンは未確認
- C-11.6 等価性ルール: **高** -- XBRL 2.1 仕様の明確な規定。order が等価性に含まれないことは仕様書に明記
