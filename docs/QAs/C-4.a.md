### 1. 連結区分サフィックスの完全な意味

[F1] の「報告期間の別」と「連結又は個別の別」を組み合わせたものがファイル名中の連結区分となる。

| サフィックス | 分解 | 意味 |
|---|---|---|
| `ac` | a(通期) + c(連結) | 通期・連結財務諸表 |
| `an` | a(通期) + n(個別) | 通期・個別財務諸表 |
| `cm` | c(科目一覧) + m(科目一覧) | 科目一覧（比較情報用の定義リンク）|
| `sc-t1` | s(中間期) + c(連結) + t1 | 中間期・連結・第一種中間財務諸表 |
| `sc-t2` | s(中間期) + c(連結) + t2 | 中間期・連結・第二種中間財務諸表 |
| `sn-t1` | s(中間期) + n(個別) + t1 | 中間期・個別・第一種中間財務諸表 |
| `sn-t2` | s(中間期) + n(個別) + t2 | 中間期・個別・第二種中間財務諸表 |

[F6] の実ファイル件数:

| 連結区分 | ファイル数 | cal | def | pre |
|---|---|---|---|---|
| an | 723 | 241 | 241 | 241 |
| ac | 671 | 223 | 224 | 224 |
| sn-t2 | 174 | 58 | 58 | 58 |
| sc-t2 | 143 | 47 | 48 | 48 |
| sn-t1 | 111 | 37 | 37 | 37 |
| sc-t1 | 98 | 32 | 33 | 33 |
| cm | 90 | 0 | 90 | 0 |

**注**: `cm` は定義リンク (def) のみで、計算・表示リンクを持たない。これは科目一覧ツリーが比較情報のための定義リンクのみを提供する特殊な用途であるため。

### 2. t1 と t2 の違い

[F2] ※5 より明確に定義:

> t1/t2 は、中間（連結）財務諸表における第一種中間（連結）財務諸表／第二種中間（連結）財務諸表の区別を表す。

- **t1 = 第一種中間（連結）財務諸表**: 中間期用の簡略化された財務諸表
- **t2 = 第二種中間（連結）財務諸表**: 中間期用のより詳細な財務諸表（通期と同等の開示水準）

[F6] の実データから、t2 の方がファイル数が多い（sc-t2: 143 > sc-t1: 98、sn-t2: 174 > sn-t1: 111）。これは第二種の方がバリエーション（パターンファイル）が多いことを意味し、より詳細な開示に対応していることと整合する。

### 7. バリエーション判別方法

企業がどのバリエーションを使用しているかは、提出者別タクソノミの XSD ファイル（`schemaRef` で参照される）が import する EDINET タクソノミのエントリーポイントから DTS（Discoverable Taxonomy Set）を辿ることで判明する。エントリーポイントは業種別・連結区分別に分かれており、そこからリンクベースファイル（基本ファイル）が参照される。パターンファイル（バリエーション）は提出者別タクソノミのリンクベースファイル内で追加的に参照される。

具体的には:
1. インスタンスの `schemaRef` → 提出者別タクソノミ XSD
2. 提出者別タクソノミ XSD の `import` → EDINET タクソノミのエントリーポイント
3. エントリーポイント → 基本リンクベース（`_bs.xml`, `_pl.xml` 等）
4. 提出者別タクソノミの `linkbaseRef` → パターンファイル（`_bs-1-BS-01-CA-Doubtful-1-ByAccount.xml` 等）

### 8. リンク種別ごとの命名規約の差異

[F2] より、リンク種別によりファイル名命名規約に微妙な差異がある:

| リンク種別 | 連結区分の後の付加情報 | CF区分 |
|---|---|---|
| 表示リンク (pre) | `-{t1\|t2}` | `-{di\|in}` |
| 定義リンク (def) | `-{q\|YTD}` | `-{di\|in}` |
| 計算リンク (cal) | `-{q[YTD]}` | `-{di[in]}` |

- `t1/t2`: 第一種/第二種中間財務諸表（表示リンクのみ）
- `q/YTD`: 四半期/年度累計（定義リンクのみ）

### 推論（Reasoning）

1. [F1] より、報告期間の略号は `a`(通期), `s`(中間期) の2種、連結/個別の略号は `c`(連結), `n`(個別) の2種。これらを組み合わせると `ac`, `an`, `sc`, `sn` の4種が基本パターンとなる。
2. [F2] ※5 より、t1/t2 は中間財務諸表の第一種/第二種の区別であり、中間期の連結/個別 (`sc`, `sn`) にのみ付加される。よって `sc-t1`, `sc-t2`, `sn-t1`, `sn-t2` が生まれる。
3. [F7] より、`cm` は科目一覧 (catalog/menu) を意味し、[F6] の実データでも `cm` は `def` リンクのみを持ち、`cal`/`pre` を持たないことから、比較情報のための定義のみを提供する特殊用途と確認。
4. [F4] の全パターン定義と [F6] の実ファイル検出結果を突合すると、仕様書に定義された全パターンが実際のタクソノミファイルとして存在することが確認できる。ただし実ファイルにはパターン名の先頭に `1-`（BS）, `2-`（PL）, `3-`（CF）, `4-`（CI）の諸表番号が付与されており、これは仕様書のパターン名と一致する。
5. [F6] より、`an`（個別）には `ci` パターンが存在しないことから、包括利益計算書は連結財務諸表でのみ使用されることが確認できる。同様に、`an` には `PL-09-CI` パターンも存在せず、1計算書方式も連結のみで使用される。
6. [F6] で検出された `pl-2-PL-01-Sales-1-Net-Lea` は仕様書の図表に明示的に記載されていない。これはリース業 (lea) 固有の売上高パターンと考えられる。
