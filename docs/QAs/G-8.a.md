## §1. jpsps タクソノミの用途

**結論**: `jpsps` は特定有価証券開示府令に基づく書類に使用されるタクソノミ。主に投資信託受益証券、投資証券（REIT等）、資産流動化証券、信託受益証券等の報告書に使用される。545行・534要素の大規模タクソノミで、25の様式（No.1-25）をカバーする（No.26の `jpsps-sbr`, No.27の `jpsps-esr` は別タクソノミ）。

### 情報源 (Fact)

- **F1**: 仕様書「EDINETタクソノミの概要説明」L.864-903: 図表 2-3-2: 特定有価証券開示府令のタクソノミ分割単位 → `jpsps`（No.1-25）+ `jpsps-sbr`（No.26: 自己株券買付状況報告書）+ `jpsps-esr`（No.27: 臨時報告書）。
- **F2**: jpsps_cor_2025-11-01.xsd: 全545行。Heading 要素名から対象書類種別が読み取れる:
  - FormNo4: 投資信託受益証券の有価証券届出書
  - FormNo43: 投資証券の有価証券届出書
  - FormNo7: 投資信託受益証券の有価証券報告書
  - FormNo73: 投資証券の有価証券報告書
  - FormNo10-12: 各種半期報告書
  - FormNo15/21: 発行登録書/追補書類
- **F3**: サンプルインスタンス 10_特定有価証券府令-有価証券届出書: `jpsps040000-srs-001_Y99009-000_...` → jpsps プレフィックス。
- **F4**: サンプルインスタンス 11_特定有価証券府令-有価証券報告書: `jpsps060700-drs-001_Y99009-000_...` → jpsps + みなし有価証券届出書(drs)。

### 推論 (Reasoning)

F1-F2 により、jpsps は投資信託・投資法人（REIT）等の特定有価証券に関する25の様式（No.1-25）を単一のタクソノミでカバーする大規模タクソノミである（No.26: jpsps-sbr, No.27: jpsps-esr は別タクソノミ）。F3-F4 のサンプルから、同一発行体の複数ファイル（`_000`, `_001`, `_002`）が存在し、これはサブファンドごとのファイル分割を示唆する。

## §2. 財務諸表を含むか

**結論**: jpsps タクソノミ自体に財務諸表の **個別勘定科目（monetaryItemType 等）は定義されていない**。ただし、財務諸表は **テキストブロック** として含まれる（BS、PL、CF、注記等のテキストブロック要素が存在）。

### 情報源 (Fact)

- **F5**: jpsps_cor XSD 要素型分析（全545行、534要素）— `G-8.jpsps_analysis.py` で機械的に確認:
  - `stringItemType`: 297要素（うち abstract=273: Heading/目次項目、非abstract=24: 表紙メタデータ）
  - `textBlockItemType`: 231要素
  - `dateItemType`: 4要素（提出日等）
  - `booleanItemType`: 2要素（訂正フラグ: `SecuritiesRegistrationStatementAmendmentFlagDeemedRegistrationStatementDEI`, `AnnualSecuritiesReportAmendmentFlagDeemedRegistrationStatementDEI`）
  - **monetaryItemType**: 0要素
  - **sharesItemType**: 0要素
  - **percentItemType**: 0要素
  - **Dimension関連 (hypercubeItem, dimensionItem)**: 0要素

- **F6**: 財務諸表テキストブロック要素の存在（L.410-429）:
  - `BalanceSheetTextBlock` — 貸借対照表
  - `StatementOfIncomeTextBlock` / `StatementOfIncomeAndRetainedEarningsTextBlock` — 損益計算書
  - `StatementOfUnitholdersEquityTextBlock` — 出資者持分変動計算書
  - `StatementOfCashFlowsTextBlock` — キャッシュ・フロー計算書
  - `StatementOfCashDistributionTextBlock` — 金銭の分配に係る計算書
  - `NotesToFinancialStatementsTextBlock` — 注記
  - 各半期版（SemiAnnual〜）も対応

- **F7**: フレームワーク設計書 L.356-371: 特定有価証券開示府令の各様式で jpsps タクソノミ列に ○。jppfs/jpigp 列には ○ なし。

### 推論 (Reasoning)

F5 により、jpsps タクソノミには数値型要素が一切定義されておらず、構造化された勘定科目データは含まれない。しかし F6 の通り、BS/PL/CF 等の財務諸表はテキストブロックとして包含される。F7 でも jppfs（日本基準）/ jpigp（IFRS）の使用が否定されている。

これは、特定有価証券（投資信託・REIT等）の財務諸表が一般事業会社と異なる独自の様式を使用しており、jppfs の勘定科目体系では表現しきれないためと推察される。財務データはテキストブロック内の HTML 表に埋め込まれるため、数値抽出には HTML パースが必要。

## §3. 基準価額 (NAV) の報告構造

**結論**: 基準価額（NAV）は jpsps タクソノミ内に **専用の数値型 Fact として定義されていない**。NAV 情報は `ChangesInNetAssetsTextBlock` 等のテキストブロック内に HTML 表として記載される。複数サブファンドの表現は、ZIP 内の複数 XBRL ファイルによるファイル分割方式。Dimension は使用しない。

### 情報源 (Fact)

- **F8**: jpsps_cor XSD: `NAV` / `NetAssetValue` / `BasePrice` を含む要素名は存在しない（全要素名を確認）。
- **F9**: 関連テキストブロック要素:
  - `ChangesInNetAssetsTextBlock` — 純資産の変動
  - `ChangesInNetAssetsEtcTextBlock` — 純資産の推移等
  - `CalculationOfNetAssetsTextBlock` — 純資産の計算
  - `SummaryOfBusinessResultsTextBlock` — 運用実績の概要
  - `BalanceSheetFinancialHighlightsTextBlock` — 貸借対照表の財務ハイライト
- **F10**: サンプルインスタンス 10_特定有価証券: 同一発行体の3ファイル(`_000`, `_001`, `_002`)が存在。これはファンド/サブファンドごとのファイル分割。

### 推論 (Reasoning)

F8 により NAV の数値型要素は存在せず、F9 のテキストブロック内でナラティブ/HTML表として報告される。F10 の複数ファイルはサブファンドごとのファイル分割を示しており、Dimension ではなくファイルレベルでの分離が行われている。パーサーの観点では、NAV を構造化データとして取得するには HTML パースが必要であり、v0.1.0 のスコープ外と考えるべき。

## §4. 信託報酬の構造

**結論**: 信託報酬率・金額も jpsps タクソノミ内に **数値型 Fact として定義されていない**。`ManagementFeesAndChargesTextBlock` 等のテキストブロックとして記載される。

### 情報源 (Fact)

- **F11**: jpsps_cor XSD の信託報酬関連テキストブロック:
  - `ManagementFeesAndChargesTextBlock` — 信託報酬（管理費用）
  - `AdministrationFeesAndChargesTextBlock` — 事務費用
  - `OtherFeesAndChargesTextBlock` — その他費用
  - `RedemptionFeeTextBlock` — 解約手数料
  - `ApplicationFeeOverviewOfFundTextBlock` — 申込手数料
- **F12**: これらはすべて `textBlockItemType` であり、数値型（monetaryItemType, percentItemType）ではない。

### 推論 (Reasoning)

F11-F12 により、信託報酬関連の情報はすべてテキストブロック型で定義されている。信託報酬率（例: 年率 0.5%）や具体的な金額は HTML コンテンツ内のテキストとして記載されるが、構造化された数値 Fact としては提供されない。これは jpsps 全体の設計方針（テキストブロック中心）と一貫している。
