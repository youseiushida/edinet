## H-2. 名前空間プレフィックスの安定性

### 質問への対応

| サブ質問 | 回答 |
|----------|------|
| H-2.1 プレフィックス名は仕様で固定か | EDINET 仕様書で推奨プレフィックスが規定されている（図表 5-2-1）。ただし XML 仕様上プレフィックスは自由にリバインド可能であり、`jppfs_cor` を `foo` にしても XML として valid。しかし EDINET の提出ルールとしてはガイドライン記載のプレフィックスを使うことが求められている |
| H-2.2 実装上の注意 | プレフィックス名ではなく名前空間 URI で要素を識別すべき。lxml の `{namespace_uri}local_name` 形式（Clark notation）で処理すれば、プレフィックス名の揺れに影響されない |

### 情報源（Fact）

- [F1] パス: `docs/仕様書/XBRL Instance Guideline 報告書インスタンス作成ガイドライン.md` L.841 — 「報告書インスタンスでは、タグ付けに利用している語彙スキーマ、インライン XBRL で利用する名前空間プレフィックスと名前空間 URI を「<html>タグ」に設定します。設定する名前空間プレフィックス及び名前空間 URI は、次の図表を参照してください。」
- [F2] パス: 同上 L.843-877 — 図表 5-2-1 名前空間プレフィックスおよび名前空間 URI の一覧（全17行）。プレフィックス名と URI の対が明示的に規定されている:
  - `ix` → `http://www.xbrl.org/2008/inlineXBRL`
  - `xbrli` → `http://www.xbrl.org/2003/instance`
  - `jppfs_cor` → `http://disclosure.edinet-fsa.go.jp/taxonomy/jppfs/{タクソノミ日付}/jppfs_cor`
  - `jpigp_cor` → `http://disclosure.edinet-fsa.go.jp/taxonomy/jpigp/[タクソノミ日付]/jpigp_cor`
  - `jpdei_cor` → `http://disclosure.edinet-fsa.go.jp/taxonomy/jpdei/[タクソノミ日付]/jpdei_cor`
  - 他 14 件（iso4217, xlink, link, xsi, num, nonnum, xbrldt, xbrldi, dtr-types, ixt 等）
- [F3] サンプルファイル: `docs/仕様書/2026/サンプルインスタンス/.../02_開示府令-有価証券報告書/S002XXXX/XBRL/PublicDoc/jpcrp030000-asr-001_X99001-000_2026-03-31_01_2026-06-12.xbrl` L.2 — ルート要素の名前空間宣言において、仕様書通りのプレフィックス名が使われている（`xmlns:jppfs_cor=...`, `xmlns:jpcrp_cor=...`, `xmlns:jpdei_cor=...`, `xmlns:xbrli=...` 等）
- [F4] P-2.a.md [F6] — トヨタ実データ (S100VWVY) の名前空間宣言でも、仕様書と同一のプレフィックス名が使用されている（`xmlns:jppfs_cor=...`, `xmlns:jpcrp_cor=...`, `xmlns:jpdei_cor=...`）
- [F5] サンプルファイル: IFRS サンプル (`03_開示府令-IFRS有価証券報告書/S003XXXX/XBRL/PublicDoc/...xbrl`) L.2 — IFRS でも同一のプレフィックス名が使用されている（`xmlns:jpigp_cor=...`, `xmlns:jppfs_cor=...` 等）

### 推論（Reasoning）

1. [F1]+[F2] より、EDINET 仕様書は図表 5-2-1 で名前空間プレフィックスと URI の対を明示的に規定している。これは「推奨」ではなく「設定する名前空間プレフィックス」として記述されており、提出者はこのプレフィックス名を使うことが期待されている。

2. ただし XML 名前空間仕様 (Namespaces in XML 1.0) 上、プレフィックスは名前空間 URI へのローカルなバインディングに過ぎず、任意に選べる。したがって `jppfs_cor` を `foo` にリバインドしても XML パーサーレベルでは valid である。ただし EDINET の提出ルール上は仕様書記載のプレフィックスを使うことが求められる。

3. [F3]+[F4]+[F5] より、サンプルインスタンス（J-GAAP / IFRS）および実提出データ（トヨタ）の全てで仕様書記載のプレフィックス名が一貫して使われている。実務上プレフィックスの揺れは発生していない。

4. **実装上の結論**: パーサーでは名前空間 URI で要素を識別すべき。lxml の Clark notation `{http://disclosure.edinet-fsa.go.jp/taxonomy/jppfs/2025-11-01/jppfs_cor}CashAndDeposits` のようにプレフィックスに依存しない方法で処理する。ただし Context ID 内に含まれるプレフィックス文字列（例: `CurrentYearDuration_jpcrp030000-asr_X99001-000CommunicationsEquipmentReportableSegmentMember`）はプレフィックスではなく Context ID 命名規則の一部であり、XML 名前空間とは無関係。
