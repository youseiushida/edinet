## §1. 一意に特定できるか

**結論**: **概ね特定可能**だが、完全に一意とは限らない。`ordinance_code` でまず府令（= タクソノミ大分類）が決まり、`form_code` で様式番号が決まる。この組み合わせから使用されるタクソノミモジュール群を高い精度で推定できる。ただし、会計基準の違い（jppfs vs jpigp）は DEI の `AccountingStandardsDEI` を読むまで確定しない。

### 情報源 (Fact)

- **F1**: form_code.py: `ordinance_code` と `form_code` の全組み合わせが `_FORM_CODE_TABLE` に定義されている。各エントリに `ordinance_code`, `form_code`, `form_number`（様式番号）, `form_name`, `doc_type_name` が含まれる。
- **F2**: 仕様書の図表 2-3-1〜2-3-7: `ordinance_code` → 府令名 → タクソノミ分割単位の対応が明確。
- **F3**: D-3.a.md: 会計基準（jppfs/jpigp）は XBRL 内の DEI 要素で判別。form_code からは不明。

### 推論 (Reasoning)

F1 の `ordinance_code` は府令コードであり、これが府令を一意に特定する。F2 の仕様書マッピングにより、府令→タクソノミ分割単位の対応が得られる。F3 により、財務諸表本表タクソノミ（jppfs vs jpigp）のみ事前特定不可だが、これは「両方ロードする」対応で回避可能。

## §3. パーサーの事前準備に使えるか

**結論**: はい、非常に有用。`ordinance_code` だけで書類タクソノミの大分類を特定でき、タクソノミスキーマのプリロードや名前空間フィルタリングの事前設定に活用可能。

### 情報源 (Fact)

- **F4**: form_code.py の `get_form_code()` 関数: `ordinance_code` + `form_code` → `FormCodeEntry` を返す。`form_name`, `doc_type_name` でさらに書類種別を特定可能。
- **F5**: G-1〜G-8b の各回答: 各タクソノミの要素型分布が明確になっており、事前に処理戦略を決定可能:
  - `jpcrp` + `jppfs`/`jpigp`: 数値型要素あり → 数値パース必要
  - `jpcrp-esr`, `jpctl`, `jpsps`, `jptoi`, `jptoo-*`: テキスト中心 → テキストパースのみ
  - `jplvh`: 数値型+Dimension → Dimension 対応パーサー必要

### 推論 (Reasoning)

F4 の `get_form_code()` を使えば、XBRL をダウンロードする前に Filing メタデータから書類の性質を判定できる。F5 の調査結果を組み合わせることで、以下の事前準備が可能:

1. **タクソノミプリロード**: `ordinance_code` から必要なタクソノミ XSD をプリロード
2. **パース戦略の選択**: テキスト専用 vs 数値含有 vs Dimension 対応の判定
3. **スキップ判定**: v0.1.0 で未対応の書類タイプ（例: jplvh の Dimension 構造）を事前にスキップ

実装例:
```python
def predict_taxonomy(ordinance_code: str, doc_type_code: str) -> list[str]:
    """Filing メタデータからタクソノミモジュールを予測する。"""
    if ordinance_code == "010":
        if doc_type_code in ("180",):
            return ["jpcrp-esr", "jpdei"]
        if doc_type_code in ("190",):
            return ["jpcrp-sbr", "jpdei"]
        return ["jpcrp", "jppfs_or_jpigp", "jpdei"]
    if ordinance_code == "015":
        return ["jpctl", "jpdei"]
    if ordinance_code == "030":
        if doc_type_code in ("185",):  # 特定有価証券の臨時報告書
            return ["jpsps-esr", "jpdei"]
        return ["jpsps", "jpdei"]
    if ordinance_code == "040":
        return ["jptoo-*", "jpdei"]  # form_code で詳細特定
    if ordinance_code == "050":
        return ["jptoi", "jpdei"]
    if ordinance_code == "060":
        return ["jplvh", "jpdei"]
    return ["unknown"]
```

## §5. 追加調査結果（mapping_table.py 実行後の検証）

G-9.mapping_table.py の実行で発見された3つの問題について追加調査を行った。

### 調査1: ordinance_code=015 の正体

**結論**: ordinance_code=015 は **内部統制府令**（財務計算に関する書類その他の情報の適正性を確保するための体制に関する内閣府令）であり、正規のコードである。

#### 根拠

- **EDINET API仕様書** 3章 府令コード一覧（2325行付近）に `015` が明記されている
- **ordinance_code.py**: `OrdinanceCode.INTERNAL_CONTROL = "015"` として定義済み
- **form_code.py**: 6エントリ（第一号〜第三号様式 × 通常/訂正）が存在
- **タクソノミ概要説明書** 図表 2-3-6: 内部統制府令のタクソノミは `jpctl`（第一号様式のみ XBRL 対象）
- **データ元**: ESE140327.xlsx（様式コードリスト）の「府令コード」列から直接取得

#### mapping_table.py で unknown になった理由

初版の mapping_table.py の `ORDINANCE_TAXONOMY_MAP` に 015 のエントリが欠落していたため。§2 の対応表を修正し、015 を追加した。

#### 015 と 060 の関係

初版の G-9.a.md では `060=内部統制府令` としていたが、これは誤り。正しくは:
- `015` = 内部統制府令 → `jpctl`
- `060` = 大量保有府令 → `jplvh`

### 調査2: ordinance_code=030 の「jptoo-?」210件

**結論**: 初版の対応表の誤りに起因する問題であり、実際には解決済み。

#### 根拠

初版では `030=他社株買付府令` としていたため、030 の 234 エントリを jptoo-* にマッピングしようとして 210 件が未解決になっていた。正しくは:

- `030` = **特定有価証券府令** → `jpsps`（234エントリ全てが特定有価証券関連の書類）
- `040` = **他社株買付府令** → `jptoo-*`（11エントリのみ）

040 の 11 エントリはすべて form_number から jptoo-* サブモジュールに正しくマッピング可能（§2 の詳細マッピング表参照）。第三号様式（別途買付け禁止の特例を受けるための申出書）2件のみ XBRL 対象外。

#### 030 の XBRL 対象外エントリ

030（特定有価証券府令）の 234 エントリのうち、以下は XBRL 対象外:
- 有価証券通知書 / 変更通知書 — 非開示書類
- 届出の取下げ願い — 非開示書類
- 発行登録通知書 / 変更通知書 — 非開示書類
- 外国特定有価証券の様式 — 図表 2-1-3 注記「外国特定有価証券は、XBRL 対象外」

### 調査3: ordinance_code=050/060 の doc_type_name 不審点

**結論**: form_code.py の doc_type_name 自体は正しい。問題は初版の対応表の誤り。

#### 根拠

form_code.py のデータ（ESE140327.xlsx 由来）:
- `050` のエントリ: doc_type_name は「公開買付届出書」「公開買付撤回届出書」「公開買付報告書」→ 自社株買付府令の書類として正しい
- `060` のエントリ: doc_type_name は「大量保有報告書」「変更報告書」「基準日の届出書」→ 大量保有府令の書類として正しい

初版の mapping_table.py では `050=大量保有`, `060=内部統制` としていたため、出力上の doc_type_name と府令名が不整合に見えた。対応表の修正により解消。
