## J-2. セグメント情報

### 質問への対応

| サブ質問 | 回答 |
|----------|------|
| J-2.1 事業セグメントのデータは XBRL で構造化されているか | **Yes。報告セグメントごとの金額データ（売上高、利益又は損失、資産、負債等）は XBRL で詳細タグ付け（構造化）されている。** ただし、セグメント情報の「関連情報」や「差異調整の脚注」はテキストブロック（`textBlockItemType`）であり、構造化されていない。概要説明 L.2197-2230 に「セグメント情報は詳細タグ付け」と明記されている |
| J-2.2 セグメント情報の dimension 名 | **`jpcrp_cor:OperatingSegmentsAxis`**。この Axis はセグメント情報のための唯一のディメンション軸であり、Context の scenario 内に `xbrldi:explicitMember dimension="jpcrp_cor:OperatingSegmentsAxis"` として設定される |
| J-2.3 将来的なセグメント情報抽出のアプローチ | Context 内の `OperatingSegmentsAxis` の有無でフィルタリングする。セグメント別の概念名（勘定科目）は財務諸表本表と**同一の要素**（例: `jppfs_cor:NetSales`, `jppfs_cor:Assets`）が使われ、セグメントの区別は Context の scenario 内のメンバー要素のみで行われる。詳細は推論セクション参照 |

### OperatingSegmentsAxis の標準メンバー

jpcrp_cor_2025-11-01.xsd に定義されているセグメント関連の標準メンバー:

| メンバー要素名 | 意味 |
|----------------|------|
| `ReportableSegmentsMember` | 報告セグメント（集約） |
| `TotalOfReportableSegmentsAndOthersMember` | 報告セグメント及びその他の合計 |
| `AllReportableSegmentsAndOthersMember` | 全報告セグメント及びその他 |
| `ReconciliationMember` | 調整額 |
| `UnallocatedAmoundsAndEliminationMember` | 配分不能額及び消去 |

上記は標準タクソノミに定義された汎用メンバーである。実際の個別セグメント名（「自動車事業」「金融事業」等）は、提出者別タクソノミで拡張要素として定義される（例: `AutomotiveReportableSegmentMember`, `FinancialServicesReportableSegmentMember`）。

### セグメント Fact の Context 構造

セグメント情報を持つ Fact の Context は以下の構造をとる:

```xml
<xbrli:context id="CurrentYearDuration_jpcrp030000-asr_E02144-000AutomotiveReportableSegmentMember">
  <xbrli:entity>
    <xbrli:identifier scheme="http://disclosure.edinet-fsa.go.jp">E02144-000</xbrli:identifier>
  </xbrli:entity>
  <xbrli:period>
    <xbrli:startDate>2024-04-01</xbrli:startDate>
    <xbrli:endDate>2025-03-31</xbrli:endDate>
  </xbrli:period>
  <xbrli:scenario>
    <xbrldi:explicitMember dimension="jpcrp_cor:OperatingSegmentsAxis">
      jpcrp030000-asr_E02144-000:AutomotiveReportableSegmentMember
    </xbrldi:explicitMember>
  </xbrli:scenario>
</xbrli:context>
```

重要な点:
- dimension 属性の値は `jpcrp_cor:OperatingSegmentsAxis` で固定
- メンバー値は提出者別タクソノミの名前空間（例: `jpcrp030000-asr_E02144-000:`）で修飾される
- Context ID にもメンバー名が含まれる（命名規則 A-2.a.md 参照）

### セグメント Fact で使用される概念名

セグメント別のデータに使われる概念名は、財務諸表本表で使われるものと**同一の要素**である（概要説明 L.2219-2225「報告セグメントごとの売上高、利益又は損失、資産、負債その他の項目の金額に関する情報の表示項目」は「財務諸表本表中の調整対象の勘定科目と同一の要素を使う」）。

代表的な概念名:
- `jppfs_cor:NetSales` — 売上高
- `jppfs_cor:OperatingIncome` — 営業利益
- `jppfs_cor:Assets` — 資産
- `jppfs_cor:DepreciationAndAmortizationOpeCF` — 減価償却費
- `jppfs_cor:IncreaseInPropertyPlantAndEquipmentAndIntangibleAssetsOtherThanGoodwill` — 設備投資額

これらの概念は、Context にセグメント dimension が設定されていなければ全社合計値、設定されていれば当該セグメントの値を表す。

### v0.1.0 での方針と将来の抽出アプローチ

**v0.1.0**: セグメント情報を無視する。具体的には、Context の scenario に `OperatingSegmentsAxis`（またはその他のディメンション軸）が設定されている Fact を除外し、dimension なしの Fact のみを採用する。

**将来（v0.2.0+）での抽出アプローチ**:

1. **Context 解析**: 全 Context を走査し、scenario 内の `xbrldi:explicitMember` で `dimension="jpcrp_cor:OperatingSegmentsAxis"` を検出
2. **メンバー名の取得**: `explicitMember` のテキスト値からメンバー QName を取得（例: `jpcrp030000-asr_E02144-000:AutomotiveReportableSegmentMember`）
3. **ラベル解決**: メンバー QName の日本語ラベルを提出者別タクソノミの `_lab.xml` から取得（例: `AutomotiveReportableSegmentMember` -> `自動車事業`）
4. **Fact のグループ化**: 同一概念名の Fact を、メンバー名でグループ化してセグメント別テーブルを構築
5. **標準メンバーの識別**: `ReportableSegmentsMember`（合計）、`ReconciliationMember`（調整額）、`UnallocatedAmoundsAndEliminationMember`（配分不能額）は標準タクソノミのメンバーであり、名前空間 `jpcrp_cor:` で判別可能

### 推論（Reasoning）

1. **セグメント情報の構造化について**: [F1] より、EDINET のセグメント情報は「詳細タグ付け」されている。これは報告セグメントごとの金額データ（売上高、利益/損失、資産、負債等）が個別の XBRL Fact としてタグ付けされることを意味する。ただし、セグメント情報のうち「関連情報」（製品・サービスごとの情報、地域ごとの情報、主要な顧客に関する情報）および「差異調整の脚注」はテキストブロックとして格納される。つまり、定量データは構造化されているが、定性的な説明・脚注は非構造化である。

2. **Dimension 軸の特定**: [F2]+[F6] より、セグメント情報は `jpcrp_cor:OperatingSegmentsAxis` という単一の Explicit Dimension 軸で表現される。EDINET には他にセグメント用の軸は存在しない（事業セグメント・地域セグメント等の区別は、軸の違いではなくメンバーの命名で表現される可能性があるが、仕様書上は `OperatingSegmentsAxis` のみが記載されている）。

3. **概念名の共有**: [F1] より、セグメント別データに使われる概念名は財務諸表本表と同一の要素（例: `jppfs_cor:NetSales`）である。セグメントの区別は概念名ではなく、Context の scenario 内のメンバー要素のみで行われる。これにより、同一の概念名で「全社合計」と「セグメント別」の両方の値が存在しうるが、Context が異なるため一意に識別可能。

4. **メンバーの2層構造**: [F3]+[F4]+[F6] より、OperatingSegmentsAxis のメンバーは2層構造をとる。(a) 標準タクソノミ（jpcrp_cor）で定義される汎用メンバー（`ReportableSegmentsMember` 等の集計・調整用メンバー）と、(b) 提出者別タクソノミで定義される企業固有メンバー（`AutomotiveReportableSegmentMember` 等の個別セグメント名）。パーサーは両方を処理する必要があり、個別セグメント名の日本語ラベルは提出者別 `_lab.xml` から取得する。

5. **v0.1.0 での除外方法**: Context の scenario に何らかの `xbrldi:explicitMember` が設定されているかどうかを判定することで、セグメント Fact を除外できる。ただし、`ConsolidatedOrNonConsolidatedAxis` のデフォルト（連結）は scenario が省略されるため、「scenario あり = セグメント Fact」とは限らない。正確には、`OperatingSegmentsAxis` を含む Context を持つ Fact を除外する必要がある。なお、v0.1.0 では dimension なしの Fact のみを採用する方針のため、連結/個別軸以外のディメンション（セグメント軸を含む）を持つ Fact は全て除外される。

6. **将来の抽出の実現可能性**: [F1]+[F2]+[F4] より、セグメント情報の抽出は技術的に十分可能である。Context の dimension 情報は XML 構造として明確に定義されており、メンバー名も提出者別タクソノミから取得可能。主な課題は (a) 提出者ごとに異なるメンバー名の正規化、(b) 日本語ラベルの取得（`_lab.xml` のパース）、(c) 標準メンバー（合計・調整額）と個別メンバーの区別、の3点である。
