## I-3. 提出者による Presentation のカスタマイズ

### 質問への対応

| サブ質問 | 回答 |
|----------|------|
| I-3.1 標準のプレゼンテーション順序の変更は可能か | **可能。** 提出者は「再構成（リキャスト）」方式により、EDINET タクソノミの表示リンクを参考にしてゼロから独自の `_pre.xml` を作成する。科目の追加・削除・並び替えは自由に行える。 |
| I-3.2 「上書き」か「拡張」か | **どちらでもない。** 提出者の `_pre.xml` は EDINET タクソノミの `_pre.xml` を「上書き」も「拡張」もしない。提出者は EDINET タクソノミとは異なるロール URI（`_std_` なし）を使用して完全に新規のファイルを作成する。標準を「参照」「修正」することも禁止されている。 |
| I-3.3 実装上の組み合わせ方法 | 提出者の `_pre.xml` **のみをパースすればよい**。EDINET タクソノミの `_pre.xml` とのマージは不要。提出者の `_pre.xml` が当該提出書類の表示構造を完全に定義しているため、これだけで財務諸表の科目構成・並び順・インデント・合計行が全て取得できる。 |

### 推論（Reasoning）

1. **I-3.1 順序変更の可否**: [F1][F3][F6] より、表示リンクは「再構成（リキャスト）」方式で作成される。リキャストとは [F6] の定義の通り「EDINET タクソノミから必要に応じてコピーして、追加、削除及び変更し」た新規ファイルを作成する方法である。したがって提出者は標準の順序に縛られず、自社の報告内容に合わせて自由にカスタマイズできる。order 属性の値を変更すれば並び順が変わり、要素を追加・削除すればその科目が財務諸表に含まれるか否かが変わる。

2. **I-3.2 上書きか拡張か**: [F3] は「そのまま使用又は編集するのではなく」と明言し、[F4] は「直接修正しないでください」と禁止している。さらに [F5] + [F7] により、提出者は `_std_` なしの提出者用ロール URI を使い、EDINET タクソノミは `_std_` 付きのロール URI を使うため、両者のロール URI が異なる。同じロール URI を共有しないので「上書き」の概念自体が成立しない。また XBRL の arc override メカニズム（`use="prohibited"`, `priority`）は表示リンクでは使用不可（C-6.a.md 推論6）であるため、「拡張」の概念も適用されない。正確には**「新規作成（置換）」**である。

3. **I-3.3 実装上の組み合わせ**: [F2] + [F7] より、提出者の `_pre.xml` は提出者用ロール URI を使って表示構造を完全に定義する独立したファイルである。EDINET タクソノミの `_pre.xml` は EDINET タクソノミ用ロール URI（`_std_*`）を使用しており、提出者の `_pre.xml` とロール URI が異なるためマージしても衝突はしないが、マージする必要もない。パーサーは提出者の `_pre.xml` のみをパースし、その中の提出者用ロール（`rol_ConsolidatedBalanceSheet` 等）からツリーを構築すればよい。

### 確信度

- I-3.1 順序変更の可否: **高** -- 複数の仕様書が一貫して「リキャスト」を明記。C-6 の調査でも確認済。
- I-3.2 上書きか拡張か: **高** -- 仕様書が「直接修正しない」と禁止し、ロール URI の分離も実ファイルで確認済。
- I-3.3 実装上の組み合わせ: **高** -- 実サンプルの roleRef 確認により、提出者用ロールのみが使用されていることを検証済。
