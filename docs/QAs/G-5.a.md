## §1. jplvh タクソノミの概要

**結論**: `jplvh` は大量保有報告書（第一号～第三号様式）専用のタクソノミ分割単位で、170行・157要素を定義する中規模タクソノミ。保有者情報・保有株式数・保有割合・取得資金の内訳等を構造化する。Dimension 構造を持ち、複数の報告者・共同保有者を表現可能。

### 情報源 (Fact)

- **F1**: 仕様書「EDINETタクソノミの概要説明」L.933-943: 図表 2-3-5: 大量保有府令のタクソノミ分割単位 → `jplvh`。3様式（第一号、第一号及び第二号、第三号）すべてを包含。
- **F2**: jplvh_cor_2025-11-01.xsd: 全170行。Dimension関連要素（hypercubeItem, dimensionItem, domainItemType）を含む。
- **F3**: 提出者別タクソノミ作成ガイドライン L.101-103: 大量保有報告書（第一号～第三号様式）がXBRL作成ツール対象。
- **F4**: XSD バージョン日付: 2025-11-01（最新更新）。

### 推論 (Reasoning)

F1 により、大量保有報告書の3つの様式すべてが単一の `jplvh` タクソノミで管理される。F2 でDimension構造（hypercubeItem, dimensionItem）が定義されており、これは保有者ごとの情報を構造的に表現するためのものである（§3で詳述）。F3 より XBRL 作成ツール対象であり、提出者は本書の詳細理解なしに準拠した XBRL を作成できる。

## §2. 保有株式数・保有割合の構造化

**結論**: はい、保有株式数と保有割合は XBRL で構造化された数値データとして含まれる。

### 情報源 (Fact)

- **F5**: jplvh_cor XSD の要素型分類（全157要素）— `G-5.verify_jplvh.py` で機械的に確認:

| 型 | 要素数 | 具体例 |
|----|--------|--------|
| `stringItemType` | 83 (abstract=52, concrete=31) | Heading系、表紙情報、保有者属性等 |
| `sharesItemType` | 50 (全て concrete) | 株式数、新株予約権、転換社債等の各種保有数量 |
| `textBlockItemType` | 9 | 注記テキストブロック、取得処分明細、担保契約等 |
| `dateItemType` | 5 | `FilingDateCoverPage`, `DateWhenFilingRequirementAroseCoverPage`, `DateOfBirth`, `DateOfEstablishment`, `BaseDate` |
| `monetaryItemType` | 4 | `AmountOfOwnFund`, `TotalAmountOfBorrowings`, `TotalAmountFromOtherSources`, `TotalAmountOfFundingForAcquisition` |
| `domainItemType` | 3 (abstract) | `FilersLargeVolumeHoldersOrJointHoldersMember`, `FilersLargeVolumeHoldersMember`, `JointHoldersMember` |
| `percentItemType` | 2 | `HoldingRatioOfShareCertificatesEtc`, `HoldingRatioOfShareCertificatesEtcPerLastReport` |
| `nonNegativeIntegerItemType` | 1 | `TotalNumberOfFilersAndJointHoldersCoverPage` |

- **F6**: サンプルインスタンス 18_大量保有報告書: `HoldingRatioOfShareCertificatesEtc` が `decimals="4"` で報告。`AmountOfOwnFund` が `decimals="-3"` で報告。

### 推論 (Reasoning)

F5（`G-5.verify_jplvh.py` による機械的カウント）により、jplvh は他の書類種別タクソノミ（jpcrp-esr, jpctl）とは異なり、豊富な数値型要素を持つ。特に `sharesItemType` が50要素と最多であり、保有株式数の内訳（条文27の23第3項の各号に対応する分類）を詳細に構造化している。`monetaryItemType` で取得資金の内訳、`percentItemType` で保有割合を表現する。

## §3. データ表現方式（Dimension / Tuple）

**結論**: 保有者リストは **Dimension（XBRL Dimensions 1.0）** で構造化されている。Tuple は使用しない。共同保有者も同じ Dimension 軸で表現される。

### 情報源 (Fact)

- **F7**: jplvh_cor XSD の Dimension 関連要素（`substitutionGroup` 属性で判定 — `G-5.verify_jplvh.py` で確認）:
  - **Hypercube (Table)**: 5個（`substitutionGroup="xbrldt:hypercubeItem"`、`type` は `stringItemType`）
    - `InformationAboutFilersLargeVolumeHoldersAndJointHoldersDEITable` — DEI用テーブル
    - `InformationAboutFilersTable` — 報告者情報テーブル
    - `InformationAboutJointHoldersTable` — 共同保有者情報テーブル
    - `FilersAndJointHoldersTable` — 報告者+共同保有者合算テーブル
    - `HoldingRatioOfShareCertificatesEtcHeldByJointHoldingTable` — 共同保有割合テーブル
  - **Axis (dimensionItem)**: 1個（`substitutionGroup="xbrldt:dimensionItem"`）
    - `FilersLargeVolumeHoldersAndJointHoldersAxis`
  - **Domain/Member (domainItemType)**: 3個
    - `FilersLargeVolumeHoldersOrJointHoldersMember` — ドメイン
    - `FilersLargeVolumeHoldersMember` — 大量保有者メンバー
    - `JointHoldersMember` — 共同保有者メンバー

- **F8**: サンプルインスタンス 18_大量保有報告書:
  - Context に `xbrldi:explicitMember dimension="jplvh_cor:FilersLargeVolumeHoldersAndJointHoldersAxis"` が使用
  - メンバー値: `FilerLargeVolumeHolder1Member`, `FilerLargeVolumeHolder2Member` 等（提出者別タクソノミで拡張）
  - 複数の報告者・共同保有者が同一 Axis 上の異なる Member として表現される
  - `xbrldi:explicitMember` は `xbrli:scenario` 内に配置されている（`entity/segment` ではない）

- **F9**: K-2.a.md: 「EDINET では `ix:tuple` の使用なし」と確認済み。

### 推論 (Reasoning)

F7 の XSD 定義で Dimension 構造（Table + Axis + Member）が明確に定義されており、F8 のサンプルインスタンスでも実際に `xbrldi:explicitMember` による Dimension 表現が確認できる。F9 により Tuple は使用されない。

パーサーの観点では、大量保有報告書の数値 Fact を正しく解釈するには、Context の `xbrli:scenario` 内の `xbrldi:explicitMember` を読み取り、どの保有者に紐づくデータかを識別する必要がある（注: XBRL Dimensions 仕様では `entity/segment` と `scenario` のどちらも使用可能だが、本サンプルでは `scenario` が使用されている）。単一の `FilersLargeVolumeHoldersAndJointHoldersAxis` 軸で報告者と共同保有者の両方を管理しており、メンバー名で区別する構造になっている。

## §4. 保有割合の decimals / unit

**結論**: 保有割合は `num:percentItemType` で定義され、`unitRef` は `xbrli:pure`。`decimals="4"` が使用される。値は**比率（小数）表現**であり、例えば保有割合 4.26% は `0.0426` として報告される。

### 情報源 (Fact)

- **F10**: jplvh_cor XSD L.123: `HoldingRatioOfShareCertificatesEtc` の型は `num:percentItemType`。
- **F11**: サンプルインスタンス 18_大量保有報告書: `HoldingRatioOfShareCertificatesEtc` が `decimals="4"` で報告。unitRef は `pure`（xbrli:pure）。
- **F12**: XBRL の `percentItemType` は `xbrli:pure` ユニットを使用する仕様。サンプルインスタンスでの実値確認（`G-5.jplvh_sample.py` 実行結果）:
  - Holder1: 株式数 980,000 / 発行済 23,000,000 = 0.04260... → XBRL値 `0.0426`（**比率表現**）
  - Holder2: 株式数 200,000 / 発行済 23,000,000 = 0.00869... → XBRL値 `0.0087`（**比率表現**）
  - 合算: 株式数 1,180,000 / 発行済 23,000,000 = 0.05130... → XBRL値 `0.0513`（**比率表現**）
  - → 値は**パーセント値ではなく比率（小数）** で記載されている。`decimals="4"` は小数点以下4桁 = 0.01% 精度を意味する。
- **F13**: 金額系要素（`AmountOfOwnFund` 等）は `decimals="-3"`（千円単位）、unitRef は `JPY`。

### 推論 (Reasoning)

F10-F12 により、保有割合は `percentItemType` / `pure` / `decimals="4"` で表現される。F12 のサンプル実データ検算（`G-5.jplvh_sample.py` 出力）により、値は**比率（小数）**で記載されていることが確定した。パーサーでの解釈時、パーセント表示するには値に100を掛ける必要がある（例: `0.0426` → `4.26%`）。

金額系(F13)は `JPY` / `decimals="-3"` が典型的であり、千円未満を四捨五入した値が報告される。
