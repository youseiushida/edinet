### 推論（Reasoning）

1. [F1] + [F3] より、設定規約書 図表 1-4-6 に定義された 23 の関係層業種略号と、実際のタクソノミディレクトリ `r/` 配下の 23 ディレクトリは完全に一致する。これにより業種カテゴリコードの一覧は確定的である。

2. [F1] + [F2] より、語彙層（図表 1-4-5）は 21 業種だが、関係層（図表 1-4-6）は 23 業種である。差異は銀行・信託業の分割にある: 語彙層では `BNK`（銀行・信託業）1コードだが、関係層では `bk1`（銀行・信託業）と `bk2`（銀行・信託業(特定取引勘定設置銀行)）の 2 コードに分割されている。また、語彙層では保険業が `INS`（保険業）1コードだが、関係層では `in1`（生命保険業）と `in2`（損害保険業）の 2 コードに分割されている。ただし語彙層は一般商工業を `CTE` とし関係層は `cai` とする等、略号体系が異なる。q.md で問われているのは関係層の 23 コードであり、`r/` 配下のディレクトリ名がそのまま関係層コードである。

3. 企業の業種カテゴリの判別方法について: XBRL インスタンス文書の `schemaRef` が提出者別 XSD を参照し、その XSD の `xs:import` 文で EDINET タクソノミのモジュール（`jppfs_cor` 等）を参照する。業種固有のタクソノミは提出者別 XSD の import には直接現れないが、**提出者別のプレゼンテーション・計算・定義リンクベースが `r/{業種コード}/` 配下のリンクベースを参照することで業種が決定される**。つまり業種判別は schemaRef → 提出者別 XSD → 提出者別リンクベース → EDINET タクソノミ `r/{業種コード}/` の参照チェーンを辿ることで可能である。DEI 要素には業種を直接示すフィールドはない。

4. [F4] + [F5] + [F6] より、CSV の「提出者業種」フィールドとタクソノミ業種コードは**異なる分類体系**である。CSV は東京証券取引所の業種分類（33業種 + 法人種別等で計 39 カテゴリ）を使用しており、タクソノミは**財務諸表の様式の違いに基づく業種分類**（23種）を使用している。例えばCSVの「サービス業」「情報・通信業」「電気機器」「小売業」等はタクソノミでは全て `cai`（一般商工業）に集約される。一方、CSV の「電気・ガス業」はタクソノミでは `ele`（電気事業）と `gas`（ガス事業）に分割される。CSV の「陸運業」はタクソノミでは `rwy`（鉄道事業）と `cai` に分かれる（鉄道事業者のみ `rwy`、バス・トラック等は `cai`）。従って CSV 業種からタクソノミ業種コードへの一対一のマッピングは存在しない。

5. [F1] + [F4] + [F6] より、**cns は建設業（Construction）であり、一般事業会社コードではない**。CSV でも「建設業」は 199 件のみである。一般事業会社（製造業、サービス業、小売業、情報通信業 等の大多数）に対応するコードは **cai（一般商工業 = Commercial and Industrial）** である。CSV で「タクソノミコード不明」とされた業種（サービス業 819 件、情報・通信業 728 件、小売業 411 件、卸売業 378 件、電気機器 280 件 等）は、財務諸表の様式としては一般的な BS/PL/CF を使用するため、全て `cai` に該当する。つまり上場企業の大多数は `cai` を使用する。
