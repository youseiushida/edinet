## C-6. プレゼンテーションリンクベースの詳細構造

### 質問への対応

| サブ質問 | 回答 |
|----------|------|
| C-6.1 role URI | 各財務諸表は異なる role URI を持つ。基底URI `http://disclosure.edinet-fsa.go.jp/role/jppfs/` の後に `rol_{std_}{諸表名}` を付与。BS/PL/CF/SS それぞれ別の拡張リンクロール。さらに連結/個別、期別、CF直接法/間接法で分岐し、多数のロールが存在する。詳細は推論セクション参照。 |
| C-6.2 order属性 | `presentationArc` の `order` 属性（0以上の半角数値、小数可）で並び順が決まる。同一拡張リンクロール内の同一階層上で一意であり、必ず設定する。 |
| C-6.3 preferredLabel | `presentationArc` の `preferredLabel` 属性にラベルロールを設定して、表示名を標準ラベル以外（合計ラベル、期首・期末ラベル、業種別ラベル、財務諸表区分別ラベル、代替ラベル等）に切り替える。ラベル切替えが必要な場合のみ設定。 |
| C-6.4 抽象要素 | あり。`abstract="true"` のタイトル項目（名前末尾 `Abstract`）や目次項目（名前末尾 `Heading`）がプレゼンテーションツリーに見出しとして登場する。例: `AssetsAbstract`, `CurrentAssetsAbstract`, `NoncurrentAssetsAbstract` 等。Fact は持たない。加えて `Heading` と科目 `Abstract` の間に `Table` / `LineItems` / `Axis` / `Member` 等の dimension 関連中間ノードが存在する（詳細は推論セクション参照）。 |
| C-6.5 提出者別と標準の関係 | 表示リンクは「再構成（リキャスト）」方式。提出者は EDINET タクソノミを参考に独自の `_pre.xml` を新規作成する。標準の `_pre.xml` を「参照」や「上書き」するのではなく、内容をコピー・編集して独立したファイルを作る。**ただし、財務諸表本表タクソノミの科目一覧ツリーは例外的にリユーズ（定義リンクのみ）。** |
| C-6.6 override/prohibition | 表示リンクでは `use="prohibited"` は**指定できない**。`use` は常に `optional`。`priority` による arc 解決は表示リンクには適用されない。名称リンク（label link）では `prohibited` と `priority` が使用されるが、表示リンクは再構成方式のため競合自体が発生しない設計。 |

### 推論（Reasoning）

1. **C-6.1 role URI**: [F1] により、関係リンクの拡張リンクロールの roleURI は命名規約 `rol_{std|業種略号}{ルート要素名}(-{修飾語})` に従い、基底 URI は `http://disclosure.edinet-fsa.go.jp/role/jppfs/` である。[F15] の実タクソノミ確認から、各財務諸表は明確に異なる role URI を持つことが確認できる。BS は `rol_std_ConsolidatedBalanceSheet` / `rol_std_BalanceSheet`、PL は `rol_std_ConsolidatedStatementOfIncome` / `rol_std_StatementOfIncome`、CF は直接法 `rol_std_...-direct` と間接法 `rol_std_...-indirect` に分岐、SS は `rol_std_ConsolidatedStatementOfChangesInEquity` / `rol_std_StatementOfChangesInEquity`。さらに連結/個別、通期/半期/四半期/中間の組合せで多数のロールが定義される。[F14] によりフレームワーク設計書の番号体系で `3_____` は日本基準本表、`5_____` は IFRS 本表に対応する。[F19] により、提出者用の拡張リンクロールは `_std` を含まない命名で別途定義される（例: `rol_ConsolidatedBalanceSheet`）。提出者は `_std` なしの提出者用ロールを使用する。

2. **C-6.2 order属性**: [F3] および [F9] の両方が明確に記載している。`order` 属性は `presentationArc` に設定する 0 以上の半角数値（小数可）であり、同一拡張リンクロール内の同一階層（同一の親を持つ兄弟要素間）で一意でなければならない。必ず設定する。[F16] の実ファイルでも `order="1.0"`, `order="2.0"` 等の設定が確認できる。

3. **C-6.3 preferredLabel**: [F3], [F9], [F13] より、`preferredLabel` 属性は `presentationArc` に設定するラベルロールであり、標準ラベル以外のラベルに表示名を切り替える機能を持つ。[F16] の実ファイルでは、連結 BS の合計行に `preferredLabel="http://www.xbrl.org/2003/role/totalLabel"` が設定され、連結用ラベルとして `preferredLabel="http://disclosure.edinet-fsa.go.jp/jppfs/Consolidated/role/label"` が設定されている。[F17] により、業種別ラベルやセグメントラベル、代替ラベルを設定した場合も preferredLabel への設定が必要。具体的ユースケースとして、銀行業PL において「営業収益」と表示したい場合、銀行業用のラベルロール（例: `http://disclosure.edinet-fsa.go.jp/jppfs/bk1/role/label`）を preferredLabel に設定することで、同一要素に対する銀行業の名称リンクが参照される。また、株主資本等変動計算書（SS）では期首残高と期末残高の区別に `periodStartLabel` / `periodEndLabel` が使われる。

4. **C-6.4 抽象要素**: [F4], [F5], [F6] より、`abstract="true"` の要素には2種類ある。タイトル項目（名前末尾 `Abstract`）と目次項目（名前末尾 `Heading`）である。両者ともインスタンス値を持たない見出し要素である。[F16] の実 `_pre.xml` で `AssetsAbstract`（資産）、`CurrentAssetsAbstract`（流動資産）、`NoncurrentAssetsAbstract`（固定資産）、`PropertyPlantAndEquipmentAbstract`（有形固定資産）、`IntangibleAssetsAbstract`（無形固定資産）、`InvestmentsAndOtherAssetsAbstract`（投資その他の資産）等の抽象要素がプレゼンテーションツリーの見出しノードとして使用されていることが確認できる。これらは Fact を持たず、ツリー構造のグルーピングのみに使用される。**なお [F16] の実ファイルでは、ルート要素 `Heading` と科目の `Abstract` 要素の間に dimension 関連の中間ノードが存在する。** 具体的には `ConsolidatedBalanceSheetHeading` > `BalanceSheetTable`（ディメンション軸の分岐: `ConsolidatedOrNonConsolidatedAxis` > `ConsolidatedMember`）および `BalanceSheetLineItems`（科目の分岐: `AssetsAbstract` > ...）という構造になっている。パーサーのツリー構築ロジックでは、`Table` / `LineItems` / `Axis` / `Member` といった dimension 関連ノードの存在を考慮する必要がある。

5. **C-6.5 提出者別と標準の関係**: [F7], [F8], [F9], [F12] が一貫して示すとおり、表示リンクは原則「再構成（リキャスト）」方式である。提出者は EDINET タクソノミの表示リンクを「そのまま使用又は編集するのではなく」、内容を参考にしてゼロから新規に作成する。標準の `_pre.xml` を直接参照することはなく（[F12]「EDINET タクソノミの表示リンクベースファイルを直接修正しないでください」）、提出者の `_pre.xml` が完全に独立したファイルとして存在する。したがって、「上書き」や「追加」ではなく「置換（新規作成）」である。提出者は自社の報告内容に合わせて並び順や科目をカスタマイズした独自の表示リンクを作成する。なお [F19] により、提出者用の拡張リンクロール（`_std` なし）は EDINET タクソノミ用の拡張リンクロール（`_std` 付き）とは別に定義されており、同一の role URI を共有しない。**ただし [F8] の但し書きにより、財務諸表本表タクソノミの「科目一覧ツリー」は例外的に再利用（リユーズ）される。** 科目一覧ツリーは定義リンクのみで定義される（表示リンクではない）ため表示リンクのパーサーには直接影響しないが、提出者が拡張要素を追加した際の科目一覧における位置付けの処理で、リユーズされた科目一覧ツリーを参照する必要がある。

6. **C-6.6 override/prohibition メカニズム**: [F9] で明確に「`use` 属性に `prohibited` は指定できません」と記載されており、表示リンクにおいては `use="prohibited"` は禁止されている。[F20] により定義リンクと計算リンクでも同様に prohibited は使用不可。[F10], [F11] によれば、`priority` 属性と `prohibited` が使われるのは名称リンク（label link）においてのみであり、これはラベルの上書き（リユーズ方式）のために必要とされる。表示リンクはリキャスト方式であるため、EDINET タクソノミと提出者別タクソノミが同一の role URI・同一の from/to を持つ arc を共有することがそもそもなく、arc の優先度解決（priority による解決ルール）が問題になる場面は設計上排除されている。
