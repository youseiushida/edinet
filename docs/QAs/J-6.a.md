### 推論（Reasoning）

#### 1. concept 名（local_name）の年度間安定性（J-6.1）

[F3] + [F9] + [F10] より、concept の `local_name` は年度間で高い安定性を持つ。

- 標準タクソノミの重複率は **96.4%**（前年ベース）
- 差分の大部分は**タクソノミ改訂ではなく、企業の開示内容の変更**による。例: トヨタの USGAAP 経営指標の記載方法変更（前年のみ18件中多数が `*USGAAP*` 関連）
- タクソノミ改訂による純粋な concept 追加/削除は年間20-30件程度（[F10]）
- 主要な財務概念（`NetSales`, `OperatingIncome`, `Assets`, `NetIncome` 等）は何年にもわたり安定しており、安全に join 可能

**ただし名前空間 URI はバージョンごとに異なる**（[F9]）。そのため年度横断のデータ結合では:
- `{namespace_uri}:{local_name}` をキーにすると **結合不可**
- `local_name` のみ、または `{module_prefix}:{local_name}`（例: `jppfs_cor:NetSales`）をキーにすべき
- deprecated 要素の改名（例: `MinorityInterests` → `NonControllingInterests`）は手動マッピングが必要

#### 2. Context ID パターンの安定性（J-6.2）

[F5] + [F7] より、Context ID の命名パターンは年度間で完全に安定している。

- `CurrentYearDuration`, `CurrentYearInstant`, `Prior1YearDuration`, `Prior1YearInstant` は毎年必ず存在
- `CurrentYearDuration` は**その filing の当年度期間**を意味する相対参照であり、異なる filing では異なる実日付を指す

**年度横断でのデータ結合方法**:
1. Context ID（`CurrentYearDuration` 等）は filing 内の相対参照 → **横断結合には使えない**
2. Context の `period` 要素に設定された実日付（`startDate`, `endDate`, `instant`）で照合すべき
3. 例: 2023年度のデータを取得するには、`period` が `2023-04-01~2024-03-31` のコンテキストを検索

#### 3. 前期比較データと前年度 filing の一致性（J-6.3）

[F6] + [F7] より、トヨタの2年分では **Prior1Year と前年 CurrentYear が 100% 一致**した。

- Duration: 106 concept 全て一致
- Instant: 69 concept 全て一致
- period 日付も完全に一致（`2023-04-01~2024-03-31` / `2024-03-31`）

**ただし、以下のケースでは不一致が生じうる**:
1. **遡及適用（会計方針の変更）**: 当期 filing の Prior1Year データは遡及適用後の修正値が入る（J-5.a.md）。前年 filing の CurrentYear データは修正前の値。
2. **遡及修正再表示（過去の誤謬の訂正）**: 同様に当期 filing の Prior1Year が修正後の値。
3. **表示方法の変更**: 科目の組替え等で数値が変わる場合がある。

不一致がある場合の「正」の値:
- **時系列分析の目的では、当期 filing の Prior1Year データ（修正後）が推奨**。これは最新の会計方針に基づいて比較可能性を確保した値であるため。
- 前年 filing の CurrentYear データは、当時の会計方針に基づくオリジナル値として別途保持する価値がある。

#### 4. entity/identifier の安定性（J-6.4）

[F2] + [F12] より、entity/identifier は同一企業で年度間安定。

- トヨタは2年分とも `E02144-000` で完全一致
- EDINET コードは法人格に紐付くため、通常は変わらない
- **変更が生じうるケース**:
  - 合併（吸収合併の場合、消滅会社のコードが廃止される）
  - 持株会社化（新設の持株会社に新コードが付与される）
  - 組織変更（株式会社→合同会社等、実務上は稀）
- **パーサーの対応**: EDINET コードをプライマリキーとして使用し、合併等のコード変更は手動マッピングテーブルで対応するのが実用的

### 確信度

- J-6.1（concept の安定性）: **高** -- 実データで 96.4% の一致率を確認（[F3]）、タクソノミ差分統計（[F10]）でも年間追加/削除は少数。H-3.a.md の分析とも整合。
- J-6.2（Context ID の安定性）: **高** -- 仕様書の命名規則（A-2.a.md）が毎年適用され、実データ（[F5]）でも基本 ID は両年に存在。period 日付での照合方法は XBRL 標準のアプローチ。
- J-6.3（前期比較データの一致）: **中-高** -- トヨタ2年分では 100% 一致（[F6]）だが、これは遡及適用がなかったケース。遡及適用時の不一致は仕様上想定される（J-5.a.md [F2]）が、実データでの不一致例は未検証。
- J-6.4（entity/identifier の安定性）: **高** -- 実データ確認（[F2]）。合併等の特殊ケースは API 仕様書（[F12]）の設計からも示唆されるが、通常運用では安定。
