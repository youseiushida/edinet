| サブ質問 | 回答 |

### 情報源（Fact）

- [F1] スクリプト: `docs/QAs/scripts/K-3.csv_sample.py` 実行結果 — CSV ファイルは UTF-16 LE (BOM付き) エンコーディング、**タブ区切り**（カンマ区切りではない）
- [F2] スクリプト実行結果 — CSV のヘッダー行（9列）: `"要素ID" "項目名" "コンテキストID" "相対年度" "連結・個別" "期間・時点" "ユニットID" "単位" "値"`
- [F3] スクリプト実行結果 — PublicDoc 対応 CSV は 159 行（ヘッダー含む）。AuditDoc 対応 CSV は各2行（ヘッダー + TextBlock 1件）
- [F4] スクリプト実行結果 — CSV の要素IDは名前空間プレフィックス付き（例: `jppfs_cor:CurrentAssets`, `jpdei_cor:NumberOfSubmissionDEI`）
- [F5] スクリプト実行結果 — コンテキストIDにディメンション情報が埋め込まれている（例: `Prior1YearInstant_NonConsolidatedMember`, `CurrentYearDuration_NonConsolidatedMember`）
- [F6] スクリプト実行結果 — ZIP 内のディレクトリ名は `XBRL_TO_CSV/` で、CSV ファイル名は対応する XBRL ファイル名と一致
- [F7] スクリプト実行結果 — file_type=1 の ZIP (702KB) に対し、file_type=5 の CSV ZIP は 52KB と大幅に小さい
- [F8] スクリプト実行結果 — CSV には Presentation/Calculation/Definition リンクベース情報は含まれない（Fact のフラットリストのみ）

### 推論（Reasoning）

1. [F1] + [F2] より、CSV は XBRL Fact を9列のタブ区切りテーブルに変換したものである。EDINET 側で `XBRL_TO_CSV` として提供しており、XBRL→CSV の機械的変換であることが名称から明確。
2. [F2] + [F4] + [F5] より、CSV には XBRL Fact の主要構成要素（Concept名、Context、Unit、値）がすべて含まれる。日本語ラベル（項目名）も付与されており、単純なデータ抽出であれば XBRL パースなしで利用可能。
3. [F8] より、CSV は以下の情報を**含まない**ため XBRL パーサーの完全代替にはならない:
   - **Presentation リンクベース**: 財務諸表の表示順序・階層構造（BS/PL の勘定科目ツリー）
   - **Calculation リンクベース**: 勘定科目間の計算関係（合計値の検証）
   - **Definition リンクベース**: ディメンション定義・次元構造
   - **DTS (Discoverable Taxonomy Set)**: スキーマ定義・拡張タクソノミ
   - **Footnote**: 注記リンク
   - **iXBRL の書式情報**: HTML 表示用の構造・スタイル
4. [F5] より、ディメンション情報はコンテキストID文字列に埋め込まれている（例: `_NonConsolidatedMember`）。構造化されたディメンション分解には XBRL パースが必要。
5. [F3] より、AuditDoc の CSV は TextBlock 1件のみを含み、監査報告書の全文テキストが「値」列に格納されている。テキスト抽出用途としては十分。

### CSV が代替可能/不可能なユースケース

| ユースケース | CSV で可能か | 理由 |
|---|---|---|
| 特定の勘定科目の数値取得 | **可能** | 要素ID + コンテキストで一意に特定可 |
| 前期比較 | **可能** | `Prior1Year*` / `CurrentYear*` でコンテキスト区別可 |
| 連結/個別の区別 | **可能** | 「連結・個別」列 + コンテキストID |
| 財務諸表の構造化表示 | **不可** | Presentation ツリーがない |
| 計算チェック | **不可** | Calculation リンクがない |
| 拡張タクソノミの意味解釈 | **不可** | DTS 情報がない |

### 結論

**CSV は XBRL パーサーの代替にならない。ただし、Fact データの簡易取得手段として補助的に活用可能。**

XBRL パーサーの開発は計画通り必須である。CSV は「数値の取得」に限定した軽量 API として、パーサー完成前の暫定手段や、パフォーマンス重視のバルク取得に活用できる。
