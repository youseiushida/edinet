# 調査大方針
##　ファイルについて
各問題に対して　hoge.q.mdとhoge.a.mdがあり、.q.mdは問題文、.a.mdは回答を記述します。

例えば A-1.q.mdの回答はA-1.a.mdに記述します。回答と回答の検証を同じセッション内で行ってはいけません。
また、調査は /docs/仕様書/ディレクトリの中（よく調べたら後からhttps://www.fsa.go.jp/search/20251111.htmlにて2026年度版のデータが見つかったのでそれは/docs/仕様書/ディレクトリ/2026にあります）、/data/source/内のデータ、あるいはプログラムを用いた検証で行い、ほしい情報が存在しない場合はその旨を回答の代わりに記述する必要があります。また、提供された情報源はOCRによるものなので図はリンクでしかなくOCRも完ぺきではないので回答にはほしい情報が見つからなかった場合もこの資料の～行目に対応する場所を見るとほしい情報があるかもしれないというようなことや、ここで示されている外部資料にあるかもしれないというようなアドバイスを含めても良いです。

## 優先順位

> **回答の優先順位**: 以下の順で回答してください。特に ★ 付きの質問はパーサーのアーキテクチャを左右するため最優先です。
> 1. **★最優先**: K-2 (iXBRL)、K-3 (CSV代替可能性)、H-1b (DTS解決)、P-1 (実ファイル検証) — 回答次第でカテゴリ A 全体の前提が変わる
> 2. **高**: A-2, C-3, D-1, E-1, F-1 — パーサーの核心設計に直結
> 3. **中**: その他 — 実装品質・網羅性に影響
>
> **Day 9 着手に必要な最小回答セット**: 全問に回答する前に、以下の 5 問だけ先に回答すれば Day 9 のパーサー MVP に着手可能です。
> 1. **P-1a** + **P-2**（実ファイル）— iXBRL か従来型か、名前空間、Context 構造が一発で判明
> 2. **P-3**（提出者 .xsd）— DTS 解決の import URL が判明
> 3. **K-2**（iXBRL）— パーサーのアーキテクチャ決定
> 4. **K-3**（CSV）— XBRL パース自体の必要性判断
>
> **スコープ注記**: 各質問の末尾に `[v0.1.0]` / `[v0.2.0+]` の記載がある場合、回答の詳細度をそれに合わせて調整してください。

> **質問間の依存関係マップ**: 矢印は「左が前提、右が依存（左を先に回答すべき）」を意味する。
> ```
> K-2 (iXBRL)        ──→ B-1.10, H-1, カテゴリ A 全体の前提, A-9/A-10
> K-3 (CSV)           ──→ XBRL パース自体の必要性
> H-3 (バージョニング) ──→ H-1b の URL→ローカルパス変換
> Q-2.4 (API提供期間)  ──→ H-3 (対応すべきバージョン数)
> H-1b (DTS)          ──→ カテゴリ C 全体のタクソノミ解決方法
> D-3 (会計基準判別)   ──→ D-1, D-2 の調査範囲
> A-2 (Context)       ──→ E-2 (連結/個別), E-3 (期間)
> C-6 (Presentation)  ──→ I-1, I-2, I-3
> I-1 (role URI)      ──→ E-1 (財務諸表識別)
> P-1e (訂正実ファイル) ──→ G-7 (訂正報告書の検証)
> H-3, J-5 (前期データ) ──→ J-6 (クロスフィリング)
> C-11, H-9 (規模感)   ──→ H-9b (キャッシュ戦略)
> P-1 (実ファイル)     ──→ A, B, K の多くの質問が自動回答
> P-6 (タクソノミサンプル) ──→ C, C-4 の多くの質問が自動回答
> ```
>
> **注意: Q-2.4 → H-3 → H-1b の連鎖について**: Q-2.4（API提供期間）は Q-2 のサブ質問だが、H-3 の前提となるため Q-2 本体より先に独立して調査する。Q-2 の残りは H-3 完了後に調査する。
>
> **回答の粒度ガイダンス**: 表形式の質問（C-1, C-3 等）は表形式で簡潔に回答してください。XML 例を求めている質問は実際のファイルからコピー＆ペーストで構いません。
>
> **質問間の重複について**: 以下の質問は異なる文脈から同じ仕様を確認しています。重複する箇所は簡潔に「→ X-Y を参照」で済ませてください:
> - A-1 の IFRS 名前空間 → D-1.2 で詳述するため A-1 では簡潔でよい
> - B-1.10 の `find_primary_xbrl_path` → K-2.7 で詳述するため B-1.10 は簡潔でよい
> - H-1.3 のタクソノミ解決方法 → H-1b で詳述するため H-1.3 は要約のみでよい
>
> **回答不要の目安**: タクソノミファイル（ALL_20251101）や実際のダウンロード ZIP を直接読めば自明に回答できる質問は、**ファイルだけでは判断できない部分のみ回答してください**。特に P カテゴリの回答が得られると以下が自動回答されます:
> - **P-1** → B-1（ZIP構造）, K-2（iXBRL有無）, B-1.10（find_primary_xbrl_path 修正要否）
> - **P-2** → A-1（名前空間）, A-2（Context構造）, H-1（schemaRef）
> - **P-3** → H-1b（DTS解決, import URL）, E-6（拡張要素）, A-1（提出者名前空間URI）
> - **P-1f** → G-5（大量保有報告書構造）
> - **P-1g** → G-6（公開買付構造）
> - **P-1h** → G-8（特定有価証券構造）
> - **P-6** → C-3（業種一覧）, C-4（サフィックス体系）, C-9（concept属性）, C-10（XML構造）, C-5（ラベル構造）

## 調査の並列可能性について

依存関係マップに基づき、調査を Wave（段階）× Lane（並列レーン）で整理する。
同一 Wave 内の Lane は互いに依存がなく並列実行可能。次の Wave は前の Wave の結果に依存する。

### Wave 1（基盤調査 — 即時着手可能、全レーン並列）

| Lane | 質問 | 概要 | 手段 |
|------|------|------|------|
| **A: アーキテクチャ決定** | P-1 → P-2, P-3, P-2b → K-2 | 実ファイルDL・展開し、iXBRL有無・ZIP構造・名前空間・schemaRefを確認 | API + コード実行 |
| **B: CSV代替判断** | K-3, P-5, K-1, K-4 | CSV(file_type=5)を取得し、XBRLパース自体の必要性を判断 | API + コード実行 |
| **C: タクソノミサンプリング** | P-6 | ALL_20251101 のディレクトリ構造・XSD・ラベル・Presentation を採取 | ローカルファイル読取 |
| **D: 仕様書先行調査** | D-3, F-1, C-6 | 会計基準判別方法、DEI要素一覧、Presentation構造を仕様書から調査 | 仕様書読取のみ |

> **Wave 1 のゲート判定**: K-2（iXBRL）と K-3（CSV）の結果がアーキテクチャを決定する。
> 結果次第で後続 Wave の前提が大きく変わるため、Needs Review の対象になりうる。

### Wave 2（コア設計 — Wave 1 完了後、全レーン並列）

| Lane | 質問（調査順） | 依存元 | 概要 |
|------|------|--------|------|
| **E: XBRLデータモデル** | A-1, A-2, A-3, A-4, A-7, A-8, H-2 | K-2, P-2 | Context/Fact/Unit の完全な構造定義、名前空間プレフィックス安定性 |
| **F: 会計基準・IFRS** | D-1, D-2, F-2, P-4 | D-3, K-2, F-1 | IFRS/US-GAAP の名前空間・ZIP差異、DEI利用シーン |
| **G: DTS解決** | Q-2.4(部分) → H-3 → H-1b, H-1 → Q-1 → Q-2(残り) | P-3, P-6 | URL→ローカルパス変換、バージョニング、フォールバック |
| **H: 財務諸表組立** | I-1, C-3, E-1b, E-1c, E-1d | C-6, P-6 | Role URI マッピング、業種コード一覧 |

> **Lane G の調査順について**: Q-2.4（API提供期間）→ H-3（バージョニング）→ H-1b（DTS解決）→ H-1（schemaRef/linkbaseRef）→ Q-1 → Q-2（残り）の順で調査する。Q-2.4 を先行させることで H-3 ↔ Q-2 の循環依存を解消している。
>
> **E-1b/c/d の配置について**: E-1b/c/d は名前上 E-1 の枝番だが、E-1（Wave 3）が I-1 に依存するのに対し、E-1b/c/d は C-6・P-6（Wave 1）の結果のみで調査可能なため Wave 2 に配置している。

### Wave 3（統合・中優先 — Wave 2 の関連レーン完了後）

| Lane | 質問 | 依存元 | 概要 |
|------|------|--------|------|
| **I: 財務諸表識別** | E-1, E-2, E-3, E-4, E-5, F-1b | I-1, A-2 | PL/BS/CF分類、連結/個別判定、期間ロジック |
| **J: ZIP・拡張要素** | B-1, B-2, B-3, E-6, E-7 | P-1, P-3, H-1b | ZIP構造詳細、拡張要素処理 |
| **K: Presentation活用** | I-1b, I-2, I-3, C-7, C-8 | C-6, I-1 | ツリー構築、ソート、フィルタリング |
| **L: 報告書バリエーション** | G-1〜G-6, G-8, G-8b, G-9 | P-1 | 各書類種別ごとの構造差異 |

### Wave 4（高度機能・v0.2.0+ — MVP後）

| 質問 | 概要 |
|------|------|
| A-5, A-6, A-9, A-10 | Footnote、Tuple、テキストFact HTML |
| G-7 (訂正報告書) | 訂正チェーン処理 |
| J-1〜J-5 → J-6 | 時系列比較・クロスフィリング（J-6 は J-5 完了後） |
| H-4, H-4b, H-5〜H-8, H-10, H-11 → H-9 → H-9b | キャッシュ戦略・パフォーマンス（H-9b は C-11, H-9 完了後） |
| C-1, C-1b, C-1c, C-2, C-4, C-5, C-9〜C-12 | タクソノミ詳細構造 |

> **Wave 4 内の依存関係**: J-6 は J-5 に依存、H-9b は C-11 と H-9 に依存する。着手時に注意すること。
> P-7 は空ファイルのため Wave 計画に含めない。

### 決定ゲート（Wave 間の判断ポイント）

```
Wave 1 完了時:
  K-2 = iXBRL?  → YES: HTMLパーサー・scale処理が必要（Wave 2-E に影響）
                → NO:  従来XMLパーサーで十分
  K-3 = CSV完全? → YES: XBRLパース不要の可能性（全体計画見直し → Needs Review）
                → NO:  XBRLパーサー必須（計画通り続行）
```

## KANBAN.md について

調査全体の進捗管理には `KANBAN.md` を使用します。各質問の調査内容の詳細は `.q.md` を、調査結果の詳細は `.a.md` を参照してください。

### 状態の定義

| 状態 | 判定方法 | KANBAN.md に記載するか |
|---|---|---|
| **TODO** | `.a.md` が存在しない `.q.md` | しない（暗黙） |
| **Blocked** | README.md の依存関係マップから自明 | しない（暗黙） |
| **In Progress** | 現在調査中 | する |
| **Done** | `.a.md` が存在する | しない（暗黙） |
| **Needs Review** | 前提崩壊・大幅方向転換など上位者の判断が必要 | する（理由とともに） |

### 運用ルール

- **In Progress**: 並列で調査する際に担当が重複しないよう、着手時に記載し完了時に除去する
- **Needs Review**: 調査中に致命的な発見（前提の崩壊、計画の大幅な方向転換が必要な事態など）があった場合、理由とともに記載し上位者のレビューを仰ぐ。該当する `.a.md` にも詳細を記述すること
- KANBAN.md には状態のみを記載し、調査の詳細や結果は一切書かない

## 回答テンプレート

回答は以下の形式で `docs/QAs/A-1.a.md` に記載してください（例としてA-1についてです。）。また、調査にプログラムを用いる場合は/docs/QAs/scripts/配下に A-1.foo.pyに書くこと。fooは任意の文字列で良く、複数のプログラムを用いる場合にこれにより識別可能です。

### テンプレート構造の設計意図

このテンプレートは以下の検証を構造的に可能にするために設計されている:

1. **対応検証**: 「質問への対応」セクションにより、q.md の各サブ質問に漏れなく回答しているかをチェックリスト形式で検証可能
2. **Fact/Opinion 分離**: 「情報源（Fact）」は観察事実のみ、「推論（Reasoning）」は Fact 番号を明示した論理式形式とすることで、根拠と主張の対応関係を追跡可能
3. **推論の連鎖の透明性**: 複数の Fact を組み合わせた推論は `[F1] + [F2] → 結論` のように明示し、検証者が各ステップの妥当性を個別に判断可能

```markdown
## A-1. 名前空間宣言の全体像

### 質問への対応

q.md のサブ質問ごとに回答箇所を対応付ける。

| サブ質問 | 回答 |
|----------|------|
| A-1.1 ○○について | （簡潔な回答。詳細は推論セクション参照） |
| A-1.2 △△について | （簡潔な回答） |
| ... | ... |

### 情報源（Fact）

観察事実のみを記載する。推論や解釈は含めない。

- [F1] パス: `docs/仕様書/foo.md` L.120-135 — 「（該当箇所の引用または要約）」
- [F2] スクリプト: `docs/QAs/scripts/A-1.ns.py` 実行結果 — 「（出力の該当部分を引用）」
- [F3] パス: `docs/仕様書/bar.md` L.45 — 「（引用）」

### 推論（Reasoning）

Fact 番号を用いて、結論に至る論理を明示する。

1. [F1] より、○○は△△である。
2. [F1] + [F2] より、○○の実ファイルでの実態は仕様書の記述と一致する。よって □□ と結論する。
3. [F3] に関連する記述があるが、◇◇の観点では確認できなかった。→ 確信度: 中

### 確信度

- 高 / 中 / 低（仕様書に明記 / 実例から推測 / 未確認）
- （確信度が中・低の場合）不確実な点: ○○について仕様書に明示的な記載がなく、[F2] の実例からの推測に基づく
```

### スクリプトの記載要件

`docs/QAs/scripts/` 配下のスクリプトには、ファイル冒頭に以下を記載すること:

```python
"""A-1. 名前空間宣言の調査スクリプト

実行方法: uv run docs/QAs/scripts/A-1.ns.py
前提: EDINET_API_KEY 環境変数、または EDINET_TAXONOMY_ROOT のパスが必要
出力: （何が出力されるかの簡潔な説明）
"""
```

## 回答の検証方法

回答（`.a.md`）の検証は以下の手順で行う。**検証者は回答を信用せず、生データに直接当たって事実を確認すること。**

### 検証の手順

#### Step 1. 完全性検証（q.md との突合）

対象の `.q.md` を開き、「質問への対応」テーブルと照合する。

- [ ] 全サブ質問に対応する行が存在するか
- [ ] 「回答なし」「該当なし」の場合にその理由が記載されているか

#### Step 2. Fact 検証（生データの直接確認）

情報源（Fact）セクションの各エントリについて、**引用元を実際に開いて確認する**。

**仕様書の引用の場合:**
- [ ] 引用パスのファイルが存在するか
- [ ] 指定行番号に該当する記述が実際にあるか
- [ ] 引用内容が原文と一致しているか（省略・意訳による意味の変質がないか）

**スクリプト結果の引用の場合:**
- [ ] スクリプトが `docs/QAs/scripts/` に存在するか
- [ ] スクリプトを実際に実行し、引用された出力が再現されるか（`uv run` で実行）
- [ ] スクリプトのロジックに明らかな誤りがないか（間違ったファイルを読んでいる、フィルタ条件が不適切、等）

**生データへの直接アクセス:**
- 仕様書引用: `docs/仕様書/` 配下の該当ファイルを直接開いて確認
- タクソノミ: `EDINET_TAXONOMY_ROOT`（ALL_20251101）の該当ファイルを直接開いて確認
- API取得データ: スクリプトを再実行するか、同等の API 呼び出しで取得して確認

#### Step 3. 推論検証（論理の妥当性）

推論（Reasoning）セクションの各ステップについて:

- [ ] 引用している Fact 番号が情報源セクションに存在するか
- [ ] Fact の内容から当該結論が論理的に導かれるか
- [ ] 飛躍がないか（Fact に書かれていない前提を暗黙に仮定していないか）
- [ ] **反証の探索**: 結論と矛盾する記述が同じ情報源や関連する仕様書に存在しないか

#### Step 4. 依存関係の整合性検証

依存関係マップで当該質問が依存している先行回答がある場合:

- [ ] 先行回答（依存元の `.a.md`）の結論と矛盾していないか
- [ ] 先行回答の確信度が「低」の場合、その不確実性を考慮した記述になっているか

### 検証における推奨事項

- **回答を読む前に q.md を読む**: 検証者自身が先に問題を把握し、期待される回答の方向性を持ってから a.md を読むこと。回答に引きずられた確認を防ぐため
- **反証を意識的に探す**: 回答の結論を支持する証拠だけでなく、矛盾する証拠がないかを仕様書や生データから能動的に探すこと
- **OCR 由来の情報源に注意**: `docs/仕様書/` の資料は OCR 由来のため、図表が欠落している場合や誤認識がありうる。Fact の引用内容が不自然な場合は原本（PDF等）の該当箇所を確認すること
- **確信度「低」は要追加調査**: 確信度が「低」の回答は、検証で追加の情報源が見つからない限り暫定回答として扱い、後続の調査で上書きされる可能性がある旨を記録すること

### 検証結果の記録

検証結果は検証対象の `.a.md` の末尾に以下の形式で追記する:

```markdown
### 検証

- 検証者: （検証者の識別子）
- 検証日: YYYY-MM-DD
- 判定: OK / NG / 要追加調査
- （NG・要追加調査の場合）指摘事項:
  - [F2] のスクリプト再実行で異なる結果が得られた: （詳細）
  - 推論 3 の飛躍: [F3] からは ○○ とは断定できない。△△ の可能性もある
```
