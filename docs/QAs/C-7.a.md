## C-7. 計算リンクベースの詳細構造

### 質問への対応

| サブ質問 | 回答 |
|----------|------|
| C-7.1 weight 属性 | `1`（加算）と `-1`（減算）の2値のみ。サンプルインスタンスの全 185 アークで確認: weight=+1 が 175 件、weight=-1 が 10 件。他の値は存在しない。仕様書では「半角数字」とのみ記載されているが（設定規約書 L.1853）、ガイドラインの図表 6-5-2 では「加算時 1、減算時 -1」と明記。 |
| C-7.2 計算ツリーの例 | 連結 PL の例: `ProfitLoss = IncomeBeforeIncomeTaxes(+1) + IncomeTaxes(-1)`、`IncomeBeforeIncomeTaxes = OrdinaryIncome(+1) + ExtraordinaryIncome(+1) + ExtraordinaryLoss(-1)`、`OrdinaryIncome = OperatingIncome(+1) + NonOperatingIncome(+1) + NonOperatingExpenses(-1)`、`OperatingIncome = GrossProfit(+1) + SellingGeneralAndAdministrativeExpenses(-1)`、`GrossProfit = NetSales(+1) + CostOfSales(-1)` |
| C-7.3 計算リンクの role URI | **Presentation と同じ提出者用ロール URI を使用。** _cal.xml の jppfs ロール 8 件は全て _pre.xml の jppfs ロール（ラベルロール 3 件を除く 8 件）と一致。_pre のみの 3 件はラベルロール（`label`, `positiveLabel`, `totalLabel`）で計算リンクには不要。 |
| C-7.4 v0.1.0 での必要性 | **検算用途のみ。** 計算リンクベースをパースしなくても財務諸表の組み立ては可能。計算リンクは Fact 値の整合性チェックに使用される。arcrole は `summation-item` のみで、親 = 合計科目、子 = 構成科目として加減算関係を定義。 |
| C-7.5 計算不一致の扱い | XBRL 2.1 仕様により、decimals 属性に基づく丸め許容が定義されている。EDINET 仕様書では「期間時点区分が異なる勘定科目間の加減算関係は計算リンクを設定できない」（ガイドライン L.3112）という制約があり、CF計算書の一部（例: 期首残高 + 増減額 = 期末残高）は計算リンクの対象外。不一致があっても Fact 値自体は提出者の報告値として信頼してよい。 |
| C-7.6 Calculations 1.1 への対応 | **EDINET は従来の `summation-item` のみを使用。** Calculations 1.1 仕様への対応は確認されない。設定規約書の計算リンクアークロールは `summation-item` の 1 種類のみ（L.1840-1844）。 |

### 推論（Reasoning）

1. **C-7.1 weight 属性**: [F1] の仕様書では「半角数字」とのみ記載だが、[F2] のガイドラインで「加算時 1、減算時 -1」と具体的に規定されている。[F6] の実ファイル解析で全 185 アークが {+1, -1} の2値のみであることを確認。XBRL 2.1 仕様上は任意の数値が可能だが、EDINET の運用上は 1 と -1 に限定されている。

2. **C-7.2 計算ツリーの例**: [F9] の連結 PL 計算ツリーが実例を示す。`summation-item` arcrole で親 = 合計科目、子 = 構成科目として表現される。例えば `GrossProfit = NetSales(+1) + CostOfSales(-1)` は「売上総利益 = 売上高 - 売上原価」を意味する。注意点として、費用科目（`NonOperatingExpenses` 等）は親に対して weight=-1 で接続されるが、その費用内の内訳科目（`InterestExpensesNOE` 等）は費用合計に対して weight=+1 で接続される。これは「営業外費用合計 = 支払利息 + 有価証券売却損 + その他」の関係を表す。

3. **C-7.3 role URI の一致**: [F8] により、_cal.xml の jppfs ロール 8 件は _pre.xml の jppfs ロール（ラベルロール除く）と完全に一致する。Presentation と Calculation は同じロール URI を共有し、同一ロール内で表示順序と計算関係を別々に定義する。

4. **C-7.4 v0.1.0 での必要性**: [F5] により計算リンクは「勘定科目間の計算上の関係」を定義するものであり、表示順序（Presentation）やディメンション構造（Definition）とは異なる補助的情報である。財務諸表の組み立て（科目の並び順、インデント、ラベル）は Presentation リンクのみで完結する。Calculation は Fact 値の検算に使用する。v0.1.0 では Calculation をパースせずとも財務諸表のレンダリングが可能。

5. **C-7.5 計算不一致**: [F3] + [F4] により、EDINET の計算リンクには「期間時点区分が同一の科目間のみ」という制約がある。CF 計算書の期首残高 + 増減額 = 期末残高 のような duration/instant 混在の関係は計算リンクで表現できない。また XBRL 2.1 仕様により、decimals 属性の値に基づく丸め誤差が許容される（子科目の合計と親科目の値が decimals の精度範囲内で一致すればよい）。不一致がある場合も、Fact 値は提出者が報告した値として信頼してよい。

6. **C-7.6 Calculations 1.1**: [F1] の設定規約書で計算リンクのアークロールは `summation-item` の 1 種類のみが規定されている。Calculations 1.1 仕様で追加されたアークロール（例: `balance-closingPeriodType` 等）は EDINET の仕様書に記載がなく、対応していないと推定される。

### 確信度

- C-7.1 weight 属性: **高** -- 仕様書（1/-1）と実ファイル（185 アーク全て 1/-1）の両方で確認
- C-7.2 計算ツリーの例: **高** -- スクリプトで実ファイルから構築した計算ツリーを直接確認
- C-7.3 role URI の一致: **高** -- スクリプトで _cal と _pre の roleRef を照合し完全一致を確認
- C-7.4 v0.1.0 での必要性: **高** -- 仕様書の定義と実ファイルの構造から、Calculation は Presentation に対する補助的情報であることが明確
- C-7.5 計算不一致: **高** -- ガイドラインに periodType 制約が明記。丸め許容は XBRL 2.1 仕様の一般的知識だが、EDINET 固有の閾値は未確認
- C-7.6 Calculations 1.1: **中** -- 仕様書に記載がないことからの推定。明示的な「非対応」声明は未確認
