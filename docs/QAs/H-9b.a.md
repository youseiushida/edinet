## H-9b. タクソノミのプリビルド・キャッシュ戦略

### 質問への対応

| サブ質問 | 回答 |
|----------|------|
| H-9b.1 プリビルドの安全性 | **ラベル辞書は安全にプリビルド可能**。ただし提出者による英語ラベル上書き（prohibited + priority 解決）が存在する場合、デルタ適用が必要。表示/計算/定義ツリーはリキャスト方式のため、標準タクソノミのプリビルドを直接再利用することは不適切 |
| H-9b.2 デルタ適用方式 | ラベル辞書へのデルタ適用は安全。prohibited による標準ラベルの無効化が必要な場合はコピーオンライトで対応可能。表示/計算/定義リンクベースは提出者が独立ファイルを作成するため、デルタ適用ではなく提出者ファイルをそのまま使用する |
| H-9b.3 キャッシュの粒度 | 3 層に分ける: (1) 標準ラベル辞書（全 filing 共通、プリビルド可）、(2) 標準 concept 辞書（全 filing 共通、プリビルド可）、(3) リンクベースツリー（提出者ごと、キャッシュ不可だが提出者ファイルのみパースすればよい） |

### 情報源（Fact）

- [F1] パス: `docs/QAs/H-9.a.md` — タクソノミの規模感:
  - Concept 数: 9,697（約 1.8 MB）
  - ラベル数: 47,352（約 13.5 MB）
  - Arc 数: 62,697（Presentation 26,578 + Calculation 10,927 + Definition 25,192）
  - 全タクソノミオンメモリ: 約 21 MB
- [F2] パス: `docs/QAs/C-11.a.md` — マージルールの要約:
  - **名称リンクのみリユーズ**: 標準タクソノミのラベルを DTS に含め、提出者が追加/上書き
  - **表示/計算/定義はリキャスト**: 提出者が独立ファイルを新規作成。標準タクソノミの同名ファイルとマージされない
  - prohibited は名称リンクにおいてのみ出現し得る
  - priority 解決: 標準=0、提出者>=1 で提出者が常に優先
  - 等価性: from + to + arcrole
- [F3] パス: `docs/QAs/C-6.a.md` — 表示リンクの特性:
  - リキャスト方式: 提出者が独立した `_pre.xml` を作成
  - prohibited 不可（表示/定義/計算の全てで明示的に禁止）
- [F4] パス: `docs/仕様書/提出者別タクソノミ作成ガイドライン.md` L.889-900 — リユーズ/リキャスト表:
  - No.1 名称リンク = リユーズ
  - No.3 表示リンク = リキャスト
  - No.4 計算リンク = リキャスト
  - No.5 定義リンク（勘定科目ツリー）= リユーズ（ただし J-GAAP の勘定科目追加時のみ）
  - No.6 定義リンク（ディメンション）= リキャスト
- [F5] パス: `docs/QAs/C-11.a.md` [F8] — 標準タクソノミの全 47,352 labelArc は priority=0（デフォルト）。サンプル提出者の labelArc も全て priority=0 で新規拡張要素向け。prohibited の使用例はサンプルに存在しない
- [F6] パス: `docs/QAs/H-9.a.md` — 有報 1 社あたり約 2,000 Fact。バッチ 1,000 社で 15-30 分の見積もり

### 推論（Reasoning）

#### H-9b.1: プリビルドの安全性

1. **ラベル辞書のプリビルド** ([F1] + [F2] + [F5]):
   - [F2] より、名称リンクはリユーズ方式であり、標準タクソノミのラベルと提出者のラベルが XBRL 2.1 マージルールで統合される
   - [F5] より、実態としてサンプル提出者の labelArc は全て新規拡張要素向けであり、標準要素のラベルと競合するケースはサンプルにはない
   - よって、標準タクソノミのラベル辞書（concept → label のマッピング）は全 filing で安全に再利用可能。提出者の拡張要素ラベルは提出者の `_lab.xml` から追加するだけでよい
   - **例外ケース**: 英語ラベルの上書き（[F2] C-11.3）。提出者が `use="prohibited"` で標準ラベルを無効化し、新しいラベルを定義するケースがあり得る。この場合、プリビルド済みのラベル辞書から該当ラベルを除去する必要がある

2. **表示/計算/定義ツリーのプリビルド** ([F2] + [F3] + [F4]):
   - [F3] + [F4] より、表示リンク・計算リンク・定義リンク（ディメンション）はリキャスト方式。提出者が EDINET タクソノミの arc 構造を「参考にしつつ」独自のファイルを作成する
   - これは提出者の `_pre.xml` が標準タクソノミの `_pre.xml` と同じ role URI を使用していても、提出者ファイルが独立した完全なツリーを定義していることを意味する
   - よって、標準タクソノミの表示/計算/定義ツリーをプリビルドしても、各 filing で提出者のファイルをパースする必要がある
   - **ただし**: 標準タクソノミの概念（concept）の属性（type, periodType, balance, abstract, nillable 等）は全 filing 共通であり、concept 辞書としてプリビルド可能

#### H-9b.2: デルタ適用方式

3. **ラベル辞書へのデルタ適用** ([F2] + [F5]):
   - 標準ラベル辞書をイミュータブルなベースとして保持し、提出者ごとにコピーオンライトでデルタを適用する方式が安全
   - 実装案:
     ```python
     # ベース（プリビルド済み、全 filing 共通）
     base_labels: dict[str, dict[str, str]]  # concept → {role: label_text}

     # 提出者デルタ適用（コピーオンライト）
     labels = ChainMap(filer_labels, base_labels)
     ```
   - prohibited による標準ラベルの無効化は、`filer_labels[concept][role] = None` 等のマーカーで表現可能
   - [F5] より、実態として prohibited の使用はサンプルに存在せず、大半のケースでは単純な追加（新規拡張要素のラベル）のみ

4. **表示/計算/定義リンクベースのデルタ適用** ([F3] + [F4]):
   - リキャスト方式のため「デルタ適用」ではなく「提出者ファイルの独立パース」が正しいアプローチ
   - 提出者の `_pre.xml` は完全なツリー構造を含んでおり、標準タクソノミの `_pre.xml` をベースとして差分を適用するのではない
   - [F4] No.5 の例外: 定義リンク（勘定科目ツリー）は J-GAAP の場合リユーズ。これは標準の勘定科目ツリーに提出者の拡張要素を追加する方式。このケースではデルタ適用（追加のみ、prohibited なし）が必要

#### H-9b.3: キャッシュの粒度

5. [F1] + 推論 1-4 を統合し、以下の 3 層キャッシュ構造を推奨:

| 層 | 内容 | 共有範囲 | サイズ見積もり | ビルドタイミング |
|----|------|---------|-------------|--------------|
| L1: Concept 辞書 | concept QName → {type, periodType, balance, abstract, nillable, substitutionGroup} | 全 filing 共通 | ~2 MB | 起動時に 1 回プリビルド |
| L2: ラベル辞書 | concept → {role → {lang → text}} | 全 filing 共通（提出者拡張分は filing ごとに追加） | ~14 MB | 起動時に 1 回プリビルド |
| L3: リンクベースツリー | role URI → ツリー構造（親子関係 + order + preferredLabel） | filing ごと | ~数百 KB/filing | filing ごとにパース |

- **L1 + L2 のシリアライズ**: pickle / msgpack / JSON で保存。21 MB 程度のデータのため、読み込み時間は 0.1-0.5 秒程度
- **L3 の最適化**: 提出者の `_pre.xml` は標準タクソノミの concept を参照するが、concept の属性は L1 から取得可能。よって L3 のパースは arc 構造の読み取りのみで済み、concept の再解決は不要

6. **バッチ処理での効果** ([F6]):
   - L1 + L2 のプリビルドにより、1,000 社のバッチ処理で DTS 解決 + ラベル辞書構築のオーバーヘッドを 999 回分削減
   - 標準タクソノミの XML パース（9,697 concepts + 47,352 labels + 62,697 arcs）は初回のみで、以降は pickle からの読み込み（0.1-0.5 秒）で代替
   - 1 社あたりの処理: ZIP 展開 + 提出者 `_lab.xml` パース（追加分のみ）+ 提出者 `_pre.xml` パース + Fact 抽出 → 0.5 秒以下を目標

### 確信度

- H-9b.1 プリビルドの安全性: **高** — C-11.a.md のマージルール分析に基づく。名称リンクのリユーズ/リキャストの区別は仕様書で明確
- H-9b.2 デルタ適用: **高** — リキャスト方式の表示/計算/定義リンクでは「デルタ適用」が不要であることが仕様上明確。ラベルのデルタ適用は XBRL 2.1 のマージルールに準拠
- H-9b.3 キャッシュの粒度: **中** — 設計提案であり、実装上の最適化（メモリ使用量、シリアライズ速度等）は実ベンチマークが必要
- 不確実な点: 定義リンク（勘定科目ツリー）のリユーズ方式（[F4] No.5）の詳細。J-GAAP の場合のみリユーズとされるが、提出者がどの程度勘定科目を追加するかはケースバイケースであり、デルタの規模感は未確認
