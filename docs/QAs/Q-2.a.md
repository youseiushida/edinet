## 回答

### Q-2.1: ALL_20251101 のみで古い書類をパースできるか。local_name フォールバックは安全か

**結論: パースは可能だが、ラベル引きに local_name フォールバックが必要。概ね安全。**

- [F1] H-3.a.md [F10] より、名前空間 URI にバージョン日付が含まれる（例: `.../jppfs/2024-11-01/jppfs_cor`）
- [F2] H-3.a.md [F13] より、deprecated 要素数は jppfs:229, jpcrp:531, jpigp:538（累積）
- [R1] [F1] → 古い書類の名前空間（例: `.../jppfs/2019-11-01/jppfs_cor`）は ALL_20251101 の名前空間（`.../jppfs/2025-11-01/jppfs_cor`）と一致しないため、`{namespace_uri}:{local_name}` キーではラベルが引けない
- [R2] [F2] → deprecated 要素は全体の 5-10% 程度。大部分の概念は `local_name` が安定しているため、`local_name` のみでラベルを引くフォールバックは **90%以上の概念で正確**
- [R3] **安全性リスク**: deprecated 要素の `local_name` が新概念と衝突する可能性は理論上あるが、EDINET の設計上、deprecated 要素は新概念とは異なる `local_name` を持つ（例: `MinorityInterests` → `NonControllingInterests`）。衝突リスクは極めて低い。

### Q-2.2: 名前空間 URI バージョンを無視して local_name のみでラベル辞書を引くのは安全か

**結論: 安全（条件付き）。**

- [R4] [R2]+[R3] → モジュール名（`jppfs_cor`, `jpcrp_cor` 等）+ `local_name` をキーにすれば、バージョン間で同一概念のラベルを正しく引ける
- [R5] **推奨キー設計**: `{module}:{local_name}`（例: `jppfs_cor:Assets`）
  - 名前空間 URI からモジュール名を抽出: `http://.../{module_name}/{version}/{prefix}` → `{prefix}`
  - これにより同一概念はバージョンに関わらず同一キーになる
- [R6] **例外処理**: deprecated 要素で `local_name` が見つからない場合:
  1. deprecated ラベルテーブルを検索
  2. 見つからなければ `local_name` をそのまま表示（例: `MinorityInterests`）

### Q-2.3: タクソノミを何年分保持すべきか

**結論: 最新1バージョンで十分（local_name フォールバック前提）。全バージョン保持は不要。**

- [F3] Q-2.4.a.md [F1] より、API 最古日は 2016-03-03（約10年分）
- [F4] Q-2.5 結果より、全期間で使用された EDINET バージョンは **8種類**
- [R7] [F3]+[F4] → 全バージョンのタクソノミを保持すると 8〜11 個の ALL_YYYYMMDD（各 75MB）が必要
- [R8] **推奨**: ALL_20251101 の最新版 **1つのみ** を保持し、以下で対応:
  1. 最新タクソノミのラベル辞書をビルド（`{module}:{local_name}` → ラベル）
  2. deprecated ラベルも含めてビルド（deprecated XSD + label XML から）
  3. 古い書類は local_name フォールバックでラベルを引く
  4. 未知の local_name はそのまま表示（graceful degradation）

## 設計への影響（まとめ）

```python
def extract_module_key(namespace_uri: str) -> str | None:
    """名前空間 URI からバージョン非依存のモジュールキーを抽出する。

    Args:
        namespace_uri: 名前空間 URI。

    Returns:
        モジュールキー（例: 'jppfs_cor'）。抽出不能なら None。

    例:
        'http://disclosure.edinet-fsa.go.jp/taxonomy/jppfs/2024-11-01/jppfs_cor'
        → 'jppfs_cor'
        'http://disclosure.edinet-fsa.go.jp/taxonomy/jpdei/2013-08-31/jpdei_cor'
        → 'jpdei_cor'
    """
    # URI の最後のセグメントがモジュール名
    parts = namespace_uri.rstrip('/').split('/')
    return parts[-1] if parts else None
```

### deprecated 概念による残存 FALLBACK

Q-2.1 [R2] の予測通り、deprecated 概念が FALLBACK の主要因。

| 書類タイプ | 実データ NS バージョン | FALLBACK | deprecated 概念の例 |
|---|---|---|---|
| 240 公開買付届出書 | `2020-11-01` | 22.6% | `XxxNA` サフィックス（28件） |
| 290 意見表明報告書 | `2013-08-31` | 11.1% | 同上（5件） |
| 350 大量保有報告書 | `2014-07-31` | 5.0% | 同上（6件） |

FALLBACK 概念の特徴:
- 全て `NA` サフィックス付き（例: `NameOfLegalRepresentativeCoverPageNA`）
- 現行版の `deprecated/` ディレクトリに `deprecatedLabel` ロールのみで存在
- 対応する現行版概念（`NA` なし）はラベルあり

### 対応方針

Q-2.3 [R8] の推奨方針に従い、`taxonomy/versioning`（FEATURES.md）で対応する:
1. deprecated XSD + label XML からラベルを含めてビルド（ただし表示用ラベルは `deprecatedLabel` ロールのみで `role/label` がないため、旧概念→新概念マッピングが必要）
2. または旧バージョンのタクソノミパッケージを並列ロード

v0.1.0 スコープ（120/160）は影響なし。v0.2.0 の公開買付・大量保有対応時に着手。
