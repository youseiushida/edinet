### 推論（Reasoning）

1. **D-3.1 DEI要素について**: [F2] より、DEI タクソノミスキーマに `AccountingStandardsDEI` 要素が `xbrli:stringItemType` 型・`nillable="true"` で定義されている。[F3][F4][F5] の documentation ラベルに、取りうる値が記載されている。さらに [F14]（添付5）により、修正国際基準を含む完全な値が確定した:
   - `"Japan GAAP"` -- 日本基準
   - `"IFRS"` -- IFRS
   - `"US GAAP"` -- 米国基準
   - `"JMIS"` -- 修正国際基準
   - `xsi:nil="true"` -- 上記以外の提出書類、または経理の状況がない場合

   注目すべき点として、`"JMIS"` は documentation ラベル（[F3][F4][F5]）には記載されておらず、添付5（[F14]）でのみ明記されている。タクソノミ XSD の型は `stringItemType` であるため XSD レベルでは制約されないが、添付5が正式な値の定義元である。また [F4] の 2024年版では四半期報告書の記述が削除されており、四半期報告書制度の廃止に対応していると推測される。

2. **D-3.2 名前空間による推定について**: [F6][F7][F8] より、`jppfs_cor` は日本基準の財務諸表本表タクソノミ、`jpigp_cor` は国際会計基準（IFRS）タクソノミに対応する。したがって、インスタンス文書中に `jpigp_cor` プレフィックスの要素が使用されていれば IFRS と推定できる。ただし [F9] より、IFRS 適用企業でも `jppfs_cor`（財務諸表本表タクソノミ）のディメンション要素（連結・個別の軸など）は併用されるため、`jppfs_cor` の存在だけでは日本基準とは断定できない。また、EDINET では IASB の `ifrs-full` 名前空間は使用されず、独自の `jpigp_cor` 名前空間が使われる点に注意が必要。US-GAAP については専用の名前空間プレフィックスがなく（[F13] より包括タグ付けのみ）、名前空間からの判別は困難。

3. **D-3.3 schemaRef による推定について**: [F10][F11] より、報告書インスタンスの schemaRef は提出者別タクソノミのスキーマファイルを参照する。このファイル名は `jpcrp030000-asr-001_X99999-000_2013-03-31_01_2013-06-28.xsd` のような形式であり、会計基準を直接示す情報は含まれない。提出者別タクソノミが内部で `jpigp_cor` をインポートしているかどうかを辿れば間接的に IFRS と判別できるが、schemaRef の URI 文字列そのものからは判別不可能。
