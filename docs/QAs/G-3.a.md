## §1. 臨時報告書に XBRL は含まれるか

**結論**: はい、臨時報告書には XBRL が含まれる。`has_xbrl` が true になるケースが存在する。臨時報告書は XBRL 作成ツールの対象であり、専用タクソノミ `jpcrp-esr` を使用する。

### 情報源 (Fact)

- **F1**: 仕様書「EDINETタクソノミの概要説明」L.844: 図表 2-3-1 No.14: 臨時報告書 → タクソノミ分割単位 `jpcrp-esr`、様式番号「第五号の三様式」。
- **F2**: フレームワーク設計書 L.331: 臨時報告書（第五号の三様式）→ DEI ○ + 専用タクソノミ列に ○。財務諸表本表タクソノミには ○ なし。
- **F3**: 提出者別タクソノミ作成ガイドライン L.93: 「臨時報告書 | 企業内容等の開示に関する内閣府令 | 第五号の三様式」がXBRL作成ツール対象として明記。
- **F4**: jpcrp-esr_cor XSD 全体（74行）: 全要素が `textBlockItemType`（テキストブロック）、`stringItemType`（文字列）、`dateItemType`（日付）のいずれか。数値型要素（`monetaryItemType`, `sharesItemType` 等）は一切定義されていない。

### 推論 (Reasoning)

F1-F3 により、臨時報告書は XBRL 提出の対象であり、XBRL 作成ツールでも対応している。F1 で `jpcrp-esr` という独立したタクソノミ分割単位が割り当てられており、通常の `jpcrp` とは別管理であることがわかる。

## §2. 財務データを含むか、DEI のみか

**結論**: 臨時報告書の XBRL は **財務数値データを含まない**。DEI + 表紙情報 + 本文テキストブロックのみで構成される。

### 情報源 (Fact)

- **F5**: jpcrp-esr_cor XSD 要素分類（`G-3.esr_elements.py` で検証）:
  - **目次項目（stringItemType、abstract=true）**: 4要素（Heading系）
  - **テキストブロック（textBlockItemType）**: 47要素 — 臨時報告の各提出理由（海外募集、合併、株式交換、訴訟、災害等）に対応
  - **表紙文字列（stringItemType、concrete）**: 11要素（会社名、提出先、代表者等）
  - **表紙日付（dateItemType）**: 1要素（FilingDateCoverPage）
  - **数値型（monetaryItemType, sharesItemType, percentItemType 等）**: **0要素**
- **F6**: フレームワーク設計書 L.331: 臨時報告書行で財務諸表本表タクソノミ（jppfs/jpigp）の列に ○ なし。
- **F7**: F5 の textBlockItemType 要素の具体例:
  - `ReasonForFilingTextBlock`: 提出理由
  - `PublicOfferingOrSecondaryDistributionOfSecuritiesOutsideJapanTextBlock`: 海外募集
  - `DecisionOnAbsorptionTypeMergerTextBlock`: 吸収合併
  - `CommencementOrResolutionOfLitigationTextBlock`: 訴訟開始・解決
  - `SignificantDisasterTextBlock`: 重大な災害
  - `FinancialCovenantsTextBlock`: 財務制限条項
  - `CorporateShareholderGovernanceAgreementTextBlock`: ガバナンス契約

### 推論 (Reasoning)

F5 の XSD 解析により、`jpcrp-esr` タクソノミには数値型要素が一切存在しない。全コンテンツ要素（abstract=false）は `textBlockItemType` または `stringItemType` / `dateItemType` であり、テキスト・メタデータの記述に限定される。F6 でも財務諸表本表タクソノミの使用が否定されている。

したがって、臨時報告書の XBRL は「DEI（文書・提出者基本情報）+ 表紙情報 + 本文のテキストブロック」で構成され、構造化された財務数値データは含まれない。パーサーの観点では、`jpcrp-esr` の Fact はすべてテキスト系であり、数値パース処理は不要。
